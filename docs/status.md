**Phase 7（PP-DocLayoutV2 内嵌版面，2026-06-14/15）**：把质量更优的 OpenDoc 版面模型 **PP-DocLayoutV2(RT-DETR, 25 类 + 原生阅读顺序)** 内嵌为 `--layout` 第二后端,与 DocLayout-YOLO **共存(按 ONNX 输入数自动识别)**。突破:之前判的"RT-DETR 跑不了 tract"实为 **tract 0.23.1 两个具体 bug**——`GatherNd` 推断规则引用错张量(一行)+ `TopK` 不收 TDim(`Cast(INT64)→TDim`);各一处 vendored 补丁(`vendor/`,`[patch.crates-io]`,见 [vendor/PATCHES.md](../../vendor/PATCHES.md))即 **eval 端到端通、tract==ONNX Runtime 5/5 页逐框一致**,速度 1.38× YOLO(过门),`GridSample×18` 虚惊(tract 能 optimize)。补丁对现有模型**可证明零影响**(PP-OCR/UniRec/SLANet/TATR 不用这两算子;YOLO 的 TopK 走非-TDim 原路径;149 单测 + 三件套 + 双后端回归通过)。`RegionKind` 统一 YOLO/PPV2 语义,formula/transcribe 顺带支持 PPV2。**记分牌 A/B 已跑**(同页同打分器,唯一变量=后端,见 [testresults/2026-06-15-ppv2-vs-yolo-omnidocbench.md](testresults/2026-06-15-ppv2-vs-yolo-omnidocbench.md)):**General 端到端表 TEDS_X 0.206→0.654(≈3×)**——PPV2 检得准救回 YOLO 在杂版面漏检的表;学术表持平(0.670 vs 0.661,YOLO 本就检得到)、文本两者平(OCR 主导)。结论:PPV2 赢在"检测是瓶颈"的杂文档,故 `--layout-model ppv2` 与 YOLO 共存按需启用。**决定(2026-06-15)**:两处 tract 补丁以 vendored 形式长期留 main,**不发上游 PR**;理由/维护(bump tract 时重打补丁)/回退条件见 [vendor/README.md](../../vendor/README.md)。见 [plans/ppv2-layout-via-tract-patch.md](plans/ppv2-layout-via-tract-patch.md)、[devlogs/2026-06-14-ppv2-tract-integration.md](devlogs/2026-06-14-ppv2-tract-integration.md)、analysis [why-tract-cant-run](analysis/2026-06-14-why-tract-cant-run-pp-doclayoutv2.md)/[vendored-on-main](analysis/2026-06-14-vendored-tract-patch-on-main.md)。用法:`--layout --layout-model models/layout-ppv2/PP-DoclayoutV2_simp.onnx`(模型由 `scripts/spike/ppv2/prepare.py` 一次性生成,gitignored)。

**Phase 8b（OCR 增强遍页级并行,2026-06-17,已实施）**:研究"识别速度还能不能提升"时实测发现——确定性解析早已 rayon 页并行,但 **OCR `enhance::apply` 这一遍是串行的**,且 `recognize()` 的 `rec_cache` `MutexGuard` **持有到 `model.run()` 结束**把 rec 推理**全局串行化**。朴素并行只到 3×(det 能并行、rec 被锁死);**取出 `Arc` 句柄→释放锁→再 run** 后,par×2=完美 2.0×/100%(证明 tract 单次 rec 推理本就近单线程,封顶纯属那把锁),**18 核多页扫描吞吐 10.2×**(spike:chinese_scan 内存复制 18 页)。落地:`apply()` 页循环改受限 `par_iter`(scoped pool `min(cores,8)`,扫描 buffer ~100MB/页 → 内存是闸,非核数;`MAX_PAGE_PARALLELISM=8` 在效率拐点)+ 索引收集保序(输出字节一致)+ rec 锁修复;`core` 加 `rayon` 依赖(通用并行库,不破"core 不依赖 PDF 库"分层)。新单测守"并行==串行+确定性";三件套字节不变、chinese_scan `--ocr` 逐字不变。**续(第二 commit):页内 rec 并行攻单页延迟**——单页拆分实测 det 57%(单次推理拆不动)/ rec 41%(14 框可并行);`ocr_boxes` 改**自适应并行**(`rayon::current_thread_index()` 判是否已在页级池:单页并行 box、多页不嵌套),**单页 1.31×(0.275→0.211s)+ 多页零回归**。见 [plans/ocr-page-parallel.md](plans/ocr-page-parallel.md)、[devlogs/2026-06-17-ocr-page-parallel.md](devlogs/2026-06-17-ocr-page-parallel.md)。

## 4. 跨阶段经验教训

1. **记分牌大跳几乎全是评测/输出管线 bug**——分数可疑先怀疑管线；
2. **参照系口径会反噬更忠实的输出**（span vs 压扁），产品价值用语义样例验收；
3. **依赖版本本身可以是性能特性**（tract 0.21→0.23 = 17×）；
4. **管道退出码会掩盖失败**；
5. **新格式 e2e 是共享层的免费测试矩阵**；
6. **提分前先验"瓶颈是模型还是尺子"**——便宜旋钮（max_tokens/重采样/分辨率）对模型天花板无效，抬 token 上限反让难表退化更久；
7. **口径判断必须看图坐实**（#4/#16 初判"口径反噬"裁图后推翻=实为模型 over-split）；
8. **"显然该有用"的杠杆也要量**——分辨率确是真杠杆（饥饿表 0→0.97）但脆模型对它净≈0 且破速度门，**真杠杆≠能用**。
