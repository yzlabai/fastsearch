# 文件解析与摄取（fastsearch ingest）

> fastsearch 把 [docparse](../vendor/docparse) **subtree 并入本仓**（融合 Option B），`fastsearch ingest <文件>`
> 即可**进程内**把多种文档解析成可溯源 chunk 并索引——无需起外部解析服务、无需 shell 出中间 JSON。
> **搜索本身零依赖**：解析能力是 cargo feature，按需开启；不开则二进制精简、不含任何解析/ONNX 依赖。

---

## TL;DR

```bash
# 1) 编译带解析能力的 CLI（多格式，轻量、无 ONNX）
cargo build -p fastsearch-cli --features parse --bin fastsearch

# 2) 摄取一个文件（按扩展名自动选解析器）→ 落盘索引
./target/debug/fastsearch ingest --data ./idx --collection kb --doc-id report.docx report.docx
./target/debug/fastsearch search --data ./idx --collection kb --query "毛利率" --json
```

支持格式（`--features parse`）：**PDF · DOCX · HTML · Markdown · CSV · XLSX · PPTX · SRT(字幕) · EML(邮件) · 图片**。

---

## 三种摄取入口（按需选）

| 入口 | 命令 | 适用 | 依赖 |
|---|---|---|---|
| **多格式解析** | `ingest <file>`（`--features parse`） | 有原始文件、想进程内解析 | 轻量、无 ONNX |
| **外部 chunks** | `index <chunks.json>` | 已用 docparse/别的工具产出 chunks（JSON/NDJSON） | 零依赖（默认构建即有） |
| **纯文本文件夹** | `index-dir <dir>` | 一堆 `.md`/`.txt`，想快速跑通 | 零依赖 |

三者都走同一条「适配成真源 `Chunk`（加 `tenant`/`acl`）→ 落盘索引 → 检索」的路；`ingest` 多了"进程内解析"。

---

## 构建分档（feature 门控）

| 构建 | 含 | 重量 |
|---|---|---|
| `cargo build`（默认） | 搜索热路径（四张脸 + 混合检索 + ACL + CDC） | **零 docparse/ONNX** |
| `--features parse` | + 多格式解析器（9 格式 + 图片） | 轻量、纯 Rust、无 ONNX |
| `--features parse-ocr` | + 扫描件/图片 **PP-OCR 抽文本** | 重（tract/ONNX） |
| `--features parse-tables` | + **非 VLM 表格结构识别**（UniRec ONNX） | 重（tract/ONNX + 纯 Rust 栅格化） |

> 重档（parse-ocr/parse-tables）只影响**摄取侧**；搜索/服务二进制可继续用默认精简构建。

---

## 扫描件 / 图片 OCR（`--features parse-ocr`）

扫描件、无文本层的 PDF、图片 → 用 **PP-OCR**（ONNX）抽出文字再索引。有文本层的 born-digital
文档**不会触发 OCR**（省算力）。

```bash
cargo build -p fastsearch-cli --features parse-ocr --bin fastsearch
FASTSEARCH_OCR_MODELS=/path/to/models/ppocr-v5 \
  ./target/debug/fastsearch ingest --data ./idx --collection kb --doc-id scan.png scan.png
# stderr: "OCR: 1/1 页经增强（PP-OCR）"
```

- env **`FASTSEARCH_OCR_MODELS`** 指向 PP-OCR 模型目录（含 `*det*.onnx` + `*rec*.onnx` + 字典）。
- 模型不随仓分发，用 docparse 的 `scripts/fetch-models.sh` 下载（`ppocr-v5`/`ppocr-v6` 等）。

---

## 表格 / 图表理解（**不用 VLM** —— `--features parse-tables`）

**澄清**：表格/公式/版面**结构**用的是**确定性 ONNX 模型**（UniRec/SLANet/版面检测），**不需要 VLM**。
VLM 只用于"自然图/图表的**语义描述**"（"这张折线图在讲什么"），那才需要外部 HTTP 服务。

```bash
cargo build -p fastsearch-cli --features parse-tables --bin fastsearch
FASTSEARCH_UNIREC_MODELS=/path/to/models/unirec \
  ./target/debug/fastsearch ingest --data ./idx --collection kb --doc-id r.pdf r.pdf
# stderr: "UniRec: 重识别 N 个表格结构（非 VLM）"
```

- 解析检测出的表格区域 → 纯 Rust 栅格化裁剪 → **UniRec** 重识别为结构化 HTML 表格 → 替换入索引。
- 对应 docparse-cli 的 `--unirec`（本地 ONNX 路），区别于 `--vlm-tables`（VLM 路）。
- env **`FASTSEARCH_UNIREC_MODELS`** 指向 UniRec 模型目录。
- ⚠️ **性能**：UniRec 是 2000-token 自回归解码，**CPU 上单个复杂表可能耗时数分钟**；大批量建议 GPU。

## 还没接（下一迭代）

- **VLM 自然图语义描述**（`parse-vlm`）：对图/图表生成 caption，需 OpenAI 兼容 VLM 服务（如 Ollama llava）。
- **公式 → LaTeX**（UniRec 同模型）、**独立版面增强**：同 ONNX 路可后续接。

## 摄取后

无论哪种入口，摄取产物都是统一的 `Chunk`（含 `kind`/`page`/`bbox`/`heading_path` + `tenant`/`acl`），
之后就是标准检索：keyword / vector / hybrid，命中带 **page+bbox 引用**。详见
[在 Agent 中使用 fastsearch](在Agent中使用fastsearch.md)。

> 真源是 Postgres：生产路径是「写 PG → 逻辑复制 CDC → 引擎派生索引」；CLI `ingest`/`index` 直接落本地
> 派生索引，便于离线/单机演示。两条路产出的 chunk schema 一致（适配器 `from_docparse_chunk` 编译期对齐）。
