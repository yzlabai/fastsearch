## 2. 关键接口边界（trait）

定义在 `fastsearch-core`，由各模块实现，保证可替换：

- `VectorBackend`（fastsearch-vector）：`upsert/delete/search(query_vec, filter, k) -> [(global_id, score)]`，两实现 a/b。
- `Embedder`（fastsearch-embed）：`embed(texts, kind: Query|Passage) -> [Vec<f32>]`。
- `Reranker`（fastsearch-rerank）：`rerank(query, candidates) -> [(id, score)]`。
- `TextIndex`（fastsearch-text）：`upsert/delete/search(query, filter, k) -> [(global_id, bm25_score, highlights)]` + `facets`。
- `Tokenizer`（fastsearch-text）：分词器抽象（lindera/jieba/icu/...）。
- `CdcSource`（fastsearch-sync）：复制流抽象（pgoutput）。
- `SnapshotStore`（生命周期）：派生索引快照持久化。

## 3. 开发推进次序（按产品设计 §6 阶段 + 依赖）

1. **P0 奠基**：workspace 骨架 + CI；fastsearch-core 数据模型/查询 AST/过滤/融合（纯逻辑，先做透，单测足）。
2. **P1**：text（全文）→ pg（真源）→ sync（CDC）→ 端到端"写 PG→同步→全文检索"；eval 起步。
3. **P2**：embed → vector(两档) → rerank → 一等混合融合。
4. **P3**：auto-merging/分面/高亮/调优/可解释（多在 core+text 内补）。
5. **P4**：server（安全/ACL/可观测/生命周期）。
6. **P5**：cli 完善 + clients + 基准 → GA。

## 4. 每个 spec 的标准结构

每个 `docs/specs/<n>-<module>.md` 必含：
1. **目的与范围**（这模块做什么、不做什么）
2. **公开接口**（trait/struct/fn 签名，错误类型）
3. **数据结构**（关键类型定义）
4. **行为规约**（每个功能点的精确语义、边界、错误处理）
5. **依赖**（crate 内/外）
6. **测试用例**（单测 + 集成，列具体 case）
7. **验收标准**（spec 完成的判据）
8. **状态**（待开发/开发中/已完成；迭代记录）

## 5. 开发循环（每模块）

```
写 spec → review spec → 实现 → cargo test（+clippy 零warning）
   → 失败/不足 → 迭代修复 → review 代码
   → spec 验收用例全绿 + review 通过 → 标记 spec 完成
   → 回写 spec（实现与 spec 偏差就更新 spec，代码是真源）
```
