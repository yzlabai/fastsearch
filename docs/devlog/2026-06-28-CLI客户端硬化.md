## 1. post_retry 类型化错误（review 修瑕）

**问题**（自审发现）：`post_retry` 靠 `e.to_string().starts_with("server 返回")` 区分"状态码拒绝（不重试）"vs"传输失败（重试）"——脆弱的错误字符串匹配，错误文案一改就失效。

**修复**：引入类型化 `PostError { Status(u16,String), Transport(String) }`：
- `post_raw` 返回 `Result<Value, PostError>`：ureq `Error::Status`→`Status`、其它（连接/读写/JSON 解析）→`Transport`。
- `post`（对外 anyhow）= `post_raw` + `PostError::into_anyhow(url)`（统一文案）。
- `post_retry` 直接 match 类型：`Status` 立即抛、`Transport` 重试到上限——不再字符串匹配。

**测试**（+2，counting mock）：
- `index_status_error_does_not_retry`：500 → 恰好 1 次请求（不重试）。
- `index_transport_failure_retries_then_succeeds`：首连不响应即关（传输失败）→ 重试 → 第二次成功；断言 ≥2 次请求。
- 重复跑 4× 无 flaky。

## 2. CLI↔server e2e CI job（契约不漂移）

此前 CLI↔server 端到端只在本地手测。新增 ci.yml `cli-server-e2e` job（无需 PG，server 用内存/落盘派生索引 + dev key）：
- build server+cli → 起 server（poll healthz）→ `index-dir`(喂文件夹) → `search 毛利率` grep `report.md`；stdin `index` → `search alpha` grep `kb:manual:0`。
- 守住"CLI 是 server 的 REST 客户端"契约：响应字段/端点/鉴权任一漂移即红。

**可移植性 review**：第一版用 `S="--server … --key …"; fastsearch $S` 简写——本地 **zsh 不对无引号变量做分词**（CI bash 会），本地复现失败暴露了这点 → 改为**逐命令显式列 flags**，两 shell 都对。本地实跑两条断言均过 + YAML ruby 校验。

## 3. index-dir 有界并发上传（Meilisearch-importer 式）

`index-dir` 原是逐文件串行 POST，大文件夹被单文件往返延迟拖慢。改为**有界并发**：
- `IndexDirOpts.concurrency`（`--concurrency`，默认 4）；`std::thread::scope` 起 `min(concurrency, n)` worker，**原子游标** `cursor.fetch_add` 取下一文件、**原子聚合** ok/failed/total（不丢不重）。
- 共享 `&Client`（不可变、Send+Sync）+ ureq 全局 agent 线程安全 → 并发 POST 安全。
- 计数确定；进度行可能交错（每文件独立 doc，无碍正确性）。文件仍排序 → doc_id 分配稳定。

**测试**：`cmd_index_dir_concurrent_uploads_all`（20 文件、concurrency=6 → ok=20/total=20，原子聚合不丢）；既有 3 文件串行测试（concurrency=1）仍绿。重复跑无 flaky。
