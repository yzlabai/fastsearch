## 1. MM2c-time — 时间区间列下推（已完成）

- **做什么**：pg 真源加 `time_start_ms/time_end_ms bigint`（由 `chunk.media.time` 派生落列）；B6 pgvector 直查把时间区间过滤从 `TRUE` 超集升级为**精确 SUPERSET 下推**（`col_kind` 纳入两列），与 modality/page 同构。
- **为什么**：音视频时间区间检索此前在 pg 直查路径只能 match-all + Rust 后过滤；下推让 PG ANN 在选择性时间过滤下仍 filter-aware（iterative scan）。后过滤 `PgVecRow` 仍取 `media.time` → 两端同源、守不变量 #5。
- **怎么验证**：收口三绿（fmt/clippy -D warnings/test --workspace）+ **Docker pgvector 真机**（DDL/16 参 INSERT/fetch 往返绿；`time_start_ms>=1000 AND time_end_ms<=5000` 仅命中区间内行）。新测试 `pgvector_sql_time_range_pushdown`、`chunkrow_media_roundtrip` 断言列值。
- **状态**：✅ 完成（commit `9099696`）。`media_bytes` 拆为 MM2c-bytes 随 MM6。
- **附带**：per-module spec 漂移回写（10-core/11-text/12-pg/16-embed），把已实现的多模态从代码回写进 spec。

## 2. MM5 — CDC `apply_upsert` modality 路由：无文本媒资不嵌（已完成 M0 部分）

- **做什么**：engine CDC 落地路径 `apply_upsert` 在 M0（仅文本嵌入器）下，**对无文本媒资 chunk（`text.trim()==""`，如无 caption 的图）跳过向量嵌入/写入**；幂等覆盖时若旧版本有向量则删除，避免残留。有文本（正文/caption/转录）照常嵌入。
- **为什么（真实问题）**：旧实现对所有 chunk 嵌 `chunk.text`。空串经 HashEmbedder 只剩前缀 token `passage` → **所有无文本媒资塌成同一向量**；真嵌入器给"空文档"向量，同样无意义 → 污染 ANN（一张图命中=全部无文本图同分命中）。plan M0 明确"退化为 caption 文本嵌入**或不嵌**"，这里取"不嵌"。无文本图仍在 BM25 + modality fast field（按模态可召回），视觉向量待 M1 图像嵌入路由。
- **怎么验证**：新单测 `mm5_textless_media_skips_vector`——经 `rebuild_from`（走 `apply_upsert`+HashEmbedder）灌入"有文本段 + 无文本图"，向量检索断言有文本段被召回、无文本图永不出现。收口三绿（fmt/clippy -D warnings/test --workspace，30 个测试套件全绿）。纯 Rust，无需 PG。
- **状态**：✅ M0 路由逻辑完成。M1 图像向量路由 `gated`（需多模态模型）。

## 5. MM2c-bytes — inline 字节存储通道（已完成）

- **做什么**：按 §4 计划做里程碑 1（纯加法、无行为变更）：`Chunk.media_bytes: Option<Vec<u8>>` 字节通道；PG `media_bytes bytea` 列（真源）；`ChunkRow` 往返；`PgStore::fetch_media_bytes(collection,doc_id,chunk_id)`（网关按需直查）；INSERT 16→17 参；fetch SELECT 携带；sync CDC 不搬字节（设 None，§4.2 决策）。
- **为什么**：`AssetPointer::Inline` 的字节此前在 docparse 已识别但 ingest 处被丢（Chunk 无字节字段）→ inline 路是死的。这一层是 MM6-inline 服务（网关吐字节）的地基。字节是**不可重派生的真源内容** → 存 PG、引擎派生层不持、网关按需直查（守"PG真源/引擎派生"）。
- **怎么验证**：① 单测 `chunkrow_media_roundtrip` 加 `media_bytes` 往返断言、DDL/insert golden 含 `media_bytes bytea`/17 参；② 新 **Docker pgvector 集成** `integration_media_bytes_roundtrip`：bytea 存字节、`fetch_media_bytes` 取回一致、文本段 NULL、不存在 None、fetch_doc 往返带回字节——**真 PG 全绿**；③ 收口三绿（fmt/clippy -D warnings/test --workspace 30 套件）；④ ~16 处 Chunk 构造点 ripple（编译器驱动逐一补 `media_bytes: None`），区分 Chunk vs Citation/VecMeta/ChunkRow。
- **范围澄清（核对代码后修正先前假设）**：先前以为"CLI 把 docparse `data_base64` 解码进 `media_bytes` 需加 `base64` 依赖"——**核实后该假设不成立**：CLI `cmd_index`/`cmd_ingest` 经 `engine.ingest()` 只写**本地 Tantivy 文本索引、不写 PG**（[lib.rs:242](../../crates/fastsearch-cli/src/lib.rs#L242)）。`media_bytes` 在 PG 真源，由 **`PgStore::upsert_doc`**（库/外部写入路径）落字节——该路径已绑 `media_bytes`。故 **CLI 无字节可携带**（CLI 根本不是 PG 写入者）；字节生产者用 `PgStore::upsert_doc` 即可。一个独立的"CLI 写 PG 真源"命令属另一特性（非 base64 依赖问题）。
- **状态**：✅ **完成**（PG 写入路径 `upsert_doc`→`media_bytes`→`fetch_media_bytes`→网关，Docker 端到端验证）。下一步 **MM6-inline**（engine `source_pg` + `resolve_citation` Inline→InlineBytes(block_in_place) + server 吐字节）；**MM6-secure**（无签名器 Object→404）。

## 6. MM6-inline — 媒资网关吐 inline 字节（已完成）

- **做什么**：打通"命中图 → `/v1/asset/{cid}` 吐图字节"的最后一环。engine 加 `source_pg` 真源句柄 + `set_source_store`；`resolve_citation` 的 `Inline` 分支 **从 PG `media_bytes` 真源 `block_in_place` 按需取字节** → `AssetFetch::InlineBytes`（ACL 已在前强制）；server main 装配 `set_source_store`（gated DATABASE_URL，任意向量后端）。server 端 `/v1/asset` handler 早已能吐 `InlineBytes`+Content-Type（无需改）。
- **为什么**：MM2c-bytes 把字节存进 PG 真源，但网关 Inline 路此前返回 None（死路）。这一环让"一个 PG 即可跑"的文档图内联展示真正成立，**不依赖对象存储**。字节按需从真源取、引擎派生层不持（守 #2）；sync↔async 经 `block_in_place` 桥（同 B6，需 multi-thread runtime）。
- **怎么验证**：新 **Docker pgvector 集成** `mm6_inline_serves_bytes_from_source_pg`（multi-thread runtime）：upsert 带 `media_bytes` 的图 → `set_source_store` → 授权 `resolve_citation` 返回 `InlineBytes(真源字节)`+Content-Type、**越权返回 None（ACL 不可绕过）**——真 PG 全绿。收口三绿。
- **server HTTP E2E（已补，2026-06-27）**：新 env-gated 测试 `asset_inline_bytes_e2e`（multi-thread + `oneshot`，比手动 curl 更可复现）——`GET /v1/asset/{cid}` 经 **auth→handler→engine→source_pg** 吐 PG 真源字节：授权 200 + `Content-Type: image/png` + 字节一致、**越权 404、无 key 401**，Docker 真机全绿。MM6-inline 的"server 待运行验证"升级为**已验证**。
- **状态**：✅ 引擎能力 + 装配 + server HTTP 路径全部验证。

## 7. MM6-secure — 堵裸 key 漏洞（已完成）

- **做什么**：`resolve_citation` 的 `Object` 路径此前返回 `SignedUrl{url:uri}` → 网关 302 到**对象存储裸 key**（违反不变量 #3）。引入 `ObjectSigner` trait + `set_object_signer`；**Object 必须经签名器签短时 URL；未配签名器 → None（404），绝不回退裸 key**。
- **为什么**：裸 key 暴露给客户端 = 可绕过 ACL 直接访问、URL 不过期。安全 > 功能：宁可 Object（音视频）暂不可服务（404）也不泄露 key。真签名器（S3 presign 类）gated 对象存储接入。
- **怎么验证**：单测 `mm6_secure_object_no_signer_is_404`（无签名器→None）+ `mm6_secure_object_with_signer_signs`（桩签名器→SignedUrl，断言 URL **不含 `s3://`**）；收口三绿；无既有测试依赖旧 Object→302。
- **状态**：✅ 堵漏完成。真签名器 gated。

## 8. code-review 发现并修复 — 时间下推的 #5 超集漏洞

- **怎么发现**：对 10 个 commit 跑 high-effort 多 agent code-review（8 finder 角度 + verify）。一条 CONFIRMED 发现。
- **问题（真 bug）**：MM2c-time 的时间下推用反规范化列 `time_start_ms`，但后过滤 `PgVecRow::get` 读权威 `media.time`（jsonb）。两者不同源——**列 NULL 而 `media.time` 有值的行（遗留/外部写入）**：下推 `time_start_ms >= $n` 排除（NULL 不满足），后过滤本会保留 → **下推排除了后过滤会保留的行 = 违反不变量 #5（漏召回）**。B6 直查走 PG 真源，遗留行真实可达。
- **修复**：① 后过滤继续读权威 `media.time`（与 brute/HNSW 侧一致）；② 下推对时间列改 **`(col OP $n OR col IS NULL)`** → 真超集（列 NULL 行也放行，交后过滤精确判定）。`nullable_denorm` 标记"权威源在别处的可空反规范化列"。
- **验证**：单测断言下推 SQL 含 `OR time_start_ms IS NULL`；新 **Docker 集成 `integration_time_filter_null_column_superset`**——把列 UPDATE 成 NULL（模拟遗留行），匹配查询仍召回、不匹配查询精确排除，真 PG 全绿。收口三绿。
- **状态**：✅ 修复并验证。
