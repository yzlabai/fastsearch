## 背景

M1（跨模态/以图搜图）的地基是 embed 层能吃图像。此前 `Embedder` trait 仅 `embed(&[String], kind)`
（纯文本）。MM8 要把它扩成多模态——但**真图像嵌入需多模态 HTTP 模型**（gated）。本轮做**可在本环境
完整验证的地基部分 MM8a**，把需真模型的 HTTP 图像路由切出为 MM8b（gated），守诚实记账。

## 设计取舍

1. **加法而非改签名**：不把 `embed` 改成 `embed(&[EmbedInput])`（那会 ripple engine/server 所有文本
   调用点）。改为**新增** `embed_multi(&[EmbedInput], kind)` + `caps()`，**保留 `embed(&[String])` 原样
   → 文本调用点零改**。trait 默认实现让旧后端"自动"具备多模态签名（Text 委托 `embed`、Image 报错）。
2. **`EmbedCaps` 自描述**：`{dim,text,image,cross_modal,semantic}`。`cross_modal` 是**以图搜文/以文搜图
   的前提**（D2 文图同空间）——查询期据此决定是否允许跨模态检索（否则拒绝，避免拿不可比向量做近邻）；
   `semantic=false` 诚实标注非语义后端。
3. **HttpEmbedder 图像路由切出为 MM8b（gated）**：真图像嵌入的请求/响应格式随模型/服务而异
   （SigLIP-2/JinaCLIP-v2/jina-v4/Voyage/Cohere），**无真模型无法敲定+验证**。实现一段投机的、未验证的
   base64 请求 = 违反诚实记账。故 `HttpEmbedder.caps{image:false}`、`embed_multi(Image)` 走默认报错，
   等真模型服务再做 MM8b。

## 做了什么

- **embed/lib.rs**：`EmbedInput::{Text,Image}` + `Modality` + `EmbedCaps` + trait `caps()`（默认纯文本语义）
  + `embed_multi()`（默认仅 Text、遇 Image 报错）。`HashEmbedder` 覆写 `caps{text,image,!cross_modal,!semantic}`
  + `embed_multi`（图像走 `embed_image_one`：字节 4-gram 特征哈希、`"img:"` 域前缀、确定性、L2 归一）。
- **embed/http.rs**：`HttpEmbedder.caps{image:false,semantic:true}`（图像 gated 注释指向 MM8b）。

## 怎么验证（本环境，纯 Rust）

+5 单测（`cargo test -p fastsearch-embed`，18 全绿）：
- `hash_caps_image_nonsemantic`：HashEmbedder caps 文图都支持但 `!cross_modal`/`!semantic`。
- `embed_multi_image_deterministic_and_distinct`：同图同向量、不同图不同向量、L2 归一。
- `embed_multi_mixed_text_and_image`：文图混批；文本项与 `embed()` 文本路径逐位一致（复用 `embed_one`）。
- `empty_image_is_zero_vector`：空图字节→零向量不 panic。
- `text_only_default_rejects_image`：默认 trait 实现的纯文本后端 `caps.image=false`、`embed_multi(Image)` 报错、`Text` OK。

收口三绿（fmt/clippy -D warnings/`cargo test --workspace` 30 套件）。**加法式 → 既有 30 套件零回归**
（engine/server 文本嵌入路径未动）。
