## 背景

MM8a 让 embed 层能吃图像（`embed_multi`/`caps`）。本轮把它**接进引擎检索两端**，打通 M1 基线链路：
写入侧让图 chunk 落视觉向量（MM10），查询侧支持以图搜图（MM9）。用 HashEmbedder 端到端可验；真跨模态
模型 gated。

## 做了什么

- **MM10 — `apply_upsert` 模态路由**（engine）：选嵌入输入——① 有可检索文本（正文/caption/转录）→ `Text`；
  ② 否则无文本但后端 `caps.image` 且有 inline `media_bytes` → `Image`（**视觉向量进检索**）；③ 都不满足
  → 不嵌（避免空串退化向量污染 ANN；仍在 BM25+modality fast field）。写穿/本地两路与 B6 一致。
- **MM9 — 以图搜图**（core+engine+server）：`SearchRequest.query_image: Option<Vec<u8>>`（core 加字段）；
  engine `embed_query_image`——`vector=None` 且带 `query_image` 且后端 `caps.image` → 嵌成查询向量走现有
  vector 召回；后端不支持图像 → 报错（不静默拿文本后端嵌图）。server **带 `query_image` 时不预嵌文本**
  （否则空文本向量遮蔽图像查询），留 `vector=None` 交引擎嵌图。
- core 排序/融合/ACL **零改**（D4：以图搜图复用 vector 路径）。

## 设计要点

- **图→图 vs 文→图**：HashEmbedder `cross_modal=false`——图像与文本向量不在同一空间，故**仅图→图有
  意义**（query_image 嵌图、近邻是图）。**以文搜图/真跨模态需 `caps.cross_modal` 真模型**（MM9-cross gated）。
  基线只验图→图 + 管线就位，不冒充跨模态。
- **生产现状诚实**：server 默认 `HttpEmbedder.caps.image=false` → 真部署 query_image 会报错、图像检索仍
  = caption-as-text，直到 MM8b 接真多模态模型。基线证明**链路**，非生产可用的视觉检索。

## 怎么验证（本环境，纯 Rust / HashEmbedder 确定）

+2 engine 测试（绿）：
- `mm10_image_routing_and_mm9_query_image`：3 张无 caption 图（各异 inline 字节）经 `apply_upsert` 落视觉
  向量；`query_image`=图 a 字节 → 最近邻是图 a 自身（图像嵌入忽略 kind 前缀，故 query/passage 同字节同
  向量、cos=1）；**无字节无文本图被跳过**（不进向量召回）。
- `mm9_query_image_requires_image_backend`：纯文本后端 + `query_image` → 报错 `caps.image=false`。

收口三绿（fmt/clippy -D warnings/`cargo test --workspace` 30 套件）。**既有套件零回归**（文本检索/CDC/B6
路径未受影响；query_image 字段 serde 默认、SearchRequest 全用 `..Default`）。

## review（自查）

- **不变量 #3 ACL**：query_image 走与文本查询**同一** vector 召回 + ACL 注入路径，无新绕过面。✓
- **不变量 #4 确定性**：HashEmbedder 图像向量确定 + GlobalId tie-break 不变。✓
- **诚实记账（#6）**：基线（图→图，本环境可验）/ gated（文→图跨模态 MM9-cross、真视觉模型 MM8b/MM10-real）
  切分清楚；生产默认 caps.image=false 的现状写进 spec/devlog，不把"链路就位"说成"图像检索可用"。
- **零 ripple**：core 排序/融合未动；server 仅加一处"带 query_image 不预嵌文本"守卫。
