# devlog 2026-06-28 — RaBitQ 无偏估计器（二值粗筛精化，迭代①）

> 上游：[后续开发计划-三条线](../plans/2026-06-28-后续开发计划-三条线.md) 第①条线首项「完整 RaBitQ（旋转 + 无偏估计）」；承接 [二值量化粗筛 devlog](2026-06-27-二值量化粗筛.md)。spec 回写 [15-vector](../specs/15-vector.md) §6。

## 做什么

把二值量化两阶段粗筛的**粗排打分**从对称 Hamming 升级为 **RaBitQ 无偏估计器**：

- 旧：`coarse = Hamming(sign(q), sign(x))`（符号不一致维数，越小越近）。只数符号一致维，**丢掉查询 `q` 的幅度**。
- 新：`coarse = ⟨q, sign(x)⟩ / ‖x‖₁`（[`binary::rabitq_estimate`](../../crates/fastsearch-vector/src/binary.rs)）。RaBitQ 对单位向量的内积估计——**用 `q` 的真实分量**（非对称）+ **逐向量 `‖x‖₁` 校正**（`‖x‖₁ = ⟨x, sign(x)⟩`，量化质量因子）。

实现（[fastsearch-vector](../../crates/fastsearch-vector/src/lib.rs)）：
- `binary.rs`：去掉 `hamming`，加 `l1_norm` + `rabitq_estimate`（只读 bit `code`、不取全精度 x，内存轻）。
- `Entry` 加 `l1: f32`（由归一化向量派生，**不落盘**，load 时同 `code` 一并重建）。
- `search` 粗筛分支：估计降序 + `GlobalId` tie-break（确定）；filter/ACL 仍在粗筛**前**（守不变量 #5）；全覆盖 oversample 仍逐条等于精确暴力。

## 为什么更准（一句话）

两条**同符号**的库向量，Hamming 必打平（无法区分）；估计器用 `q` 的幅度 + 各自 `‖x‖₁`，仍能按真实余弦把它们分开 → 同 oversample 下候选集更优 → 候选内 f32 重排后召回更高。

## 怎么验证（本环境，无外部依赖）

- `estimate_separates_what_hamming_ties`：构造同符号、真余弦不同的两库向量，断言估计器能分开（Hamming 打平）。
- `rabitq_estimator_beats_hamming`：400×96d 合成集、10 查询头对头——同 oversample 各自选候选 → 候选内 f32 重排取 top-k → 比 recall。**实测 est 0.870 vs ham 0.710（oversample=4，~16pt 增益）**；门禁 `est ≥ ham + 0.05` 且 `est ≥ 0.85`。
- 既有 `binary_full_oversample_equals_exact` / `binary_recall_high_with_moderate_oversample` / `binary_is_filter_aware` / `vectorstore_brute_binary_roundtrip` 均仍绿（行为兼容）。
- **收口三绿**：`cargo fmt --all --check` + `cargo clippy --workspace --all-targets -D warnings` + `cargo test --workspace`（vector 26/26）。

## 取舍 / 诚实记账

- **速度**：粗排从 `popcount(~d/64 字)` 变为 `~d` 次加法（仍只读 bit code、无乘、无全精度向量 I/O，内存仍轻）。换来召回明显提升 → 同召回目标可用**更小 oversample** → 更少 f32 重排，净算路常更划算。后续可加查询分平面量化恢复 popcount 级速度（下一迭代候选）。
- **未做（下一迭代）**：**随机旋转**。当前估计器未旋转 → 严格优于 Hamming 但**非**论文意义的无偏估计；旋转把量化误差摊匀后才得带误差界的无偏估计，对各向异性（聚簇）数据增益更大。合成集近各向同性，本迭代增益主要来自"用查询幅度 + `‖x‖₁` 校正"。
- 默认仍精确暴力（`binary_oversample=None`，零回归）；on-disk 格式不变（仍 f32 快照，`code`/`l1` 重建）。

## 迭代② — 随机旋转（2026-06-28，同日续作）

**做什么**：量化前对向量/查询做一次**数据无关的固定种子正交变换**，把信息/量化误差摊匀到各维，使符号码更均匀有信息 → 估计器趋**无偏**、对各向异性数据增益大。

- `binary::Rotation`：高斯随机矩阵（Box–Muller）+ 改进 Gram-Schmidt 行正交化；`new(dim, seed)` / `apply(v)`。
- `MemVectorIndex::with_binary_prefilter_rotated(oversample)`：opt-in；首次 upsert（dim 已知）惰性建矩阵。`code`/`l1` 由**旋转后**向量派生；`vector`（原向量）不变——正交不改内积，精排仍用原向量。search 粗筛时查询也旋转。
- **固定种子**（`RABITQ_ROTATION_SEED`，数据无关、定常）→ 多副本/重开生成同一矩阵，无需持久化矩阵、确定。搜索策略不落盘（同 `binary_oversample`；`load` 默认不旋转，engine 接线按需重启用——下一迭代）。

**为什么有效**：符号量化最怕能量集中在少数维（其余维符号是噪声）。随机旋转把能量摊到所有维 → 每个符号位都携带信息 → 粗筛候选更优。各向同性数据增益小（已均匀），各向异性数据增益大。

**怎么验证**：
- `rabitq_rotation_helps_anisotropic`：构造能量随维指数衰减的各向异性集，旋转档 vs 非旋转档同 oversample 比 recall。**实测 rot 0.967 vs norot 0.775（oversample=3，~19pt 增益）**；门禁 `rot ≥ norot + 0.05`。
- `rabitq_rotation_full_oversample_equals_exact`：旋转档全覆盖 oversample 仍逐条等于精确暴力（正确性不破）。
- `rabitq_rotation_deterministic`：两次独立构建旋转档结果逐条一致（固定种子）。
- **收口三绿**：fmt + clippy -D warnings + test --workspace（vector 29/29）。

**取舍**：旋转 `O(d²)` 一次性（建矩阵 `O(d³)` 一次、每向量/查询 `apply` `O(d²)`）——对 build/query 各加一次矩阵乘，索引构建期可接受；对召回的提升（各向异性 ~19pt）划算。默认仍精确暴力、on-disk 不变。

## 迭代③ — engine/server 接线旋转档（2026-06-28，同日续作）

**做什么**：把迭代②的旋转能力从 vector-crate 原语接成端到端可用的后端档（此前只 `MemVectorIndex` 构造器可达，生产路径触不到 → 避免"半接线"）。

- `VectorBackendKind::BruteBinaryRotated(oversample)` 新档；`VectorStore::new/kind_str/load` 接线（`kind_str="brute_binary_rotated"`；`load` 重建旋转矩阵 + 旋转空间重算 code/l1）。
- `MemVectorIndex::set_rabitq_rotation(enabled)`：开/关旋转并在新空间重算所有 `code/l1`（供 load 后翻档，同 `set_binary_prefilter` 的"策略不落盘、调用方设"模式）+ `rabitq_rotation_enabled()` 访问器（`kind_str` 据此区分）。
- engine `open_with`：检查点 `vector_backend="brute_binary_rotated"` → 恢复旋转档（覆盖默认）。
- server：`FASTSEARCH_VECTOR_BACKEND=brute_binary_rotated`（`FASTSEARCH_BINARY_OVERSAMPLE` 调 oversample）；env 文档更新。

**怎么验证**：
- `vectorstore_brute_binary_rotated_roundtrip`：`kind_str` 正确 + 落盘往返（load 重建矩阵）后仍旋转档、结果一致。
- `persist_reopen_restores_brute_binary_rotated_backend`：首启旋转档 → 检查点记 → 重开（默认 brute）仍恢复旋转档、检索可用。
- **收口三绿**：fmt + clippy -D warnings + test --workspace（vector 30/30、engine 35/35）。

**为什么这样接**：旋转矩阵由固定种子重建、不落盘（on-disk 仍纯 f32），故 `load` 只需 `set_rabitq_rotation(true)` 即在旋转空间重算 code——与 `brute_binary` 完全同构的"格式入检查点、矩阵/码重建"策略，零新增落盘字段、守"派生可重建"。
