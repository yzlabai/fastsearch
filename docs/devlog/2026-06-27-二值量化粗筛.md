## 背景与取舍

看板"RaBitQ/量化"下一迭代项。**完整 RaBitQ**（随机旋转 + 1-bit + **无偏内积估计器**）实现精巧、估计器
极易写错且"验证"是统计性质——错的 RaBitQ 比诚实的"未做"更糟。故本轮落 **RaBitQ 的核心子集 = 二值量化
（BQ）两阶段粗筛**（成熟技术，Qdrant/Weaviate 在用），明确正确、可对账验证；完整 RaBitQ 的旋转+无偏
估计器作其上的精化、下一迭代。

## 做了什么

- **`binary.rs` 原语**：`pack_signs(归一化向量) → Vec<u64>`（每维符号 1 bit，按 u64 字打包）；
  `hamming(a,b) = Σ popcount(a^b)`。+3 单测（pack/hamming、跨字、0.0 视正号）。
- **`MemVectorIndex` 两阶段粗筛（opt-in）**：新 `with_binary_prefilter(oversample)`；`Entry` 加 `code`
  （upsert/load 时由归一化向量派生，不落盘）。search：开启时 → **Hamming 粗筛 top-`k·oversample`**
  （键 `(hamming 升, gid 升)`，确定候选集）→ 仅对候选做 **f32 精确重排**；关闭时（默认 `None`）= 原精确暴力。
  filter/ACL 仍在粗筛**之前**施加（守不变量 #5）；重排 + GlobalId tie-break → 确定。

## 为什么正确 + 快

- **正确**：符号 Hamming 越小 ⇒ 符号一致维越多 ⇒ 余弦越高（**粗代理**）；候选集内用 f32 精确重排 → 最终
  top-k 在候选集内精确。vs 全局精确的 recall 由 oversample 决定。
- **快**：粗筛每条 ~`d/64` 次 u64 `xor+popcount`（d=384 → 6 字操作）vs 精确 `d` 次浮点乘加 → 粗筛阶段
  快一两个数量级；只对 `k·oversample`（远小于 N）条做 f32 重排。

## 怎么验证（本环境，纯 Rust，确定）

+5 单测（`cargo test -p fastsearch-vector`，22 全绿）：
- **`binary_full_oversample_equals_exact`**（强保证、无统计抖动）：oversample 大到覆盖全集 → 候选=全部 →
  结果与精确暴力**逐条 chunk_id 相同 + 分值 f32 精确**。证明重排路径正确。
- **`binary_recall_high_with_moderate_oversample`**：oversample=8、7 个查询，**recall@10 ≥ 0.85**。
- **`binary_is_filter_aware`**：二值档 + `kind=table` 过滤 → 只返回 table（粗筛前置过滤，守 #5）。
- `pack_and_hamming_basics` / `pack_spans_multiple_words`：原语正确性。

收口三绿（fmt/clippy -D warnings/`cargo test --workspace`）。**默认 `None`=精确暴力 → 既有 22 套件零回归**
（exact 路径逐字节不变）。
