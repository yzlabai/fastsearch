## 背景

R4 触及崩溃安全的 CDC 核心路径，**全程用真 pgvector 容器（pg16 + `wal_level=logical`）e2e 验证**（review 本身就是靠真容器发现这些的）。本轮做了 5 项高价值、可 Docker 验证的修复；另有 5 项需 Change-enum/IndexSink 大手术或属边界/perf 的，**如实标记延后**（见下"延后与记账"）。

## 做了什么（Docker 实测）

- **H3 CDC UnchangedToast 毒丸**（真·高危）：TOAST 大列（text）在 UPDATE 里未变时 pgoutput 发 `'u'`，旧代码当 missing/null → `row_to_chunk` 报错 → 整批失败 → slot 永不 advance → **CDC 永久卡死**。修复：`map` 检测元组含 `UnchangedToast` → 提取 gid（gid 列小、不 TOAST、必在）→ 交 `fetch_changes` 用 `fastsearch_pg::fetch_chunk` **从 PG 真源重取整行**（"PG 是真源、索引派生"）；行已被删则跳过（后续 Delete 事件清）。**e2e 复现+负控**：`ALTER COLUMN text SET STORAGE EXTERNAL` + 大 text + 只改小列的 UPDATE → 未修时 `column 'text' missing/null`（正是 review 诊断），修后重取保住 text、可检索。
- **H3 附带·更正失效架构断言**：实测证伪"列清单 publication → 写穿 UPDATE 不产生复制事件"——列清单只过滤"列的值"、**不抑制"Update 事件本身"**。回写 `sql.rs` DDL 注释：写穿反馈环靠 `set_embedding` 的 `IS DISTINCT FROM` 幂等守卫阻尼（一轮收敛），**不是**靠"不产生事件"；且该写穿事件对大 text 行正是 `'u'` 毒丸载体 → 依赖 H3 才不卡死。
- **M16 复制流 OOM**：`media_bytes`（可达 20MB/张）此前在 publication 列清单里 → 整字节走 WAL 逻辑解码进内存（与"复制流只搬指针"注释矛盾）。从 `COLUMNS` 移除；`fetch_changes` 加 `upto_nchanges=10000` 批量上限（PG 于事务边界停止，崩溃安全水位不受影响）。
- **M17 halfvec 直查断裂**：`ann_index_sql` 硬编码 `vector_cosine_ops`（halfvec 列建 hnsw 报错）且**从不被调用**（直查全程顺序扫描）；查询 SQL 也硬编码 `$1::text::vector`（halfvec 列 `<=>` 类型不符）。修复：opclass/cast 都随 `VectorType`（halfvec→`halfvec_cosine_ops`/`::halfvec`），并把 ANN 索引建进 `ddl`。**e2e**：默认 HalfVec 档 `ensure_schema` 现真的建出 halfvec HNSW 索引（真 PG 通过，此前 halfvec 全链零覆盖）。
- **M18 维度硬编码 384**：`PgConfig::new` 默认 384 与运行时 embedder 维度零联动 → 配非 384 模型则 `set_embedding` 逐行报维度不匹配、embedding 列永空。加 `PgConfig::with_vector_dim`，server 提前读 `ecfg.dim` 并贯穿三处 PgConfig 建点。
- **M12 HttpEmbedder 无批量/重试/超时可配**：整 doc 全部 chunk（含 base64 图片）合成单请求 → 413/超条数上限/超时、无部分成功。加 `max_batch`（默认 64）按条数拆多请求拼接、transient（429/5xx/传输）指数退避重试（默认 2）、`timeout_secs`/`max_batch`/`retries` 全 env 可配。**mock TCP server 测**：5 条/max_batch=2 → 3 次请求；503 后重试成功；retries=0 立即失败。

## 取舍

- **H3 重取 vs 部分更新**：选"从真源重取整行"而非"派生索引按字段部分更新"——前者概念最简（"不完整 WAL → 读真源"）、对任意 TOAST 列统一、与"索引派生"不变量一致；后者需 Change/Chunk 表达"某列未变"+ 引擎字段级合并，复杂且易错。重取的 PG 往返只在遇 `'u'` 时（罕见），代价可控；幂等（重取当前真源、索引收敛到真值），崩溃重放安全。
- **M17 ANN 索引进 ddl**：embedding 列全 NULL（非直查部署）时 HNSW 索引为空、近零代价；直查填充后即生效。故无条件建是安全的。

## 延后与记账（诚实，非遗漏）

以下需 Change-enum/IndexSink 大手术或属边界/perf，本轮不塞进已很长的会话、避免在崩溃安全路径上仓促改动，均清楚标注留后续：

- **M14 改主键 UPDATE 留幽灵行**：`PgMessage::Update` 已带 `key`/`old`（可取旧 gid），但发一条 Update 需同时产 Delete(old)+Upsert(new)，与 H3 的 Refetch 路径交织需重构 `map` 的多变更发射。**且极罕见**：本系统 chunk 经 `upsert_doc`(DELETE+INSERT) 替换，从不 PK-UPDATE；仅外部直改 PG 主键才触发。
- **M13 TRUNCATE 被吞**：需新增 `IndexSink::apply_clear`（清整表派生索引）+ Change 变体 + 引擎实现。TRUNCATE 是罕见运维动作。
- **M10 IndexSink 无原子性**：text.upsert 成功后嵌入失败留半状态。需引擎 apply_upsert 的事务式回滚/暂存，较大重构。
- **M11 逐条同步嵌入阻塞**：perf，需并发/批处理 + bench 佐证。
- **M15 LSN 水位契约与 pgoutput 语义不符**：生产已绕开（每拍 `Applier::new(Lsn(0))`），属公开 API 文档澄清项。
