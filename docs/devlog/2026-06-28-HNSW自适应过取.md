# devlog 2026-06-28 — HNSW 自适应过取（强过滤召回兜底，迭代④）

> 上游：[后续开发计划-三条线](../plans/2026-06-28-后续开发计划-三条线.md) 第①条线「HNSW filtered-traversal」。spec 回写 [15-vector](../specs/15-vector.md) §6。

## 问题（真 bug，触不变量 #5）

HNSW 后端 `search` 是"固定过取 + 后过滤"：向 HNSW 要 `k×over_fetch`（默认 80）候选，再用 `Filter::eval`/`AclFilter::visible` 精确后过滤。**强选择性 filter 下召回崩**——若只有 ~1.7% 条目匹配 filter（如 1500 条里 25 条 `kind=table`），80 个 ANN 候选里平均只有 ~1.3 条命中 filter → 远不足 `k=10` → 召回 ~0.13。这违反不变量 #5"选择性过滤不掉召回"（HNSW 档此前是已知限制）。

## 做什么（自适应过取 = filtered-traversal 的务实形式）

有 `filter`/`acl` 时，若过滤+重排后结果不足 `k`，就**翻倍 `want`（并把 `ef` 抬到 `want`）重搜**，上限=全集 `total`：

```
needs_escalation = filter.is_some() || acl.is_some()
want = k * over_fetch
loop:
    s = hnsw.search(qcode, want, ef=max(ef_search,want)) → 后过滤 + f32 重排
    if !needs_escalation || s.len() >= k || want >= total: break s
    want = (want * 4).min(total)
```

- 无 filter/acl → 单发，纯 ANN 近似，不升级（零额外开销）。
- 最坏（极选择性 filter）→ `want` 升到 `total` = 对 filter 精确的全扫（正确性优先于速度）。
- `qcode = quantize_u8(q)` 提到循环外，避免重复量化。
- **确定**：同输入 → 同升级路径 → 同结果（HNSW 本身近似抖动不变）。

## 怎么验证（本环境）

- `adaptive_overfetch_selective_filter_recall`：1500 条（>brute 回退阈值 1000 → 真走 HNSW）、~25 条 `kind=table`，filter=`kind=table`、k=10、30 查询，对暴力 filtered ground truth 比 recall。**断言 ≥0.85**（实测远高；对比固定过取理论 ~0.13）。
- 既有 `recall_vs_brute_force`（无过滤单发路径不变）、`filter_aware_acl_exact`（精度/ACL 仍精确）均绿。
- **收口三绿**：fmt + clippy -D warnings + test --workspace（30 套件全绿）。

## 取舍 / 诚实记账

- 极选择性 filter 下升级到全扫 = `O(N)`，失去 ANN 亚线性——这是"固定过取 + 后过滤"型 filtered-ANN 的固有代价，自适应只在**需要时**才付（普通过滤命中率高时第一发即够、零升级）。
- **真·图内 filtered-traversal**（遍历时即按 filter 剪枝、免重搜，如 Qdrant/Weaviate 的 payload 过滤）更优，但需 `hnsw_rs` 暴露遍历钩子 → 下一迭代。
- 默认仍小集合回退暴力（≤1000 精确）；本改动只影响 HNSW 大集合 + 带过滤的查询。
