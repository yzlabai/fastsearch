## 背景

看板 A 组唯一未 `[x]` 项。`TextIndexConfig.k1/b` 早已留字段但**不生效**——Tantivy 0.26.1 把
BM25 的 `K1=1.2/B=0.75` 写死成 `const`（`query/bm25.rs`），`Bm25Weight` 不暴露注入入口

"beat ParadeDB 的 k1/b 可调"是 text 模块卖点，故必须真做。

## 做什么 / 怎么实现

**匹配与排序解耦**（IR 里 matching vs ranking 的正交分解）：

- **匹配不变**：哪些 doc 命中仍由 Tantivy 全权负责——boolean/短语/邻近/末词前缀/filter/ACL 一律照旧，召回语义零变化。
- **排序自算**：新 [`crates/fastsearch-text/src/bm25.rs`](../../crates/fastsearch-text/src/bm25.rs) `score_candidates`——对 Tantivy 召回的候选集，用配置 `k1/b` **自算 BM25 重排**。公式逐项对齐 Tantivy（`score = Σ_term boost·idf·tf(k1+1)/(tf+k1(1-b+b·dl/avgdl))`，`idf=ln(1+(N-df+0.5)/(df+0.5))`，`dl`=量化 fieldnorm 经 `FieldNormReader::fieldnorm` 取，与 Tantivy 同源；`avgdl=total_num_tokens/N`）。多字段（text + heading×`heading_boost`）求和，复刻 `set_field_boost` 结构。
- **默认零改动**：仅当 `k1/b` 偏离 (1.2,0.75) 才走自算路径（`custom_params_active`）；默认值仍走 Tantivy 原生 → **golden 门禁不受影响**（已验证）。自算时放宽候选窗口 `k*8`，并按 `(分降序, DocAddress 升序)` 确定性重排。
- **取词**：`query.query_terms()` 收集落在 text/heading 字段的项（filter/ACL 项在别字段，跳过），天然覆盖短语构成词；纯前缀 `RegexQuery` 不产词 → 那些 doc 回退原生分。
- **取 tf**：按 segment 分组候选、`read_postings(WithFreqs)` + 前向 `advance` 到目标 doc（postings 只能单向，候选先排序）。

**配置面**：server `main.rs` 加 `FASTSEARCH_BM25_K1`/`FASTSEARCH_BM25_B`（env_f32 解析，非法值 warn 回退默认），与 `FASTSEARCH_TOKENIZER` 同构。

## 怎么验证

- **+3 单测**（`cargo test -p fastsearch-text bm25`，全绿）：
  - `bm25_b_length_norm_takes_effect`：短文 vs 长文高 tf——`b=0` 长文(高 tf)居首、`b=1` 短文反超，**排序翻转**证明 `b` 真生效。
  - `bm25_k1_changes_score`：同一单文档，`k1=0.5` vs `2.5` 实际分不同（>1e-3）证明 `k1` 真生效。
  - `custom_bm25_near_default_matches_native_order`：≈默认参数(偏移 >EPSILON 触发自算)排序 == 原生默认排序 → 自算公式与 Tantivy 对齐。
- **收口三绿**：`cargo fmt --all --check` + `cargo clippy --workspace --all-targets -D warnings` + `cargo test --workspace`（含 **3 个 golden 相关性门禁** `no_regression_against_baseline`/`en_no_regression`/`multimodal_no_regression` 全绿——默认路径未变，零回归）。
- **server 实跑**（改检索行为，按 DEV_SPEC 再跑 server）：`FASTSEARCH_BM25_B=0.0 FASTSEARCH_BM25_K1=2.0` 启动 → healthz 200；非法 `FASTSEARCH_BM25_B=oops` → warn 回退 0.75 仍正常启动。
