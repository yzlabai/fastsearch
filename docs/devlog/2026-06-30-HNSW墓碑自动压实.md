# HNSW 墓碑自动压实

> 2026-06-30｜crate `fastsearch-vector`｜spec [15-vector §6](../specs/15-vector.md)｜计划 [下一步改进路线图 §1.2](../plans/2026-06-30-下一步改进路线图.md)

## 背景

HNSW 后端删除/更新只置墓碑（`entries[id]=None`、向量仍留图中，检索时跳过），此前仅 `save`→`load` 或 `clear` 能回收。长生命周期进程在高频 upsert/delete 下 `entries`/图**无界增长**——触不变量 #6（墓碑增长项），原 spec 列为「下一迭代」。

## 做了什么

- `HnswVectorIndex::compact()`：丢弃墓碑，用**活条目**原地重建图 + 稠密 id 映射（绕 hnsw_rs 无原地删除）。等价 `save`→`load` 的活集重建，但全程内存、不落盘。`e.vector` 存入即归一化，重建直接 `quantize_u8` 重插，无需再归一化。
- `dead_count()`：墓碑数 = 槽位 - 活条目（introspection / 测试 / 未来 metrics）。
- `maybe_compact()`：删除/更新后触发，条件「总槽位 > `COMPACT_MIN_TOTAL`(32) 且墓碑过半（dead > live）」。**过半才压实** → 摊还 O(1) 重建（仿动态数组倍增）。
- 接线：`delete` / `delete_doc` / `upsert`（仅 `was_update` 分支，纯新增 dead=0 永不触发，不扰 bulk load）。

## 取舍

- **阈值**：dead > live（>50% 墓碑）保守且摊还代价低；`COMPACT_MIN_TOTAL=32` 避免微小索引在增删抖动中反复重建。
- **语义不变**：活集与检索结果不变（图内部 id 重排；HNSW 近似抖动本就是其固有性质，确定档用默认暴力 `MemVectorIndex`）。

## 验证

- +2 单测：`auto_compaction_reclaims_tombstones`（插 40 删 21 跨阈值 → 槽位回落到活条目数 19、墓碑清零、活条目仍可检索）、`manual_compact_preserves_live_set`（小集合暴力档压实前后 top-k 完全一致、已删 gid 不复现）。
- 收口三绿：`cargo fmt --all --check` + `cargo clippy --workspace --all-targets -- -D warnings` + `cargo test --workspace`（vector crate 32→34 测试）。
