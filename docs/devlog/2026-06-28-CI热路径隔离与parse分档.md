# devlog 2026-06-28 — CI 热路径隔离 + parse 分档门禁（迭代⑤）

> 上游：[后续开发计划-三条线](../plans/2026-06-28-后续开发计划-三条线.md) 第①条线「CI 矩阵」；docparse 融合 DF3.1/DF6.1。

## 做什么

给 `.github/workflows/ci.yml` 加两个 job，把两条此前只靠人工的约束变成 CI 硬门禁：

1. **`hot-path-isolation`**（DF3.1）：对 `fastsearch-server/engine/core/text/vector` 五个**搜索热路径** crate 跑 `cargo tree -p <c> -e normal | grep -qi docparse`，命中即红。守"解析能力 feature 门控、不进检索热路径"（融合 Option B 的隔离约束）——谁不慎把 docparse 传递依赖拖进热路径，CI 立刻变红。
2. **`parse-feature`**（DF6.1）：`cargo clippy -p fastsearch-cli --features parse -D warnings` + `cargo test -p fastsearch-cli --features parse`。把多格式摄取分档纳入常态门禁（此前默认 lean job 不编译 `parse`）。

lean job（默认 workspace fmt/clippy/test + PG 集成）保持不变、必跑。

## 怎么验证（本环境）

逐条本地实跑 CI 将执行的命令：
- 五热路径 crate `cargo tree -e normal` → **均 0 docparse**。
- `cargo test -p fastsearch-cli --features parse` → **11 测试绿**。
- `cargo clippy -p fastsearch-cli --features parse -D warnings` → **零警告**。
- `ci.yml` YAML 语法 ruby `YAML.load_file` 校验通过。

（CI job 本身在 GitHub Actions 跑 = `待运行验证`，但每条 step 命令已本地证过。）
