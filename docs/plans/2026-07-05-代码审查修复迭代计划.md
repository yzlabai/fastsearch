## R1 · 正确性急救（① 本机即可收口，P0）

> 这批全部是"启用某功能反而劣化"或"单请求即致命"。改动都小，务必**先补能复现问题的失败测试再修**（多数问题正因无该测试而长期潜伏）。

### R1.1 rerank 补全候选文本 + CJK 分词 —— 修 H1

- **做什么**：(A) rerank 前对 `text_map` 缺失的命中（向量独有）补 `text.stored_text(gid)`（引擎已有该方法，more_like_this 在用）；(B) `rerank::tokenize` 对 CJK 退化为字符 bigram（或接入已在树内的 jieba），使中文候选不再整句成单 token。
- **为什么**：H1（[review §H1](../reviews/2026-07-05-代码审查报告.md)）。现状 `mode=Vector`+rerank 全候选空文本 → 退化 gid 升序丢弃向量排名；中文候选 Jaccard 恒 0 → 同样退化 gid 序。**请求 rerank 反而摧毁排序**，中文/向量主场景必现。
- **验收**：+2 回归测试——(a) `mode=Vector`+rerank，断言排序随向量分变化、非 gid 序；(b) CJK 查询 rerank，断言相关候选分 > 无关候选。三绿 + 实跑 server 中文检索确认。
- **落点**：[engine/lib.rs:1562-1637](../../crates/fastsearch-engine/src/lib.rs#L1562)、[rerank/lib.rs:20-25](../../crates/fastsearch-rerank/src/lib.rs#L20)；回写 [spec 21-rerank](../specs/21-rerank.md)。

### R1.2 浮点过滤值保超集 + Eq/Ne 数值互比 —— 修 H2

- **做什么**：翻译端 [query_build.rs](../../crates/fastsearch-text/src/query_build.rs) 的 `exact_eq`/`exact_range` 对 `Float` 一律返回 `None`（退 AllQuery 交后过滤），或 `Lt` 用 `Excluded(f.ceil())`、`Gt` 用 `Included(f.floor()+1)` 等保超集的边界；求值端 [core/filter.rs:76-77](../../crates/fastsearch-core/src/filter.rs#L76) 的 `Eq`/`Ne`/`In` 对数值走 `as_f64` 互比（与 `Gt/Gte/Lt/Lte` 一致），使 `Int(6) == Float(6.0)`。
- **为什么**：H2。现状 `{"lt":["page",5.5]}` 漏掉 page=5（截断成 Excluded(5)）、`{"eq":["page",10.0]}` 命中恒 0，且 `Eq ≠ (Gte∧Lte)`。数值过滤经 JSON 传入即静默丢结果。**这是全部后端翻译的语义锚点，改完两端都要守**（不变量 #5）。
- **验收**：core 补 Eq/Ne/In 的 Int↔Float 互比单测；text 补 property 风格测试——随机 Filter（含 Float 值）断言 `translate 召回集 ⊇ eval 命中集`。已复现的 `Lt(page,5.5)`→含 page=5、`Ne(page,6.0)`→两条 作为定点回归。三绿。
- **落点**：core/filter.rs、text/query_build.rs；回写 [spec 10-core](../specs/10-core.md) 过滤语义节 + [spec 11-text](../specs/11-text.md)。

### R1.3 `SearchRequest::validate` 给 top_k/candidates 设上界 —— 修 H4

- **做什么**：[core/query.rs:123-131](../../crates/fastsearch-core/src/query.rs#L123) 增加上界校验（如 `top_k ≤ 10_000`、`candidates ≤ 100_000`，常量可配），拒绝而非放行；顺带修 [text/lib.rs:349](../../crates/fastsearch-text/src/lib.rs#L349) `k*8` 的整数溢出（`saturating_mul`）。
- **为什么**：H4。`candidates` 直达 `Vec::with_capacity` 无裁剪，`POST {"top_k":1,"candidates":1e11}` → `handle_alloc_error` 进程 **abort（不可 unwind）** 打崩多租户服务。`/v1/similar` 同理。
- **验收**：+回归测试断言超大 `candidates`/`top_k` 返 `InvalidRequest`（400）而非 abort；三绿 + 实跑 server 发超大请求得 400。
- **落点**：core/query.rs、text/lib.rs；回写 [spec 10-core](../specs/10-core.md) validate 契约。

### R1.4 HNSW 墓碑聚簇触发过取 —— 修 H5

- **落点**：hnsw.rs；回写 [spec 15-vector §6](../specs/15-vector.md)。

## R3 · 安全加固（① 本机即可收口，P1）

> 前提正面结论：**所有检索入口 ACL 强制注入经逐一核查无遗漏**，object token 绑定扎实。以下为其余问题，均可纯 Rust 单测。

### R3.1 `/v1/similar` 种子加 ACL 校验 —— 修 M19

- **做什么**：[engine/lib.rs:1453](../../crates/fastsearch-engine/src/lib.rs#L1453) `more_like_this` 取种子正文前对种子 gid 做 `AclFilter::visible` 校验（对齐 `resolve_citation`），不可见则返回空/404。
- **为什么**：M19。现跨租户 `{"citation_id":"kb:<A租户doc>:1"}` 用 A 的机密正文构造 MLT → 跨租户存在性预言机 + 语义相似性泄露。
- **验收**：+跨租户种子测试（B 用 A 的 citation → 空结果，不泄露存在性）。三绿。**这是新检索行为，务必带越权用例**（不变量 #3）。

### R3.2 HMAC 签名串消除分隔符歧义 —— 修 M20

- **做什么**：[server/lib.rs:84](../../crates/fastsearch-server/src/lib.rs#L84) 的 `format!("{cid}|{exp}|{ct}")` 改为长度前缀编码或各字段先独立 hash 再签（消除 `cid`/`ct` 内嵌 `|` 的规范化歧义）；顺带对 `media_type`/`doc_id` 做字符白名单。
- **为什么**：M20。现可构造 `(cid1,ct1)` 与 `(cid2,ct2)` 重算消息逐字节相等使签名跨对复用（利用受 `GlobalId::parse` 约束较窄，但真实存在）。
- **验收**：+测试断言含 `|` 的 cid/ct 不产生签名碰撞。三绿。

### R3.3 限流桶有界 + 移到鉴权后 —— 修 M21

- **做什么**：[server/lib.rs:294-318](../../crates/fastsearch-server/src/lib.rs#L294) 限流桶加 LRU/TTL 逐出上限；`allow()` 调用移到 `principal_from_headers` **之后**、按已认证 principal 限流（而非客户端可控原始 token）。
- **为什么**：M21。现桶只增不逐出且限流在鉴权前 → 未认证攻击者每次伪造不同 token 建永不回收的桶，**开启限流反而引入内存 DoS**。
- **验收**：+测试断言大量不同伪造 token 不致桶无界增长；限流对已认证 principal 生效。三绿。

### R3.4 集合注册表租户隔离 —— 修 M22

- **做什么**：[server/lib.rs:2154-2206](../../crates/fastsearch-server/src/lib.rs#L2154) `create/get/list_collections` 增租户归属与过滤（创建记 principal 的 tenant，读/列表按 tenant 过滤，跨租户覆盖拒绝）。
- **为什么**：M22。现任意认证用户可覆盖他人集合维度（针对性 DoS）或枚举全部集合名。
- **验收**：+跨租户测试（B 不能覆盖/枚举 A 的集合）。三绿。

### R3.5 inline 字节响应头收敛 + 5xx 错误串脱敏 —— 修低危 XSS/信息泄露

- **做什么**：`serve_inline_bytes`/`asset` 加 `X-Content-Type-Options: nosniff` + `Content-Disposition: attachment`（或 MIME 白名单）；多处 `(INTERNAL_SERVER_ERROR, e.to_string())` 改为固定文案 + 服务端日志记详情。
- **为什么**：review §四（XSS 缓解语境为 header 型 API Key、面较小；5xx 可能泄露表名/SQL 片段）。
- **验收**：+响应头测试；错误响应不含内部细节。三绿。

> **R3 进度（2026-07-05 全部落地 ✅）**：M19（similar 种子 ACL）· M20（HMAC framing）· M21（限流有界+移后）· M22（集合租户隔离）· R3.5（XSS 白名单 + nosniff + PG 5xx 脱敏，MIME 白名单法保留 inline `<img>`）。每项负控确认，收口三绿 **268→273**（+5）。[devlog](../devlog/2026-07-05-代码审查修复-R3-安全加固.md)。下一步转 **R4**（CDC/PG，纯函数本机单测 + e2e 标 gated 待 Docker）。

## R4 · CDC / PG 健壮性（② 代码本机可写+单测，e2e 需 Docker PG → 标 gated，P1）

### R4.1 CDC 毒丸修复 + 失效架构断言回写 —— 修 H3（**最高优先**）

- **做什么**：(a) [replication.rs:190-213](../../crates/fastsearch-sync/src/replication.rs#L190) 的 `UnchangedToast('u')` 分支不再折叠成 None-then-Err：对必填列（text）遇 'u' 应**保留旧值/跳过重嵌**（该行 text 未变），对可空列遇 'u' 保持旧值而非静默丢字段；(b) 引入毒丸跳过/死信，单条持续失败不阻塞整批与 slot advance；(c) 回写文档去掉 [sql.rs:59-68](../../crates/fastsearch-pg/src/sql.rs#L59) 一带"列清单 publication → 写穿不产生事件"的**已被实测证伪**的结构性断言，明确防环只靠 `IS DISTINCT FROM` 幂等守卫。
- **为什么**：H3。真容器实测：大 text 行（TOAST）的任何"未改 text 的 UPDATE"（含 `set_embedding` 写穿自身）复制出 text='u' → 整批 Err → slot 永不 advance → **CDC 永久卡死**；即"开启 CDC+嵌入摄取一个大 chunk 的正常主流程会自我卡死"。
- **验收**：`map`/`row_to_chunk` 补 `UnchangedToast` 的 byte 级单测（本机可收口）；毒丸跳过单测；Docker PG e2e——摄取 >8KB text chunk 后 CDC 不卡死（gated `待运行验证`）。
- **落点**：sync/replication.rs、pg/sql.rs、CLAUDE.md、spec 13-sync/12-pg。
