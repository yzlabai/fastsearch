## 0. 决定性事实（先看这个，它改变结论）

docparse 的**重依赖是干净隔离的**，可按需取舍——这是整份评估的支点：

| crate | 重依赖 | fastsearch 是否需要 |
|---|---|---|
| **`docparse-core`** | **无**（serde/anyhow/encoding_rs/chardetng/rayon/libc） | ✅ 必需——`Chunk`/IR/分块/阅读序/大纲在此，**极轻** |
| `docparse-pdf` | `lopdf` + `hayro-ccitt/jbig2/jpeg2000`（图像编解码，中等） | ✅ 想要 PDF 解析则需；**不含 ONNX** |
| `docparse-docx/html/xlsx/pptx/md/csv/srt/tex/eml/img/adoc` | 各自轻量解析库 | ✅ 按需选 |
| **`docparse-ocr`** | **`tract-onnx` + 自维护 vendored tract 补丁**（最重） | ⚠ 仅扫描件 OCR 才需 → **opt-in** |
| **`docparse-raster`** | **`hayro`** 整页渲染（重） | ⚠ 仅需整页栅格化才需 → **opt-in** |
| `docparse-vlm` | `ureq`（HTTP，轻） | ⚠ 仅 VLM caption 才需 → **opt-in** |

**结论**：fastsearch 可以**只依赖 `docparse-core` + 选定解析器**拿到全部"解析→chunk"能力，**完全不碰 ONNX/渲染**；OCR/栅格/VLM 作为 cargo feature 留给"全功能摄取二进制"。→ **Option A（整体无条件并入、ONNX 进默认构建）没有必要；Option B 是干净的目标态。**

## 1. 三个方案

### Option A — 整体并入，单 workspace 单默认二进制
所有 17 crate 进 fastsearch workspace，默认构建即含 OCR/渲染。
- ✅ 一个仓、一套版本、schema 类型级共享。
- ❌ **ONNX + vendored tract 补丁 + hayro 进默认搜索构建**：`fastsearch-server` 二进制膨胀、CI 变慢、引入 tract 补丁维护负担。把 ML/渲染塞进一个 PG 检索引擎，违背"精简单二进制"基调。
- **判定：不推荐**（§0 表明无此必要）。

### Option B — 单仓 monorepo + feature 门控（推荐）⭐
docparse 源码经 **`git subtree`** 并入本仓（保历史）；但**依赖按粒度取**：
- `fastsearch-core` / `fastsearch-server`（检索热路径）**不依赖任何 docparse crate**——保持精简。
- 新 `fastsearch-ingest`（或扩现有 `fastsearch-cli`）依赖 `docparse-core` + 选定解析器；
- 重增强器 **feature 门控**：`--features parse`（core+解析器，无 ONNX）/ `parse-ocr`（加 tract ONNX）/ `parse-raster` / `parse-vlm`。
- ✅ 单仓、schema 类型级共享、单"全功能"摄取二进制可选；搜索服务零膨胀。
- ⚠ 一仓维护两套生命周期；需调和两个 `Chunk` 类型（见 §2）；vendored tract 补丁随 OCR 进树（但只在开 `parse-ocr` 时编译）。
- **判定：推荐**——既得统一产品，又不污染检索热路径。

### Option C — 库依赖（不并源码，过渡可选）
保留 docparse 仓，fastsearch 经 **git 依赖** `docparse-core`（+按需解析器 crate）。
- ✅ 最轻、最可逆、零源码复制；schema 经真实类型共享（消除手工锁步）。
- ❌ 与"并入后不再独立"定调不符——长期还是两仓两份 CI/发版；git 依赖要钉 commit/tag。
- **判定：作为 B 的过渡垫脚石**——可先 C 验证依赖关系跑通，再 subtree 收成 B。

## 2. 关键技术点：两个 `Chunk` 类型怎么调和

`docparse_core::Chunk` 与 `fastsearch_core::Chunk` **几乎同构**，但 fastsearch 多了 `tenant`/`acl`（且本轮多模态正在加 `media`/`time`/Audio·Video kind）。`tenant`/`acl` 本质是**摄取期注入的元数据，不是解析关注点**。

**方案**：保留 fastsearch 自己的 `core::Chunk`（真源 schema），在摄取边界写一个**编译期检查的适配器** `From<docparse_core::Chunk> → fastsearch_core::Chunk`（补 tenant/acl/media 映射）。
- 好处：解析与权限/真源 schema 解耦；"跨仓手工锁步"退化为"**单仓内一个 `From` 函数,改 schema 编译即报错**"——这正是融合要消除的痛点。
- docparse 的 `ImageMeta{file,data_base64,media_type,caption,caption_source}` → fastsearch `MediaRef`（`data_base64→AssetPointer::Inline`、`file→Object`）的映射也落在此适配器（衔接[多模态功能设计 §2.2](2026-06-25-多模态功能设计与开发计划.md)）。

> 注：多模态共享字段（`time`/Audio·Video kind）仍建议**两边同源**——并入后即"同仓同 PR"，不再是跨仓阻塞（消解 [职责划分 §3](2026-06-25-多模态职责划分-docparse与fastsearch.md) 的硬协调项）。

## 3. 依赖 / 二进制 / CI 影响（Option B）

| 维度 | 影响 | 缓解 |
|---|---|---|
| **搜索热路径**（`core`/`server`/`vector`/`text`/`engine`） | **零**——不依赖 docparse | 依赖图上严格隔离 |
| **摄取二进制**（`--features parse`） | +`docparse-core`+解析器（lopdf 等中等量） | 无 ONNX；可接受 |
| **全功能二进制**（`parse-ocr`） | +tract-onnx + vendored tract（重，编译慢） | 仅此 feature 编译；CI 拆 job |
| **CI 时长** | 默认 job 不变；新增 `parse*` feature job | 矩阵化：lean job 必跑，重 feature job 可并行/夜跑 |
| **仓库体积** | subtree 只带**源码**（docparse `.git` 25M 级，非 45G——45G 是 `target/`+模型） | `.gitignore` 模型/target；ONNX 模型走下载/LFS 不进仓 |
| **workspace 依赖统一** | docparse 与 fastsearch 的 `[workspace.dependencies]` 版本需合并去重 | 一次性对齐（serde/anyhow/rayon/tokio/axum 多半同源） |
| **vendored tract 补丁** | 随 `docparse-ocr` 进树，维护责任转移到本仓 | 仅 `parse-ocr` 用户承担；不开则零负担 |
| **许可（分发前）** | Apache-2.0 兼容；但 **lindera/jieba CC-BY-SA 字典**（C1）、**ONNX 模型许可** 需审 | 入 `parse-ocr`/分词 feature 时审；纯检索分发不受影响 |
| **两套 face 重叠** | docparse 自带 CLI/MCP/REST（axum）、fastsearch 也有 REST | 并入后**收敛 face**：检索 REST 留 fastsearch-server；解析作 ingest 子命令/库,不重复起 axum |

## 4. 迁移步骤（Option B，建议执行序）

> 每步可独立收口（fmt+clippy+test 三绿），可在任一步暂停/回退。

1. **C 试探（低风险预演）**：fastsearch 加 `docparse-core` 为 git 依赖，写 `From<docparse::Chunk>` 适配器 + 一条 `fastsearch ingest <file>`（仅 PDF）打通"解析→适配→现有 index 路径"。验证依赖/类型调和无坑。
2. **subtree 并入**：`git subtree add --prefix=vendor/docparse git@github.com:yzlabai/docparse-rs.git main`（保历史）；`.gitignore` 其 `target/`、模型、test 大文件。
3. **workspace 合并**：把 docparse crates 纳入本仓 `members`，合并 `[workspace.dependencies]` 去重对齐版本；改 git 依赖为 path 依赖。
4. **依赖隔离校验**：确认 `core/server/...` 依赖图**不含**任何 docparse crate（加 CI 断言/`cargo tree` 检查）。
5. **feature 门控**：`fastsearch-cli`/`-ingest` 定义 `parse`/`parse-ocr`/`parse-raster`/`parse-vlm`；默认仅 `parse`。
6. **face 收敛**：解析能力以 `ingest` 子命令 + 库 API 暴露；停用 docparse 自带 axum REST/MCP（或择一保留），避免双服务。
7. **CI 矩阵 + 许可审**：lean job（必跑）+ `parse*` feature job；分发前过 CC-BY-SA 字典 / ONNX 模型许可。
8. **文档收口**：更新 CLAUDE.md 架构大图（docparse 从"上游孪生"变"本仓 ingest 子系统")、迭代看板加并入项、回写职责划分文档（跨仓锁步项已消解）。

**回退**：步骤 1–2 之间——删 git 依赖即恢复；步骤 2 之后——`git subtree` 是普通 commit，`git revert`/删 `vendor/docparse` 目录即回退；因热路径零依赖，回退不触及检索代码。

## 5. 推荐与待决策点

**推荐：Option B（subtree 并入 + feature 门控 + 摄取期适配器），以 Option C 作第 1 步预演。**
理由：契合"docparse 不再独立"的定调拿到统一产品与单仓 schema；§0 证明可把 ONNX/渲染挡在搜索热路径外，不付"精简单二进制"的代价;迁移分步可逆、回退不碰检索。

**待你拍板：**
- **D-1 二进制形态**：要"一个 `localkb` 全功能二进制（含解析,feature 开关）"，还是"`fastsearch-server`(检索) + `fastsearch-ingest`(解析) 两个二进制"？（影响 face 收敛与默认 feature）
- **D-2 解析器范围**：先并 `docparse-core`+`pdf`（最小可用），还是一次并全部 12 解析器？
- **D-3 OCR/渲染/VLM**：是否纳入（开 `parse-ocr` 等）？纳入则承接 vendored tract 维护 + 模型分发/许可；不纳入则扫描件/整页视觉检索能力暂缺。
- **D-4 时机**：现在并（多模态 MM1 刚落地,适配器正好衔接 MediaRef 映射），还是等 M0 多模态稳定后再并？

> 一句话：先按 **C** 花小成本把"解析→适配→索引"跑通验证，再 **subtree 收成 B**；A 不必做。
