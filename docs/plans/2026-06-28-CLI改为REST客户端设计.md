# CLI 改为 REST 客户端 — 设计计划

> 状态：**设计（未实施）**｜日期：2026-06-28｜上游：[后续开发计划-三条线](2026-06-28-后续开发计划-三条线.md)、[迭代循环看板](2026-06-25-迭代循环.md)、[17-cli spec](../specs/17-cli.md)。
>
> **决策来源（用户拍板）**：① CLI **完全改为纯 REST 客户端**——删除嵌入式引擎（本地 Tantivy 索引/本地搜索/eval-via-engine），所有检索/写入走 server REST，与 `clients/{python,ts}` 同模型。② **"喂文件夹来检索"必须保留**——但实现从"离线嵌入式"改为"**客户端遍历+分块+POST /v1/index**"（经 server 索引，反而**白嫖全套混合检索**）。
>
> **代码现状是真源**：签名/契约为设计意图，落地以代码为准并回写。未真机验证项标 `待运行验证`。

---

## 1. 需求三件套

- **要做什么**：把 `fastsearch` CLI 从"进程内嵌引擎、读写本地索引"改为"**HTTP 客户端**"——调 server 的 `/v1/search`、`/v1/index`、`/v1/similar`、`/v1/asset`。保留"喂文件夹→检索"，改为客户端把文件夹分块后上传 server。
- **为什么**：CLI 当前是**关键词-only**（不算嵌入、向量不跨进程落盘，见 [B4b]）。改为 client 后，检索/嵌入/落盘全归 server，CLI **白嫖全套混合检索 + ACL + 真语义**，且与四张脸里"REST 是服务、SDK 是客户端"一致；消除 CLI 自维护引擎/嵌入器/落盘的重复与缺口。
- **验收**：① `search`/`similar`/`index`/`ingest`/`index-dir`/`eval` 全部经 server REST 工作；② **喂文件夹**：`index-dir <dir>` 遍历→分块→POST→`search` 能检索到（混合）；③ 默认构建（无 `parse`）是**零引擎依赖的瘦客户端**；④ 鉴权/错误/超时有清晰反馈；⑤ 收口三绿。

---

## 2. 范围与不做什么

**纳入**：
- CLI 重写为 REST 客户端：薄 HTTP 层（`ureq`）+ 命令→端点映射 + 配置（`--server`/`--key`）。
- 保留客户端侧**分块/解析**：`index`（chunks.json）、`ingest`（docparse 单文件，`parse` feature）、`index-dir`（遍历 .md/.txt + `parse` 时含多格式）→ 统一 POST `/v1/index`。
- `eval` 客户端化：golden 语料 POST 入库 → 逐查询 POST `/v1/search` → 用 `fastsearch-eval`（纯指标）算 nDCG/recall/MRR + 回归门禁。

**不做（本期）**：
- 不在 server 端加"上传文件解析"端点（**解析留客户端**，守"搜索热路径零 docparse"不变量 + 已加的 CI 门禁）。
- 不引入新鉴权/连接复用池（单发 `ureq`，够用）。
- 不保留离线嵌入式档（用户已定**完全替换**；旧 `Mem/HNSW` 引擎仍在 `fastsearch-engine`，server 用）。

## 3. 设计决策（拍板）

| # | 决策点 | 选定方案 | 理由 |
|---|---|---|---|
| C1 | CLI 形态 | **纯 REST 客户端**（删嵌入式引擎/本地索引/本地搜索） | 用户拍板；与 SDK 一致；白嫖 server 全套混合 |
| C2 | 喂文件夹 | **客户端遍历+分块+POST /v1/index**（非离线嵌入式） | 用户要求保留该功能；经 server 反得混合检索；分块逻辑（`chunk_text`）本就纯函数、无引擎 |
| C3 | 解析放哪 | **客户端**（docparse `parse` feature；`ingest`/`index-dir` 本地解析→POST chunks） | 守"搜索热路径零 docparse"+ 已加 CI 门禁；server 不碰解析 |
| C4 | HTTP 库 | **`ureq`**（已在 workspace，embed 用） | 轻量、阻塞式（CLI 同步够用）、无 async 运行时 |
| C5 | 类型复用 | 复用 **`fastsearch-core`**（`Chunk`/`SearchRequest`/`Filter`/`GlobalId`）做请求体/响应解析 | core 是**纯类型逻辑、无后端**，不算"嵌引擎"；类型安全、与 server 契约同源 |
| C6 | eval | 客户端跑：语料 POST 入库 → HTTP 查询 → `fastsearch-eval` 纯指标 + `assert_no_regression` | eval crate 纯（仅依赖 core）；保留 CLI eval 便捷 + 回归门禁 |
| C7 | 配置 | `--server`(env `FASTSEARCH_SERVER`，默认 `http://localhost:8642`) + `--key`(env `FASTSEARCH_KEY`) | 与 server `FASTSEARCH_KEYS` 对应；env 优先级低于显式 flag |

## 4. REST 契约（核对 server，落地以代码为准）

- **鉴权**：所有请求头 `Authorization: Bearer <key>`（server `principal_from_headers` 亦接受 `x-api-key`）。
- `POST /v1/search` ← body = `core::SearchRequest` JSON（`query`/`mode`/`top_k`/`candidates`/`filter`/…）→ `{ "hits": [Hit…], "facets": [...] }`。**collection 作用域**用 `filter` 注入 `Eq("collection", …)`（落地核对 text/vector 是否支持 collection 过滤；server search 体本身无 collection 字段）。
- `POST /v1/similar` ← `{ "citation_id": String, "top_k": usize }` → `{ "hits": [Hit…] }`。
- `POST /v1/index` ← `{ "collection": String, "doc_id": String, "chunks": [core::Chunk] }` → `{ "indexed": n }`。
- `GET /v1/asset/{citation_id}` → 字节(inline) / 302(object) / JSON(doc_render)；越权/不存在 404。
- `Hit` 字段：`citation_id`/`score`/`bm25`/`vector`/`rerank`/`doc_id`/`chunk_id`/`page`/`bbox`/`heading_path`/`section_id`/`highlight`/`merged_chunk_ids`/`time`/`media`/`cursor`。

## 5. 命令 → 端点映射

| 命令 | 行为 |
|---|---|
| `search --collection C --query Q [--mode hybrid\|keyword\|vector] [--top-k N] [--kind K] [--page-min/max] [--json]` | 构造 `SearchRequest`（+collection/kind/page filter）→ POST `/v1/search` → 打印 hits（默认人读 / `--json`） |
| `similar --citation-id CID [--top-k N]` | POST `/v1/similar` → 打印 hits |
| `index --collection C --doc-id ID [INPUT\|-]` | 读 chunks.json（数组/NDJSON）→ POST `/v1/index` |
| `ingest --collection C --doc-id ID <FILE>`（`parse`） | docparse 本地解析→适配 chunks → POST `/v1/index` |
| `index-dir --collection C <DIR>` | 遍历 .md/.txt（`parse` 时 + 多格式）→ 每文件 `chunk_text`/解析（doc_id=相对路径）→ POST `/v1/index` |
| `eval --golden G [--baseline B] [--tol T] [--k K] [--mode M]` | golden 语料按 doc 分组 POST 入库 → 逐查询 POST `/v1/search` → `fastsearch-eval` 算指标 + 门禁 |

全命令统一加 `--server`/`--key`（env 兜底）。

## 6. 改造清单（crate：fastsearch-cli 重写）

### 6.1 Cargo.toml
- **删依赖**：`fastsearch-engine`、`fastsearch-text`。
- **加依赖**：`ureq`（workspace；新增到 `[workspace.dependencies]`）。
- **留依赖**：`fastsearch-core`（纯类型）、`fastsearch-eval`（纯指标）、`clap`/`serde`/`serde_json`/`anyhow`；`parse`/`parse-ocr`/`parse-tables` feature 的 docparse-* 不变。

### 6.2 src/lib.rs（重写）
- **新增** `Client { base: String, key: String }`：`post_json(path, body) -> Result<Value>`、`get_bytes(path)`（ureq；非 2xx→带 body 报错；超时）。
- **保留（纯函数，去引擎）**：`chunk_text` / markdown 标题解析 / `collect_text_files` / `build_filter` / docparse 适配（`ingest::from_docparse_chunk`）。
- **改写**：`cmd_search`/`cmd_similar`/`cmd_index`/`cmd_index_dir`/`cmd_eval`/`ingest::cmd_ingest` 的"sink/source"从 `engine.*` 改为 `Client` HTTP；删 `open_engine`/`load_or_init_meta`/`Meta`/`tokenizer` 相关（tokenizer 是 server 侧 index 配置，客户端不管）。
- **响应解析**：`Hit` 轻量 DTO（serde）或直接取 `Value` 字段打印。

### 6.3 src/main.rs（clap 壳）
- 各子命令去 `--data`/`--tokenizer`，加 `--server`/`--key`；`Ingest`/`IndexDir` 的 sink 改 server；`eval` 去 engine。
- 错误：连接失败/鉴权失败/超时 → 友好 stderr + 非零退出。

### 6.4 测试
- 单测（无网）：`chunk_text`/`build_filter`/docparse 适配/响应 JSON 解析（喂构造的 `{hits:[…]}`）保持。
- HTTP 端到端：**起进程内 server（`fastsearch-server` 作 dev-dependency + 内存 engine + 临时 key）或 mock**，跑 `index→search`、`index-dir→search`、`similar`、`eval` 闭环。（落地择一：进程内 axum oneshot 最可复现。）

## 9. 守的不变量 + 风险

- **#3 ACL 不可绕过**：CLI 作客户端，ACL 仍由 server 按 key 注入；客户端无法传/放宽 ✓。
- **搜索热路径零 docparse**：解析留客户端（CLI），server 不引 docparse；CI 门禁已守 ✓。
- **四张脸契约一致**：CLI 现与 python/ts SDK 同走 REST、同契约 ✓。
- **风险**：① CLI 失去离线能力——**用户已确认接受**（喂文件夹改为联网上传；无 server 时 CLI 不可用）；② `index-dir` 大文件夹批量 POST 的吞吐/失败重试——见 §10 调研：研究产品都做**进度 + 失败重试**，本期 `index-dir` 加进度 + 逐文件 continue-on-error + 简单重试；③ ~~server search 体无 collection 字段~~ **已核实**：`collection` 在 text（[query_build.rs:110](../../crates/fastsearch-text/src/query_build.rs#L110)，term 翻译）+ vector（`VecMeta` FieldSource）**两端均可过滤** → CLI `--collection` 用 `Eq("collection",…)` filter 限定，**无需 server 改动** ✓；④ eval 把语料灌进目标 server 会污染其索引——eval **必填 `--collection`** 指向专用/临时集合（文档标注）。

## 10. 相似产品调研 + 推荐（2026-06-28）

调研服务端检索/向量产品的 CLI 形态——**结论：服务端产品的 CLI 普遍是"瘦 HTTP 客户端"**，与本设计一致：

| 产品 | CLI 形态 | 要点 |
|---|---|---|
| **Typesense** | 瘦客户端（typesense-js 之上） | "client 库只是 HTTP API 的薄包装"；CLI 经 server API 管理集合/文档 |
| **Qdrant** | 官方 `qcloud`（Cloud CLI）+ 社区 `qdrant-rest-cli`（每个 REST 端点=一条 typed 命令） | "全部交互经 REST API" |
| **Algolia** | 官方 CLI = 云 API 瘦客户端 | **profile（app+key）**、**NDJSON I/O**、**结构化 JSON 输出（schemaVersion）便于 AI agent** 解析；"为 AI agent 重写了 CLI" |
| **Meilisearch** | 批量导入用 `meilisearch-importer` CLI（上传到 server） | **批量 + 自动重试 + 进度/ETA**；server `--import-dump` 启动加载 |
| **tantivy-cli**（反例） | **嵌入式**（CLI 内嵌引擎、本地磁盘索引；可起小 REST server） | tantivy 是**嵌入式库**定位（"消除网络延迟、随应用数据"）——这是"嵌引擎"档，与服务端产品不同 |

**定位判断**：fastsearch 是**服务端产品**（PG 真源 + server 部署，CLAUDE.md），**不是**嵌入式库（tantivy 那类）。故对标 Typesense/Qdrant/Algolia/Meilisearch → **CLI = 瘦 HTTP 客户端**是业界norm，**支持用户的决策与本计划**。

**调研带来的细化（纳入实施）**：
1. **配置**：`--server`/`--key` + env（Algolia 的 profile 更进一步，留下一迭代）。
2. **NDJSON**：`index` 已收 NDJSON；批量友好。
3. **批量稳健**（Meilisearch-importer）：`index-dir` 加**进度输出 + 逐文件 continue-on-error + 简单重试**（瞬时失败重试 N 次）。
4. **agent 友好**（Algolia）：`--json` 输出干净可解析。
5. **命令 1:1 映射端点**（qdrant-rest-cli）：本计划 §5 即如此。

**推荐：按本计划实施（CLI 纯瘦客户端 + 喂文件夹改客户端上传），纳入上述细化**。
