## 1. 目标与验收

### 1.1 目标

实现图片的全模态检索：

1. **以图搜图**：上传一张图片作为 query，返回视觉相似或语义相近的图片 chunk。
2. **以文搜图**：输入自然语言 query，返回匹配语义的图片 chunk。
3. **图文混合召回**：图片既可因视觉向量命中，也可因 caption、上下文、OCR 文本命中。
4. **S3/MinIO 对象存储接入**：大图/原图走对象存储；输入侧可上传或引用对象，索引侧可读取对象生成视觉向量，结果侧可解析为短时 presigned URL。
5. **安全取图**：搜索结果只返回 citation/media 指针；图片字节经现有 ACL 媒资网关、inline token URL 或 S3/MinIO 短时签名 URL 获取。
6. **保持现有不变量**：PG 是真源，引擎索引可重建；ACL 服务端注入不可绕过；托管 PG 只依赖 pgvector；模型推理由外部 HTTP 服务提供，不引进程内模型依赖。

### 1.2 非目标

1. 不在 Postgres 安装 ParadeDB、Timescale、pg_search 等原生扩展。
2. 不把 CLIP/SigLIP/Jina 模型以 Candle/ort 方式嵌入 fastsearch 进程。
3. 不在本阶段实现 ColPali 多向量 MaxSim；该能力保留为 P2。
4. 不让客户端传 ACL 或绕过 `/v1/search`、`/v1/asset` 的服务端身份注入。

### 1.3 最终验收

1. 使用真实跨模态模型服务，索引 20 张以上图片，文本 query 能召回对应图片，图片 query 能召回同类/同图图片。
2. `POST /v1/search` 同时支持 multipart 图片上传；JSON 形态新增显式 `query_image_base64` 字段，避免把 `Vec<u8>` 的 serde 数字数组误当 base64。
3. `POST /v1/index` / CLI ingest 支持本地图片上传到 S3/MinIO 后入库，也支持直接引用 `s3://bucket/key` 或 `minio://bucket/key`。
4. CLI 支持 `search --image path.png` 和 `search --query "..." --filter modality=image`。
5. 搜索结果里的图片 citation 可通过 `/v1/assets/resolve` 或 `/v1/asset/{cid}` 安全取图；Object 图片返回 S3/MinIO presigned URL，Inline 图片返回 fastsearch token URL。
6. 单测、集成测试、golden eval 均覆盖图→图、文→图、ACL 越权、无模型/维度不匹配错误、S3/MinIO 上传/读取/签名。
7. 收口命令通过：

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

涉及真模型和 PG 的测试使用 env-gated 集成测试，不具备资源时明确标 `待运行验证`。

## 2. 当前代码基线

### 2.1 已具备能力

1. `fastsearch-core` 已有 `ChunkKind::Image`、`Modality::Image`、`MediaRef`、`AssetPointer`、`media_bytes`。
2. `SearchRequest` 已有 `query_image: Option<Vec<u8>>`。
3. `fastsearch-embed` 已有 `EmbedInput::{Text, Image}`、`EmbedCaps`、`embed_multi()`。
4. `HashEmbedder` 已支持确定性图片向量，但 `semantic=false`、`cross_modal=false`，仅用于 CI/基线。
5. `HttpEmbedder` 当前只支持文本，`caps.image=false`、`caps.cross_modal=false`。
6. `fastsearch-engine` 已能在写入侧对无文本 inline 图片走图片向量路由，在查询侧对 `query_image` 嵌图并复用 vector 检索。
7. `fastsearch-server` 已有 `/v1/asset/{cid}`、`/v1/assets/resolve`、`/v1/asset/{cid}/bytes`，具备 ACL 媒资出口。

### 2.2 关键缺口

1. **真多模态 HTTP 嵌入缺口**：`HttpEmbedder` 还不能把图片作为 input 发送到外部模型。
2. **跨模态能力声明缺口**：文→图必须要求 `caps.cross_modal=true`，否则文本向量和图片向量不可比较。
3. **图片对象字节缺口**：当前 MM10 基线只处理 inline `media_bytes`；对象存储图片还不能在索引时取字节嵌入。
4. **REST/CLI 输入缺口**：内部已有 `query_image` 字节字段，但用户侧 JSON 需要显式 `query_image_base64` DTO，且还缺 multipart 上传与 CLI `--image`。
5. **S3/MinIO 对象存储缺口**：已有 `ObjectSigner` 抽象和 asset resolve 控制面，但还缺真实 S3/MinIO signer、reader、writer，以及输入侧上传/引用对象到 `MediaRef::Object` 的完整链路。
6. **评测缺口**：缺真实或可复现的图文 golden，用于证明文→图、图→图质量不回归。

## 3. 总体架构

```text
图片文件/docparse 图片 chunk
        |
        | ingest: caption/text + media_bytes 或 upload/reference S3-MinIO object
        v
Postgres 真源 chunks/media(Object uri)/media_bytes/embedding(pgvector)
        |
        | CDC / snapshot rebuild
        v
fastsearch-engine 派生索引
  - text: caption/OCR/上下文 BM25
  - vector: 跨模态图片/文本同空间向量
        |
        | /v1/search
        v
结果: image chunks + citation/media
        |
        | /v1/assets/resolve or /v1/asset/{cid}
        v
ACL 校验后的 inline token URL / S3-MinIO presigned URL / DocRegion
```

核心原则：**PG 是 chunk 元数据、ACL、对象引用与 embedding 的真源；对象存储是大图/原图字节真源；搜索索引只是派生视图**。Inline 小图字节仍可在 PG `media_bytes`，Object 图片字节不复制进 PG。对象和 PG chunk 生命周期必须同步：原图对象删除时，同步删除对应 PG chunk（或整 doc 替换路径删除该 chunk），随后 CDC 删除派生索引，避免保留不可解析引用。

数据隔离以现有 `collection` 作为集合边界：写入请求必须指定目标 `collection`，检索入口继续通过 `collection` 过滤限定作用域；需要按不同最终用户隔离数据时，由调用方使用不同集合 Id（例如把用户维度编码进 collection 命名）。本方案不新增独立用户 Id 参数，避免形成与 PG/ACL 真源并列的第二套隔离语义。

1. `text` 信号：caption、OCR、相邻段落上下文、文件名等文本表示。
2. `vector` 信号：跨模态模型产生的同空间向量，支持文→图和图→图。

## 4. 数据与索引设计

### 4.1 图片 chunk 形态

图片 chunk 继续使用现有 `Chunk`：

```rust
Chunk {
    kind: ChunkKind::Image,
    text: "...caption/OCR/上下文，可为空...",
    media: Some(MediaRef {
        asset: AssetPointer::Inline | AssetPointer::Object { uri } | AssetPointer::DocRegion { .. },
        media_type: Some("image/png".into()),
        region: Some(bbox),
        caption_source: Some("docparse|vlm|alt|filename".into()),
        ..
    }),
    media_bytes: Some(bytes) // 仅小图/裁图 inline；Object 图片可为空
}
```

`modality=image` 不新增 search 参数，继续走 `Filter::Eq/In("modality", ...)`。

### 4.2 图片向量写入规则

写入/重建时对图片 chunk 采用以下优先级：

1. `media_bytes` 存在且 `embedder.caps().image=true`：用 `EmbedInput::Image(bytes)` 生成图片向量。
2. `AssetPointer::Object` 且配置了 S3/MinIO 对象读取器：从对象存储取字节，用 `EmbedInput::Image(bytes)` 生成图片向量。
3. 无图片字节但 `text` 非空：用 caption/OCR 文本向量作为退化语义向量。
4. 无图片字节且 `text` 为空：不写向量，只进入 text 元数据索引，避免空向量污染 ANN。

当 `caps.cross_modal=false` 时，允许图→图，但默认拒绝文→图的“只搜图片”语义向量路径，避免不可比向量导致伪召回。HashEmbedder 继续只作为确定性基线。

### 4.3 PG schema

现有 `chunks` 表的 `embedding` 仍作为统一向量列，不新增 PG 原生扩展。需要补充或确认：

1. `modality='image'` 正确落库并可 SQL/filter 下推。
2. `media` JSONB 和 `media_bytes` bytea 写入路径覆盖 CLI ingest、REST index、PG `upsert_doc`。
3. `embed_model` 写入真实模型名，例如 `jina-clip-v2@1024`，用于维度漂移诊断和重建。
4. Object 图片需在 `media` JSON 中保存可校验元数据：`uri`、`bucket`、`key`、`provider`、`size`、`sha256`、`etag`、`content_type`、`owner_tenant`。搜索响应不得原样暴露这些内部字段。

如果当前 DDL 已满足，不做 schema 变更，只补测试。

## 5. 嵌入后端实现

### 5.1 扩展配置

在 `fastsearch-embed::EmbedderConfig` 增加多模态配置：

```text
FASTSEARCH_EMBEDDER=openai
FASTSEARCH_EMBED_URL=http://localhost:8080
FASTSEARCH_EMBED_MODEL=jina-clip-v2
FASTSEARCH_EMBED_DIM=1024
FASTSEARCH_EMBED_API_KEY=...
FASTSEARCH_EMBED_IMAGE=true
FASTSEARCH_EMBED_CROSS_MODAL=true
FASTSEARCH_EMBED_IMAGE_INPUT_FORMAT=data_url|base64_object|openai_content
FASTSEARCH_EMBED_TEXT_FIELD=input
```

建议第一阶段优先实现两类协议：

1. **OpenAI-compatible multimodal embeddings**：请求体可表达文本和图片 data URL。
2. **Generic JSON multimodal**：通过配置指定文本/图片字段，兼容自建 embedding gateway。

Ollama 只在确认具体模型和接口能稳定返回图片 embedding 后再开启图片能力，否则保持 `image=false`。

### 5.2 `HttpEmbedder::embed_multi`

实现规则：

1. `EmbedInput::Text`：沿用现有 `embed()` 请求格式。
2. `EmbedInput::Image(bytes)`：
   - 根据 MIME 猜测或调用方传入 media type，组装 `data:image/png;base64,...`。
   - 请求外部模型服务。
   - 校验返回数量和维度。
3. 混批策略：
   - 如果目标协议支持混批，保持输入顺序一次发送。
   - 如果协议不支持文本/图片混批，则按模态拆批，最后按原顺序合并。
4. `caps()` 必须来自配置与启动探测：
   - `image=true` 表示支持图片 input。
   - `cross_modal=true` 表示文本和图片在同一向量空间。
   - `semantic=true` 表示真实语义模型。

启动时可选做一次 probe：用固定小图和文本请求模型，确认维度、数量和错误格式。probe 失败则服务启动失败或降级为 `image=false`，由配置控制。

### 5.3 模型选择建议

优先选择能统一 text/image embedding 的外部模型服务：

1. Jina CLIP / Jina Embeddings 多模态模型。
2. SigLIP/CLIP 兼容 embedding service。
3. Voyage/Cohere 等明确支持 image embeddings 的商业 API。

部署要求：

1. 文本 query 和图片 passage 必须同维同空间。
2. 返回向量必须稳定，不同请求同输入近似一致。
3. 服务必须有超时、限流和错误体，便于 fastsearch 报错。

## 6. 引擎检索实现

### 6.1 查询语义

新增或明确三种查询形态：

1. 文本搜全部：

```json
{"query":"红色柱状图", "mode":"hybrid"}
```

2. 文本搜图片：

```json
{
  "query":"红色柱状图",
  "mode":"hybrid",
  "filter":{"eq":["modality","image"]}
}
```

3. 图片搜图片：

```json
{
  "query":"",
  "mode":"vector",
  "query_image_base64":"<base64 image bytes>",
  "filter":{"eq":["modality","image"]}
}
```

上面 `filter` 示例是当前 `fastsearch-core::Filter` 的真实 serde 形态：externally tagged tuple enum，例如 `Filter::Eq("modality", "image")` 序列化为 `{"eq":["modality","image"]}`。OpenAPI/README 需保持该样例可直接运行。
当前 Rust 内部结构仍保留 `SearchRequest.query_image: Option<Vec<u8>>`；REST JSON 层新增 `query_image_base64` DTO 字段并在 handler 中解码到内部 `query_image`，避免把 `Vec<u8>` 的默认 JSON 数字数组暴露为用户 API。multipart 仍是主推荐路径。

### 6.2 文→图 gating

在 engine 层增加明确校验：

1. 当请求是文本 query，目标过滤为 `modality=image`，且走 vector/hybrid 时：
   - 如果 embedder `caps.cross_modal=true`，允许文本 query 向量召回图片向量。
   - 如果 `caps.cross_modal=false`，仍可走 BM25/caption，但不能声称跨模态向量召回；可返回 warning/explain。
2. 当请求带 `query_image`：
   - 必须 `caps.image=true`。
   - 图→图不强制 `cross_modal=true`，但若要图→文则需要 `cross_modal=true`。

### 6.3 排序融合

保持现有融合：

1. 图片 caption/OCR 命中来自 keyword。
2. 跨模态语义命中来自 vector。
3. `hybrid` 用现有 RRF/归一化/加权融合。
4. 同分继续按 `GlobalId` 升序 tie-break。

建议为图片检索增加 explain 字段：

```json
{
  "modality":"image",
  "match_reasons":["vector:image_embedding","keyword:caption"],
  "embed_model":"jina-clip-v2"
}
```

如果当前 `SearchHit` 不适合马上扩展，可先在 `explain=true` 时返回最小可观测信息。

## 7. S3/MinIO 对象存储接入

### 7.1 接入范围

对象存储需要同时接入三条链路：

1. **输入链路**：本地图片、multipart 图片或 docparse 导出的图片文件可上传到 S3/MinIO，并把 `MediaRef.asset` 写成 `AssetPointer::Object { uri }`；也允许用户自带 bucket 的 `s3://bucket/key` / `minio://bucket/key` 引用，但必须通过 bucket allowlist、tenant namespace 和对象元数据归属校验。
2. **索引链路**：图片 chunk 写入或重建向量时，engine 可从 S3/MinIO 读取对象字节，生成视觉向量。
3. **结果链路**：搜索结果仍只带 citation/media；`/v1/assets/resolve` 经 ACL 后把 Object 图片签成短时 presigned URL，前端直连对象存储取图。

fastsearch 不长期代理对象字节，不把对象裸 key 暴露给客户端。

### 7.2 配置

新增 S3-compatible 配置，MinIO 使用同一套参数：

```text
FASTSEARCH_OBJECT_STORE=s3|minio|none
FASTSEARCH_S3_ENDPOINT=http://127.0.0.1:9000        # MinIO 必填；AWS S3 可空
FASTSEARCH_S3_REGION=us-east-1
FASTSEARCH_S3_BUCKET=fastsearch-assets
FASTSEARCH_S3_ALLOWED_BUCKETS=fastsearch-assets,tenant-a-assets
FASTSEARCH_S3_ACCESS_KEY=...
FASTSEARCH_S3_SECRET_KEY=...
FASTSEARCH_S3_BUCKET_CREDENTIALS_JSON=/path/to/bucket-creds.json  # 可选：用户自带 bucket 的 per-bucket 凭证/role
FASTSEARCH_S3_FORCE_PATH_STYLE=true                 # MinIO 通常 true
FASTSEARCH_S3_PREFIX=images/
FASTSEARCH_S3_PRESIGN_TTL=300
FASTSEARCH_S3_MAX_IMAGE_BYTES=20971520
FASTSEARCH_S3_UPLOAD_THRESHOLD_BYTES=262144         # 超过阈值从 inline 转 object
```

凭证只在服务端/索引端使用。搜索响应和普通 API 响应不得返回 access key、secret、未签名 object uri。
凭证模型：

1. 默认使用一组服务端 S3/MinIO 凭证，适用于 fastsearch 管理 bucket。
2. 允许用户自带 bucket 时，bucket 必须加入 `FASTSEARCH_S3_ALLOWED_BUCKETS`，并且服务端必须具备该 bucket 的读取/HEAD/presign 权限：可以通过同一服务账号授权、per-bucket credential 配置文件，或云厂商 role/STS assume-role 实现。
3. 未配置可用凭证的自带 bucket 不可入库；`validate_ref` 返回 403/400。
4. 对象 key 必须落在调用者 tenant namespace 下，或对象 metadata/tag 中带有 fastsearch 可验证的 `owner_tenant`/`sha256`。

### 7.3 对象接口

把当前 `ObjectSigner` 扩展为完整对象后端，或新增并注入 `ObjectStore`：

```rust
pub trait ObjectStore: Send + Sync {
    fn put(&self, key: &str, bytes: &[u8], content_type: &str) -> anyhow::Result<ObjectRef>;
    fn get(&self, uri: &str, max_bytes: usize) -> anyhow::Result<ObjectBytes>;
    fn presign_get(&self, uri: &str, content_type: Option<&str>) -> anyhow::Result<PresignedUrl>;
    fn validate_ref(&self, uri: &str, principal_tenant: Option<&str>) -> anyhow::Result<ObjectRef>;
    fn delete(&self, uri: &str) -> anyhow::Result<()>;
}

pub enum ObjectErrorKind {
    NotFound,
    Forbidden,
    Transient,
    InvalidMetadata,
    TooLarge,
    UnsupportedMediaType,
}

pub struct ObjectRef {
    pub uri: String,          // s3://bucket/key 或 minio://bucket/key
    pub bucket: String,
    pub key: String,
    pub provider: String,
    pub content_type: String,
    pub size: u64,
    pub sha256: String,
    pub etag: Option<String>,
    pub owner_tenant: Option<String>,
}

pub struct ObjectBytes {
    pub bytes: Vec<u8>,
    pub content_type: Option<String>,
    pub size: u64,
}

pub struct PresignedUrl {
    pub url: String,
    pub expires_s: u64,
}
```

实现建议：

1. 新增 `fastsearch-object` crate，避免把 S3 SDK 扩散到 core/text/vector。
2. 使用 AWS S3 compatible SDK 或轻量 SigV4 实现，支持 MinIO path-style endpoint。
3. `engine` 只依赖 trait；`server/main` 负责按 env 构造并注入。
4. 依照默认 `cargo build` 的重依赖隔离原则：`fastsearch-object` 的 S3/MinIO 实现必须 feature-gated（例如 `--features object-s3`），根默认构建不得拉 `aws-sdk-s3`/TLS 大依赖；默认只编译 trait、空实现和纯逻辑测试。
5. `ObjectStore` 对外必须返回可分类错误（见 `ObjectErrorKind`）：只有确认的 `NotFound` 才能触发对象缺失清理；`Forbidden`/`Transient` 不得删除 PG chunk。
6. S3 SDK 如采用 async 客户端，调用边界必须放在 server/后台任务或显式 `block_in_place` 包装中；不得在 CDC `IndexSink::apply_upsert` 同步热路径里直接阻塞 runtime。推荐对象图片视觉嵌入进入后台补嵌入队列：PG 先写 chunk 和 `image_vector_status=pending`，后台读取对象并写向量，完成后更新状态。

### 7.4 输入链路

#### 7.4.1 REST index

扩展 `/v1/index` 支持三种图片输入：

1. **inline bytes**：请求直接带 `media_bytes`，小图继续落 PG bytea。
2. **upload-to-object**：multipart 或 JSON base64 图片由 server 上传到 S3/MinIO，随后 chunk 入库为 `AssetPointer::Object { uri }`。
3. **object-ref**：请求只提供 `media.asset = Object { uri }`，server 调 `ObjectStore::validate_ref(uri, principal.tenant)` 校验 bucket allowlist、tenant namespace、对象存在性、size/MIME、sha256/etag 和 owner metadata 后入库。

建议新增显式参数：

```json
{
  "collection": "kb",
  "doc_id": "fig-1",
  "store_media": "auto|inline|object|reference",
  "chunks": []
}
```

`auto` 规则：小于 `FASTSEARCH_S3_UPLOAD_THRESHOLD_BYTES` 走 inline；大于阈值且配置对象存储走 object。未配置对象存储时策略固定为：`FASTSEARCH_IMAGE_STRICT=true` 返回 400；默认非严格模式写入无字节图片但必须在 ingest response 返回 warning，并在 PG 真源持久化 `image_vector_status="missing_bytes"`，只保留 caption/OCR/DocRegion。

#### 7.4.2 CLI ingest

新增/明确：

```bash
fastsearch ingest ./fig.png --collection kb --doc-id fig-1 --store-media object
fastsearch ingest s3://fastsearch-assets/images/fig.png --collection kb --doc-id fig-1 --store-media reference
fastsearch ingest ./docs --image-store object
```

行为：

1. 本地图片 + `--store-media object`：CLI 把图片交给 server，由 server 统一写对象存储，避免把 S3 凭证下发到客户端。
2. `s3://...` 输入：CLI 只传 object uri；server 通过 `validate_ref` 校验 bucket allowlist、tenant namespace 和对象元数据归属。
3. docparse 导出图片文件时：CLI 优先上传图片文件到对象存储并写 `MediaRef::Object`，caption/OCR 仍写 `text`。

#### 7.4.3 Object key 规范

对象 key 必须可重建、可去重、不会泄露本地路径：

```text
{FASTSEARCH_S3_PREFIX}/{collection}/{doc_id}/{chunk_id}-{sha256_16}.{ext}
```

`doc_id` 需做 URL/path 安全编码。可选记录 `etag/sha256/size/content_type` 到 `media` JSON，便于重建和审计。

用户自带 bucket 的 key 规范：

1. 推荐 key 前缀为 `{tenant}/{collection}/{doc_id}/...`。
2. 若无法满足前缀规范，必须在对象 metadata/tag 中写入 `fastsearch-owner-tenant`、`fastsearch-sha256`，由 `validate_ref` 校验。
3. 仅猜中 bucket/key 不构成授权；未通过归属校验的 object-ref 一律 403/400，不入 PG。

### 7.5 索引链路

engine 写入图片向量时：

1. `Inline`：继续读取 `media_bytes`。
2. `Object`：调用 `ObjectStore::get(uri, max_bytes)`，拿到图片字节后调用 `embed_multi(Image)`。
3. `DocRegion`：无字节，只走 caption/OCR 文本。

读取失败策略：

1. `FASTSEARCH_IMAGE_STRICT=true`：本 chunk 写入失败，返回错误。
2. 默认：记录 warning，退回 caption/OCR 文本向量；若文本也为空则跳过向量，并更新 PG `image_vector_status`。

重建派生索引时也必须走同一逻辑，确保 PG 真源 + 对象存储可以完整恢复图片向量。

`image_vector_status` 必须进入 PG 真源，建议为枚举文本列或 `media` JSON 中的稳定字段，并在 `ChunkRow`/CDC/搜索 explain 中透出：

| 状态 | 含义 |
|---|---|
| `pending` | chunk 已入 PG，等待读取图片字节并生成视觉向量 |
| `embedded` | 已用图片字节生成视觉向量 |
| `text_fallback` | 图片字节不可用，已用 caption/OCR 文本向量兜底 |
| `missing_bytes` | 无 inline 字节、对象存储未配置或对象缺失 |
| `asset_missing` | 确认对象 404/NoSuchKey，等待删除或重 ingest |
| `error` | 临时错误/权限错误/模型错误，保留 PG chunk 等待重试 |

该状态用于补嵌入、运维排障和召回解释，不能只存在于 ingest response。

### 7.6 删除与生命周期

Object 图片删除采用**复用现有 doc/chunk delete/replace 语义，同步删除 PG chunk**的策略，不新增独立语义的对象删除 API：

1. 删除入口复用已有 doc 级替换/delete 或未来 chunk delete：先在 PG 内把 chunk 标记为 deleting/tombstone 或直接删除 chunk 行，提交后由 CDC 删除派生 text/vector 索引。
2. fastsearch 管理的对象：PG 删除提交后，再由后台清理任务调用 `ObjectStore::delete(uri)` 删除对象。对象删除失败只留下可重试的 orphan object，不得回滚已删除 PG chunk。
3. 用户自带 bucket 的对象：PG chunk 删除不得反向删除用户自带 bucket 的对象，除非该 object-ref 明确标记为 fastsearch-owned。
4. 用户在自带 bucket 删除原图后，必须调用现有 doc/chunk delete/replace 语义同步删除 PG chunk；若索引读取或 resolve 发现对象缺失，只在确认 `ObjectErrorKind::NotFound` 且重复确认后将 chunk 标记 `image_vector_status="asset_missing"` 并进入清理队列。`Forbidden`、网络超时、MinIO/S3 5xx 等 transient/权限错误只进入 `error` 并重试，不得自动删 PG。
5. 对象 `sha256/etag/size` 与 PG 记录不一致时，视为对象被替换：严格模式报错，默认模式将 chunk 标记 `error` 或 `asset_missing`，要求重新 ingest。

### 7.7 结果链路

当前 `/v1/assets/resolve` 已有 Object 分支，缺真实 S3/MinIO presign。补齐后：

1. `AssetPointer::Object { uri }` → `ObjectStore::presign_get(uri, media_type)`。
2. 返回：

```json
{
  "citation_id": "kb:report.pdf:42",
  "type": "object",
  "url": "http://minio.local/fastsearch-assets/images/...?X-Amz-Signature=...",
  "expires_s": 300,
  "media_type": "image/png"
}
```

3. `GET /v1/asset/{cid}` 对 Object 继续 302 到 presigned URL。
4. 搜索响应本身不内嵌 URL，避免缓存短时凭证。
5. 所有对外响应里的 `media` 必须是脱敏后的 public 形态：允许返回 `asset.kind`、`media_type`、`region`、`time`、`caption_source`，不得返回 `Object.uri`、bucket、key、etag、sha256、owner_tenant。完整对象定位只在服务端内部和 PG 真源中保存。
6. 脱敏必须在统一 DTO 层实现，覆盖 REST search、`/v1/assets/resolve` 非 URL 元信息、MCP `resolve_citation`/search 工具、clients SDK、debug/explain、eval 输出；禁止直接序列化内部 `Citation.media` / `MediaRef` 到外部协议。

公开响应建议：

```json
{
  "citation_id": "kb:report.pdf:42",
  "media": {
    "asset": {"kind": "object"},
    "media_type": "image/png",
    "region": {"x0": 1.0, "y0": 2.0, "x1": 3.0, "y1": 4.0}
  }
}
```

### 7.8 安全约束

1. 默认不允许任意 HTTP URL 作为对象输入，防 SSRF。
2. 允许用户自带 bucket，但只允许配置 allowlist 中的 bucket；跨 bucket 需显式 allowlist。
3. `max_bytes` 默认限制，例如 20 MB。
4. 只接受 `image/*` 或可验证的图片 MIME。
5. presigned URL TTL 默认 300 秒，可配置但不建议长于 15 分钟。
6. 越权用户调用 `/v1/assets/resolve` 时不返回 object URL，不暴露对象是否存在。
7. 对象 uri 是内部定位信息，不作为可直接访问 URL 暴露给普通客户端。
8. object-ref 只在 `validate_ref` 通过对象归属校验后入库，防止用户引用其他租户对象后借自身 ACL 签出 URL。

## 8. REST 与 CLI

### 8.1 REST

REST 用户 API 提供两种图片 query 输入：JSON `query_image_base64` 和 multipart `image`；handler 统一解码/读取到内部 `SearchRequest.query_image`：

1. `POST /v1/search` 根据 `Content-Type` 分流 JSON 与 multipart。
2. JSON 请求使用 API DTO：`query_image_base64` 解码到内部 `SearchRequest.query_image`；multipart 的 `request` 字段解析其余搜索参数。
3. `image` 字段读入 `req.query_image`。
4. 限制上传大小，默认 20 MB，可配置。
5. OpenAPI 补 multipart schema。

### 8.2 CLI

新增：

```bash
fastsearch search --collection kb --query "红色柱状图" --modality image
```

CLI 行为：

2. `--modality image` 编译成普通 `Filter`，不引入服务端新概念。

## 12. 风险与回退

1. **模型协议不统一**：先实现 OpenAI-compatible + generic gateway，避免把第三方特殊格式硬编码进核心。
2. **文图不同空间**：用 `caps.cross_modal` 硬门禁，不允许配置不明时开启文→图向量召回。
3. **对象 URI 泄露/SSRF**：对象读写和对象签名统一走 allowlist；默认不读任意 URL；搜索响应使用 public media DTO，不返回裸 uri/bucket/key。
4. **维度漂移**：启动和写入时校验 `FASTSEARCH_EMBED_DIM`，模型切换必须重建向量。
5. **图片过大**：上传、读取、inline 都有限制；大图走对象存储和缩略图。
6. **用户自带 bucket 越权引用**：仅 allowlist 不够，必须校验 tenant namespace 或对象 metadata/tag 归属；未通过不入库。
7. **用户自带 bucket 凭证不可用**：allowlist 不代表可访问，必须有服务端凭证/per-bucket role/STS；无凭证时拒绝 object-ref。
8. **对象与 PG 生命周期分裂**：删除复用已有 doc/chunk delete/replace，PG 删除先行；对象清理后台可重试；只有确认 NotFound 才进入 asset_missing，临时错误不得误删 PG。
9. **对象读取阻塞运行时**：S3 SDK async/blocking 边界必须清楚；CDC apply 热路径不直接做慢对象读取，推荐 `image_vector_status=pending` + 后台补嵌入。
10. **重依赖污染默认构建**：S3/MinIO SDK 必须 feature-gated，默认收口仍验证搜索热路径零对象存储重依赖。
11. **质量不可控**：caption/BM25 保底，真实跨模态能力必须通过 eval golden 后才标完成。
