## 1. 迭代循环（每一轮固定动作）

```
① 选题      → 从 §3 看板取"就绪 / 优先级最高 / 当前环境可完整验证"的一项
② 判复杂度  → 复杂（新后端/跨 crate/触不变量/有不确定性）→ 先写/补 spec；简单 → 直接做
③ 实施      → 源码 + 测试同一改动；守住 §2 不变量；偏离计划回写 spec
④ 收口      → cargo fmt --all --check && clippy -D warnings && test --workspace 三绿
              （改检索行为/契约：再实跑 CLI/server 验证一遍）
⑤ 记账      → 回写对应 spec 的"状态/已知限制"，更新本看板该项状态
⑥ 提交      → 一项一 commit（main 本地提交；push 需用户显式要求）
⑦ 回到 ①
```

**选题优先级**（高→低）：(a) 解锁后续的基础件 > (b) 用户可见价值 > (c) 质量护栏。同级别里**当前环境可完整验证**的优先；需外部资源（PG/模型/K8s）的项只能写代码 + 标 `待运行验证`，不算"done"，排到 §3.B。

## 2. 每轮都要守的不变量（摘自 CLAUDE.md，违反即回退）

1. 托管 PG 可移植：PG 侧只用 pgvector + 逻辑复制，绝不要求原生扩展。
2. PG 是真源，引擎索引派生可重建。
3. ACL 不可绕过：新检索入口必须服务端注入 `engine.search(req, Some(acl))`，带越权用例。
4. 确定性：同分按 `GlobalId` 升序 tie-break。
5. 预过滤：SUPERSET 后端查询 + `core::Filter::eval`/`AclFilter::visible` 精确后过滤（两端同步）。
6. 重依赖 opt-in、诚实记账：未真机验证标 `待运行验证`，未落地标 `下一迭代`，别把基线说成生产级。
