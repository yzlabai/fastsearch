## 0. TL;DR（给赶时间的人，反映 v2 决议）

- **一句话定位**：fastsearch 是一个**外部单二进制混合检索引擎**，以**托管 Postgres + pgvector 为真源/持久层**，专为"docparse 解析后的可溯源文档 chunk"做全文/向量/混合检索，并把 page+bbox 引用端到端带到答案层。
- **架构（§6.8）**：Postgres 持有真源（文档/chunk/元数据/向量）→ fastsearch 经**逻辑复制 CDC** 同步 → **自持派生 BM25 倒排索引**（Tantivy/mmap，在 PG 外）→ 服务统一混合检索。原型是 **Quickwit 式控制面/数据面分离**（PG 当真源/metastore，引擎当无状态服务层，可多副本横向扩展）。
- **怎么超越 ParadeDB**（已被 VectorChord/pg_textsearch/CDC 调研坐实）：
  1. **托管 PG 可移植**——能跑在 RDS/Aurora/Supabase/Neon 上（只需 pgvector，它在所有云的 allowlist 上）；ParadeDB/VectorChord/pg_textsearch 因需 `shared_preload_libraries` 原生扩展**一个都进不去**。这是它们结构上过不去、我们能钻的缝。
  2. **一等混合融合**（RRF + 分数归一化内置）——三家都没有，全靠手写 SQL。
  3. **真正预过滤 ANN**——修掉 pgvector 后过滤召回崩（选择性强时返 ~1%）的坑。
  4. **BM25 可调**（k1/b + 自定义 Similarity）——pg_search 做不到。
  5. **引用原生**（page+bbox+section 端到端）——独有护城河。
  6. **PG 零搜索写放大**——倒排在引擎侧，不像 ParadeDB 每段+merge 都 WAL 写进 PG。
- **技术主线**：Tantivy（BM25 + fast fields）＋ lindera/jieba/icu（CJK 分词）＋ 引擎侧向量索引（hnsw_rs，借鉴 VectorChord 的 RaBitQ 量化）或直接用 pgvector ANN ＋ **Block-Max WAND** top-k ＋ RRF/归一化融合 ＋ 逻辑复制 CDC 同步层（Sequin/Debezium 式）＋ Candle/fastembed 本地嵌入（multilingual-e5-small）。
- **诚实的代价**：派生索引与 PG **亚秒级最终一致**（非同事务，但等同 ParadeDB 给托管用户推荐的搜索副本模型，对 RAG 灌库无所谓）；要**自建 CDC 同步层**；fastsearch **不再零依赖**（需要一个 Postgres）。
- **仍待拍板**：向量 ANN 在哪服务——完善产品**两者都要**（(a) pgvector 直查做托管省事档、(b) 引擎自建做高性能/预过滤/量化档），运行时可切；MVP 思维下的"二选一"被否。见 §6.8 / §11。
- **完善产品（非 MVP）视角**：除了"检索三件套 + 超越 ParadeDB"，一个 GA 级产品还必须有——**安全与访问控制**（私有 RAG 的逐文档/集合级权限、API Key/RBAC、审计）、**运维与可观测**（指标/追踪/慢查询/健康探针/优雅重启）、**索引零停机生命周期**（蓝绿重建、快照恢复、格式版本迁移）、**CDC 健壮性**（exactly-once、断点续传、DDL 演进、自愈）、**相关性质量体系**（golden 评测集、nDCG/recall 回归、调参面板、同义词词典管理）、**查询能力完备**（typo/前缀/短语/分面/高亮/深分页/折叠/建议）、**集成面**（REST+OpenAPI、MCP、Python/TS SDK、LangChain/LlamaIndex 适配）、**规模与 HA 边界**（明确容量信封 + 副本 + 分区 + 何时该换方案）。判准见 **§4.5**。

## 1. 背景与定位

### 1.1 上下游：fastsearch 在 localkb 里的位置

`localkb`（本地知识库）目录下当前有两个孪生项目：

```
localkb/
├── docparse-rs/   ← 文档解析：12+ 格式 → 带位置的 RAG chunk / 结构树 / OKF bundle
└── fastsearch/    ← 本项目：检索引擎，消费 docparse 的 chunk，提供全文/向量/混合检索
```

docparse-rs 已经把"**把任意文档变成可溯源的 chunk**"这件事做到极致：每个 chunk 带 `page` + `bbox` + `heading_path`（标题面包屑）+ `section_id`（回指结构树）+ `kind`（paragraph/table/image…），并且保证**确定性**（同输入字节级一致）、**单文件 ~29MB**、**零运行时依赖**、**四张脸**（CLI / 库 / MCP stdio / REST）。

**fastsearch 要补上的是"检索"这一环**：chunk 解析出来之后，得有地方存、能被快/准地检索回来、并把引用带回去。这正是一个本地 RAG 流水线缺的最后一块。

### 1.2 与 docparse-rs 共享的工程哲学（继承的基因）

为了"很好地配合"，fastsearch 尽量继承 docparse-rs 的设计基因（见 docparse-rs 的 [CLAUDE.md](../../../docparse-rs/CLAUDE.md) / [README.md](../../../docparse-rs/README.md)）。**注意 v2 决议（§6.8）后，"零运行时依赖/纯本地"这一条已主动放弃**（引擎现在需要一个 Postgres 真源），其余基因仍守：

| 维度 | docparse-rs 的约定 | fastsearch 对齐情况 |
|---|---|---|
| 语言 | 纯 Rust | ✅ 引擎纯 Rust |
| 部署 | 单二进制，零运行时依赖 | ⚠️ **v2 变更**：引擎仍是单二进制，但**依赖一个 Postgres**（pgvector + 逻辑复制）。换取托管/复制/可移植（§6.8）。不再"零依赖/纯本地" |
| 接口 | 四张脸 CLI / 库 / MCP / REST，**输出字节一致** | ✅ 检索 API 在四张脸上语义一致 |
| 确定性 | 同输入 → 字节级一致输出 | ✅ **限定到"索引已收敛"**：同一稳定索引快照 + 同查询 → 同结果、稳定 tie-break。CDC 同步在途时（lag 窗口）结果会变——这是最终一致的固有性质，须可观测 |
| 契约 | 每种输出有 JSON Schema（draft 2020-12），golden 测试防漂移 | ✅ 索引文档 / 查询 / 结果三类 schema |
| 错误处理 | 边界 `anyhow::Result`；失败页返回空不 panic | ✅ 坏 chunk/向量不中断整批；近似/兜底显式标注 |
| 模型边界 | 确定性核心独立，神经模型 opt-in、按需下载 | ✅ 嵌入/rerank 模型 opt-in、首次用时下载、可纯检索零模型 |

> **设计口号（v2）**：docparse-rs 是"**速度快、质量好**的解析"，fastsearch 是"**速度快、可溯源、可托管**的检索"。两者拼起来 = 一条纯 Rust、可私有化、**以 Postgres 为真源**的 RAG 检索栈。

## 2. 竞品与技术调研

> 三路并行深度调研的结论摘要，完整事实与出处见各小节末。结论用于支撑 §3 功能需求与 §6 技术选型。

### 2.1 要打败的对象：Elasticsearch 为什么"重"

ES 的"重"是 JVM + Lucene + 集群运维三重叠加，给了我们清晰的发力口：

- **JVM 堆**：官方建议堆设为 RAM 的 50%、上限 ~31GB（压缩指针边界），另一半留给 OS 文件缓存——**设计上就浪费一半 RAM**；生产最低 8GB，ES 进程本身至少要 2GB 才能跑。GC 锯齿（30%↔70%）带来延迟尖刺，需要持续监控堆/GC——这类问题在无 GC 的 Rust 里**根本不存在**。
- **冷启动**：容器 ~15s 才能响应 9200，慢环境要几分钟。对比 Quickwit（Rust/Tantivy）从 S3 **70ms 热启动**——约 200× 差距，是可引用的硬数字。
- **运维复杂度**：分片分配、集群状态、节点角色、ILM、堆调优、线程池……Elastic 暴露"**几千个配置参数**"，很多团队要专门一个团队运维。Typesense/Meilisearch 的卖点正是反面："单个自包含原生二进制、智能默认、开箱即用"。
- **Lucene segment 模型**：写入进不可变 segment，refresh（默认 1s）刷小 segment、后台不断 merge；**每个 segment 给每个查询加固定 CPU/堆/GC 成本**，commit 要 fsync 多文件。refresh/merge 是绕不开的延迟/吞吐平衡调参负担。
- **轻量替代的 footprint 参考**：Typesense（RAM 常驻）2.2M 食谱 ≈ 900MB RAM，28M 书 ≈ 14GB RAM——**索引必须装进内存**是其扩展上限。

> 出处：Elastic 官方堆/内存/JVM 文档、Opster、Quickwit 101、Typesense 对比、youngju.dev 2026 搜索引擎横评。

### 2.2 要参考的对象：Meilisearch 怎么做到"快且轻"

Meilisearch 的轻量优势是**架构性**的，值得逐条对照取舍：

- **存储 = LMDB（经 heed 类型化封装）**：内存映射、ACID、**零拷贝读**（直接读 mmap，无分配无拷贝）。选 LMDB 而非 sled/RocksDB 是"性能+稳定+特性的最佳组合"。不用调 JVM 堆，OS 页缓存管真实 RAM，~1/3 RAM∶磁盘比不显著影响性能。
- **索引数据结构 = FST + roaring bitmap**：两个 FST（全词典 + 高频前缀）做压缩 + Levenshtein 自动机做 typo 容错；约 20 个倒排库（word/exactword/word_fid/word_position/word_pair_proximity…）的值都是 roaring bitmap（docid 集合，快速交并差）。代价：**索引能到文档的 ~20×**。
- **全文相关性 = bucket-sort 排序规则**：默认 6 条规则按桶顺序应用 `words → typo → proximity → attribute → sort → exactness`，逐步细分候选集而非一次全排序——这是 <50ms 即时搜索的来源。typo 默认 5 字符起 1 错、9 字符起 2 错（封顶）。分词走 charabia（脚本/语言自适应，中文 jieba、日文 lindera+IPA、韩文 lindera+ko-dic）。
- **向量检索 = arroy（不是 HNSW！）**：arroy = Spotify Annoy 的 Rust 移植，**随机投影树森林**，建在 LMDB 上（mmap、多进程共享、原子增量更新、可做 filtered-disk-ANN 边遍历边过滤）。v1.11 加二值量化（f32→1bit，磁盘 ~15%、>1024 维才划算）。**后继 hannoy**：把树换成图（HNSW 风格）仍在 LMDB，号称建索引 2天→2小时、磁盘减半、搜索快 ~10×，已是近期 Meilisearch 默认。
- **混合检索 = 分数归一化，明确拒绝 RRF**：keyword（BM25 桶分）与 semantic（ANN）并行，按**相关性分数**而非排名位置合并；`semanticRatio` 默认 0.5。关键机制是 embedder 的 `distribution`（mean、sigma）**仿射变换**——把语义相似度（常窄聚在 ≥0.5）拉开到 0–1，使其与 keyword 分数可比。**无独立 cross-encoder rerank 阶段**。
- **已知短板（我们的机会）**：社区版单机、无原生分片/集群（分片+复制是企业版/云专属）；扩展上限约几百万文档/几十 GB（100M+ 官方都建议用 ES）；索引吃 RAM（默认封顶 2/3 RAM）；改 settings 触发**全量重建索引**；无聚合分析框架；CJK 有延迟/质量问题（jieba 质量、lindera ~10× 慢、无一等繁体中文管线）。

> 出处：meilisearch.com/docs（storage、relevancy、hybrid_search、embedders）、blog.kerollmops.com（hybrid-search、arroy、hannoy、binary-quantization、filtered-disk-ann）、github.com/meilisearch/{heed,milli,arroy,charabia}。两个最易被误传的关键事实——**混合 = 分数归一化非 RRF**、**向量 = 树非 HNSW**——已对一手源核实。

### 2.3 Rust 生态可用的积木（建什么用什么）

调研确认：**今天用全部宽松许可（MIT/Apache）的 crate 就能搭出本地、离线、单文件的混合检索引擎**。关键积木与成熟度：

**全文 / 倒排：**
- **Tantivy** 0.26.1（MIT，~15.5k★）——Rust 版 Lucene 库。BM25、可配分词管线、**fast fields**（列存 doc-values 做排序/facet/聚合）、Lucene 式不可变 segment+merge、mmap + OS 页缓存（无 JVM 堆无 GC），号称 ~2× Lucene 搜索延迟。**是库不是服务**（无分布式/REST/集群，schema 要先定）。Quickwit 在它之上做了对象存储版分布式日志搜索。
- Meilisearch 自研 **milli** ≠ Tantivy：存储走 LMDB（非 segment）、排序是 bucket-sort 排序规则（非纯 BM25）、向量走 arroy。

**分词（CJK/多语）：** lindera 4.0（MIT 引擎，字典各自许可——**中文 CC-CEDICT 是 CC-BY-SA 3.0 copyleft，分发前要审**）、jieba-rs 0.10、charabia 0.9.9（Meilisearch 的脚本自适应管线）、whatlang 0.16（70 语言检测）。Tantivy CJK：纯中文用 cang-jie/tantivy-jieba，全 CJK 用 lindera-tantivy。

**向量 / ANN：**
| 方案 | 版本/许可 | 特点 | 落盘 |
|---|---|---|---|
| hnsw_rs | 0.3.4 MIT/Apache | 最完整维护的纯 Rust HNSW，唯一有真 mmap 重载 | mmap |
| instant-distance | 0.6.1 MIT/Apache | 小而干净、生产验证、功能冻结 | serde 快照(RAM) |
| usearch | 2.25 Apache | C++ HNSW 绑定，量化丰富(f16/bf16/i8/二值)、SIMD、谓词过滤、mmap view | mmap，需 C++ 工具链 |
| arroy | 0.6.4 MIT | Annoy 树，LMDB mmap、增量、roaring 过滤、二值量化 | LMDB |
| hannoy | — MIT | arroy 后继，图(HNSW)在 LMDB，~10× 搜索 | LMDB |
| Qdrant(作库参考) | 1.18 Apache | 段式 + WAL + **可过滤 HNSW**（把 payload 感知边建进图）+ 标量/乘积/二值量化 | 可配 mmap |
| LanceDB/Lance | 0.30/7.0 Apache | Arrow 列存、**磁盘 ANN**(IVF_PQ/HNSW_PQ)、工作集不必装进 RAM | 落盘 |

**混合检索 SOTA（2024–25）**：通用管线 = 检索(稀疏+稠密并行) → 融合 → [量化重打分] → [cross-encoder rerank] → top-N。
- **RRF**：`Σ 1/(k+rank)`，k=60 是从未重调的事实默认（ES/OpenSearch/Azure/Weaviate=60，**Qdrant=2**）。尺度无关、抗离群、免归一化；缺点是忽略分数幅度。
- **分数归一化融合**：min-max（脆）、凸组合/alpha 加权（Weaviate relativeScoreFusion 默认，~6% nDCG 优于 RRF）、**DBSF**（Qdrant，mean±3σ 修 min-max 离群脆性）。归一化在分数可比时赢，**RRF 是异构/不可信检索器的稳健兜底**。
- **rerank**：bi-encoder（可缓存、可索引）vs cross-encoder（`[q,doc]` 联合、查询时 O(N)、不可缓存）。典型 top-100→rerank→top-5~10，+5~20% nDCG@10。模型如 bge-reranker-v2-m3、mxbai-rerank-v2、jina-reranker-v2。
- **量化**：标量 int8（4×、~99% 召回、最安全）；二值（~32×、需 >1024 维 + 重打分恢复召回）；PQ（16–64× 但已被 BBQ/RQ 超越）。通用恢复法 = **超采样 + 对全精度重打分**。

**本地嵌入（离线单文件）：**
- **Candle** 0.10（MIT/Apache，HF 维护）——纯 Rust，可 `include_bytes!` 静态链模型，CPU/CUDA/**Metal**/WASM。**最贴合单文件**，代价是要自己写 tokenize+pooling 胶水、CPU 略慢于调优过的 ONNX。
- **ort** 2.0-rc + **fastembed** 5.17（Apache）——ONNX Runtime，CPU 最快 + 一等 int8，但要带 native lib（非单文件）、模型首次下载。
- 推荐模型：bge-small-en-v1.5（384 维 33M，英文 RAG 主力）/ **multilingual-e5-small（384 维 118M，100 语含中文）**——单文件场景首选小而多语的 e5-small。

|---|---|---|---|---|---|

| redb 4.1 | **是** | B+树 COW | 否(无 mmap) | 读+好写 | **稳定** |

| rust-rocksdb | 否(C++) | LSM | 可选 | 写 | 很成熟 |
| fjall 3.1 | **是** | LSM | 否 | 写 | 稳定 |

### 2.4 RAG / 文档检索的领域要求（我们的真实用例）

我们的 chunk 恰好携带 `text` + `page` + `bbox` + `heading_path` + `section_id` + `kind`，几乎完美对上现代文档 RAG 最佳实践：

- **父上下文分块取胜**：小 chunk 检索、生成时补父上下文（Stanford 2025）。docparse 的结构化 chunk + section_id = 在解析时就做好了结构分块。
- **混合检索是默认**：BM25 + 稠密向量，RRF 融合。
- **Contextual Retrieval（Anthropic 2024）**：在 chunk 前**预置 50–100 token 上下文**再嵌入+BM25，top-20 检索失败率 5.7%→3.7%（仅上下文嵌入）→2.9%（+BM25）→1.9%（+rerank）。成本 $1.02/M token（带 prompt 缓存）。**我们的面包屑/标题几乎是免费的上下文**——可确定性地从 heading_path 合成前缀，比 LLM 调用更省。
- **父文档/auto-merging 检索**：只嵌入叶 chunk，父/section 放 docstore；多数兄弟子块命中时返回父块。**直接映射到我们的 section_id 层级**。
- **rerank 管线**：宽召回(top-100~150) → cross-encoder → top-K(20)。要求引擎返回足够候选 + 保留分数。
- **元数据过滤（直接来自我们的 schema）**：按 document / page(范围) / section_id / 面包屑前缀 / kind（如只查 table、排除 image）过滤，需类型化 payload 索引 + 可嵌套 AND/OR/NOT，且能与 hybrid 打分一次查询组合。
- **引用溯源流（page+bbox）**——最贴合用例的发现：① 入库时每 chunk 给紧凑引用 ID，bbox/page 存为 chunk 元数据（不内联进文本）；② 检索/rerank 全程带着不变；③ 合成时让 LLM 输出 `{answer, citations:[id]}`；④ 解析 ID → `{page,bbox}` → 生成深链/高亮跳到 PDF 精确区域。**我们的 bbox+page+section_id 正是这套溯源所需的键。**

> 出处：Anthropic Contextual Retrieval、LlamaIndex AutoMergingRetriever、Qdrant filtering/hybrid、Typesense vector search、Tensorlake citation-aware RAG、arXiv 2510.24402 / 2412.14457。

### 2.5 "Postgres 上做检索"的竞品全景（v2 决策的直接依据）

调研了所有"用 Postgres 把 BM25/向量做得比 ParadeDB 好"的现存项目，五种哲学：

| 项目 | 引擎 | 原生扩展? | 需 preload? | 许可 | 成熟度 / 数据 |
|---|---|---|---|---|---|
| **ParadeDB pg_search** | 嵌入 **Tantivy** | 是(pgrx) | PG<17 需 | **AGPLv3** | 最成熟 v0.24；**装不了 RDS**；建索引快但搜索吞吐独立测输 ES、索引更大 |
| **VectorChord-bm25** | 自研 Rust(Block-WeakAnd) | 是 | 否 | **AGPLv3/ELv2** | bm25vector 类型 + `<&>`；宣称 3× ES（无 vs ParadeDB 数据）；英文 only、早期 |
| **Tiger pg_textsearch** | 自研 C(LSM+WAND) | 是 | 是 | **PostgreSQL** | v1.3.1；宣称 2.4–6.5× ParadeDB（短查询、warm、不存位置→非对等；建索引慢 2×、长查询优势塌到 1.1×） |
| **pg_bestmatch.rs** | 无(→稀疏向量) | 是 | 否 | Apache | v0.0.1 PoC，HNSW 导航稀疏向量差 |
| **VectorChord(vchord)** | IVF+**RaBitQ** | 是 | 是 | AGPLv3/ELv2 | pgvecto.rs 后继；1-bit 量化、低 RAM、~192B/1536维；Immich 在用；**装不了 RDS** |
| **pgvectorscale** | StreamingDiskANN | 是 | 可选 | PostgreSQL | Tiger 出品；vs Pinecone 28× p95；**仍是原生 .so，装不了 RDS** |

**关键结论（决定 v2 走路线 B）**：① 这些"PG 内"方案除 pgvector 本身外**全部需要原生扩展、装不进托管云**——这是它们共同的、结构性的天花板；② 那些 2–6× 的数字多是 vs 羸弱基线（原生 PG FTS 缺 GIN）或 warm/短查询/非对等，独立复现里 **ES 搜索吞吐反而赢 ParadeDB**；③ **pgvector 是唯一在所有托管云 allowlist 上的**。所以"外部引擎 + 托管 PG/pgvector"（§6.8 路线 B）是唯一能既用 Postgres 又跑遍托管云、从而真正绕过 ParadeDB 短板的路。

**CDC 同步层选型**（路线 B 的关键新组件）：逻辑复制 **pgoutput**（PG 内置、托管云唯一可用插件）是基础；**Sequin** 是最佳参照（逻辑复制 CDC、亚秒、exactly-once 幂等键、严格行序、一等 ES/Typesense/Meilisearch sink，无需 Kafka）；Debezium 成熟但默认 at-least-once（sink 需幂等）。坑：未消费的复制 slot 会无限留 WAL **撑爆磁盘**，必须 `max_slot_wal_keep_size` + 心跳 + 监控 `pg_replication_slots`。LISTEN/NOTIFY 有损不可重放，只能当 wake-up。

> 出处：docs.vectorchord.ai、github.com/{tensorchord/VectorChord, tensorchord/VectorChord-bm25, timescale/pg_textsearch}、tigerdata.com/blog、neon.com pg_search 弃用公告、Sequin/Debezium 文档、各云 RDS 扩展 allowlist。厂商基准均标注、与独立数据分离。

## 3. 功能需求

> 优先级（v3 重释）：标记的是**交付推进次序**，不是"砍不砍"。**完善产品终态 = 全部交付**（§4.5）。P0=奠基先行、P1/P2=后续阶段；"必备"= 1.0 GA 不可缺。每条尽量给"用户使用例子"。

### 3.1 索引与文档模型（P0）

- **F1（P0）原生消费 docparse chunk**：直接吃 docparse `-f chunks` 的 JSON / NDJSON（schema 见 docparse [schemas/chunk.json](../../../docparse-rs/schemas/chunk.json)）。文档模型 = 一个 chunk 一条记录，字段至少含 `id`、`kind`、`text`、`page`、`bbox{x0,y0,x1,y1}`、`heading_path[]`、`section_id`、`char_len`，以及外层注入的 `doc_id`（来源文件标识）。`image` chunk 的 `image{caption,file,...}` 作为可选元数据存储。
  - 例：`docparse report.pdf -f chunks | fastsearch index --collection kb --doc-id report.pdf`
- **F2（P0）通用文档接入**：也接受任意 JSON 文档（不止 docparse），可配置主键、可搜索字段、可过滤字段、向量字段——保证不被 docparse 绑死。
- **F3（P0）集合（collection/index）管理**：创建/删除/列出集合，每集合独立 schema 与设置。
- **F4（P0）增量更新**：按 `doc_id` 增删改某文档的所有 chunk（重新解析一个文件 → 替换它的全部 chunk），不必全库重建。
- **F5（P1）父文档/section docstore**：按 `section_id` 存储 section 级父文本，支持 auto-merging 检索（F14）。

### 3.2 全文检索（P0）

- **F6（P0）BM25 全文检索**：默认 BM25 打分；可搜索字段可配权重（如 heading_path 权重 > 正文）。
- **F7（P0）CJK + 多语分词**：中文分词（jieba/lindera）开箱即用，兼顾英文/混排；语言/脚本自适应。**这是相对 ES 默认配置和 Meilisearch CJK 短板的明确加分项**。
- **F8（P1）typo 容错 + 前缀/即时搜索**：Levenshtein 容错（可配字长阈值）、最后一词前缀匹配，支持 search-as-you-type。
- **F9（P1）短语 / 邻近 / 精确匹配、同义词、停用词**。

### 3.3 向量检索（P0）

> ANN 在哪跑由 §6.8 子选择定，**完善产品两档都交付、运行时可切**：(a) pgvector 在 PG 内跑（托管省事档）、(b) 引擎侧自建索引（高性能档，兑现 F11 真预过滤、可上 RaBitQ 量化）。向量真源始终在 pgvector 列。

- **F10（P0）稠密向量 ANN 检索**：近邻检索，余弦/点积/欧氏，top-k。向量存 pgvector 列（真源）；可外部提供或由内置 embedder 生成（F12）。
- **F11（P0）向量 + 过滤组合**：ANN 与元数据过滤一次查询组合（filter-aware 召回，非纯后过滤）。**注**：子选择 (a) pgvector 受其后过滤召回限制（需 iterative scan/partial index 缓解）；(b) 引擎侧可做真预过滤——这正是超越 pgvector 的点。
- **F12（P1）内置离线嵌入（opt-in）**：内置 embedder（Candle 或 ort）跑本地模型（默认 multilingual-e5-small / bge-small），首次用时按需下载（仿 docparse 模型 opt-in 哲学）；纯检索可零模型（外部传向量）。
- **F13（P2）量化**：标量 int8（默认安全档）/ 二值量化（大库高维档），超采样 + 全精度重打分恢复召回。

### 3.4 混合检索（P0）

- **F14（P0）混合检索 = 全文 + 向量融合**：keyword 与 semantic 并行召回后融合。
  - **融合策略**：默认 **RRF**（鲁棒、免归一化、对位 ES/Qdrant），并提供**分数归一化/凸加权**（对位 Meilisearch，分数可比时更优）作为可选；暴露 `semantic_ratio` / `rank_constant` 等旋钮。决策见 §6.4。
  - 例：`fastsearch search --collection kb --query "毛利率为什么下降" --mode hybrid --filter 'kind=table' --semantic-ratio 0.5`
- **F15（P1）宽召回 + 分数保真 + rerank 钩子**：返回足够候选（默认可配 ~100+）且保留各路分数与融合分，便于外接 cross-encoder rerank；可选内置 rerank（bge-reranker-v2-m3，opt-in）。
- **F16（P1）父文档/auto-merging 检索**：召回叶 chunk，当某 section 多数子块命中时"归并上提"返回 section 父块（基于 F5 docstore + section_id）。

### 3.5 过滤、分面、溯源（P0）

- **F17（P0）类型化元数据过滤**：可嵌套 `AND/OR/NOT`，操作符 `= != > >= < <= IN EXISTS`。针对文档 RAG 必备的过滤维度：`doc_id`、`page`/页范围、`section_id`、`heading_path` 前缀（"第3章>方法"下所有内容）、`kind`（只 table / 排除 image）。
- **F18（P0）引用溯源端到端**：每条命中稳定返回 `chunk_id` + `doc_id` + `page` + `bbox` + `heading_path`，全程不丢；提供把引用 ID 解析回 `{page,bbox}` 的能力，供答案层生成深链/高亮。**这是与 docparse `locate(x,y)` 反查闭环的关键。**
- **F19（P1）分面/分组**：按 doc_id / section / kind 分组聚合命中计数。
- **F20（P1）高亮**：返回命中片段的高亮（文本 offset），与 bbox 互补。

### 3.6 四张脸接口（P0/P1）

- **F21（P0）CLI**：`fastsearch index|search|delete|collection|serve|mcp ...`，与 docparse CLI 风格一致；TTY 显示进度，stdout 不污染。
- **F22（P0）库（crate）**：`fastsearch` 作为 Rust 库嵌入（docparse 流水线可直接 `use`）。
- **F23（P1）REST**：`fastsearch serve`（axum，绑 127.0.0.1），`POST /index`、`POST /search`、`GET /healthz`、`GET /openapi.json`、`GET /schema/{name}`。
- **F24（P1）MCP stdio**：`fastsearch mcp`，工具如 `search` / `index_document` / `get_chunk` / `resolve_citation`，供 agent（Claude Code/Cursor）直连，与 docparse MCP 工具链拼成"解析→检索→引用"闭环。

### 3.7 与 docparse-rs 的端到端配合（P0，差异化护城河）

- **F25（P0）一条命令的流水线**：`docparse *.pdf -f chunks | fastsearch index` 即可建库；`fastsearch search ...` 即可检索并拿回带 page+bbox 的引用。
- **F26（P1）OKF / 结构树感知**：可消费 docparse 的 OKF bundle 或 outline 结构树，利用 section 层级做 auto-merging（F16）。
- **F27（P1）Contextual 前缀（确定性）**：入库时可选地把 `heading_path` 面包屑确定性地拼进嵌入/BM25 输入（廉价版 Contextual Retrieval），不需 LLM。
- **F28（P2）Skill 包**：仿 docparse 的 SKILL.md，教 agent 驱动 fastsearch 的"建库→检索→引用解析"。

### 3.8 Postgres 真源 + CDC 同步（P0，v2 架构核心，见 §6.8）

- **F29（P0）Postgres 作真源/持久层**：chunk/元数据/向量写入用户的 Postgres（chunks 表 + pgvector 向量列）。fastsearch 不自建持久存储，**写路径落 PG**（直写或经 fastsearch 代理写）。`fastsearch index` 实际是"写 PG + 触发同步"。
- **F30（P0）逻辑复制 CDC 同步**：经 **pgoutput** 逻辑复制 slot 把 PG 变更（增/改/删，含 `doc_id` 级替换）增量同步到引擎侧派生 BM25 索引；亚秒延迟；断点续传（记 `restart_lsn`）、幂等。**必须管好 slot 生命周期**（`max_slot_wal_keep_size` + 心跳 + 监控，防 WAL 撑爆磁盘）。
- **F31（P0）托管 PG 可移植**：仅依赖 pgvector + 逻辑复制（两者在 RDS/Aurora/Supabase/Neon 全可用），**不在 PG 服务器装任何原生扩展**。这是超越 ParadeDB/VectorChord/pg_textsearch 的结构性优势，须作为硬约束守住。
- **F32（P1）无状态只读副本**：引擎可起多副本（各持派生索引、共享同一 PG 真源/同一复制流），前挂 LB 横向扩展在线多人读。
- **F33（P1）初始快照 + 重建**：新建索引时从 PG 全量快照建派生索引，再切到增量流；支持单集合重建（坏索引/换分词器/换嵌入模型时）。
- **F34（P2）从已有 PG 表接入**：不止 docparse——可指向用户已有的任意 PG 表（配主键/文本列/向量列/过滤列），把 fastsearch 当"任意 Postgres 的检索加速层"。

### 3.9 查询能力完备（完善产品必备，§4.5#3）

- **F35（必备）查询能力对齐 ES/ParadeDB**：typo 容错（Levenshtein 可配，含转置）、前缀/即时、短语+slop、邻近(ordered/unordered)、布尔(must/should/must_not)、范围、正则/通配、term_set、more_like_this、boost/const 权重。（把 v1 的 F8/F9 从 P1 提为必备。）
- **F36（必备）分面与聚合**：terms/histogram/range 桶 + avg/sum/min/max/count/cardinality 度量，单遍随检索返回（对标 ParadeDB fast-field faceting）。
- **F37（必备）结果呈现**：命中片段高亮（文本 offset + byte 位置，与 bbox 互补）、深分页（`search_after` 游标，非 offset）、分组/折叠（按 doc_id/section 去重归组）。
- **F38（P1）查询辅助**：autocomplete/suggestions、did-you-mean 纠错、查询扩展。

### 3.10 相关性质量与调优体系（完善产品必备，§4.5#4——MVP 最爱砍、产品最该有）

- **F39（必备）评测护栏**：内建 golden 评测集机制（用 docparse 真实 chunk + 标注 query→相关 chunk），算 nDCG@k / recall@k / MRR；**相关性回归进 CI 门禁**（改分词/融合/权重不许悄悄掉点）。
- **F40（必备）调优面板**：可调 BM25 `k1/b`（**ParadeDB 做不到**）、字段权重（heading_path > 正文）、融合权重/`semantic_ratio`/`rank_constant`、rerank 开关，且配置可版本化、可 A/B。
- **F41（必备）词典治理**：同义词、停用词、用户自定义词典（中文尤需）的增删管理 + 热更新，不必全量重建。
- **F42（P1）可解释性**：debug 模式返回打分明细（各路分数、融合分、命中词、字段贡献），供调相关性与排障。

### 3.11 安全与访问控制（完善产品必备，§4.5#9——私有 RAG 刚需，v3 新增）

- **F43（必备）认证**：API Key / Token；REST 默认要求认证，不裸绑 127.0.0.1。
- **F44（必备）逐文档/集合级访问控制**：每个 chunk 可带 ACL（owner/tenant/visibility 标签）；**检索时按调用者身份强制注入过滤**，杜绝越权命中——这是企业私有 RAG 的硬门槛。RBAC 角色到集合/文档的映射。
- **F45（必备）审计与合规**：检索/写入审计日志（谁、何时、查了什么、命中什么）；传输 TLS + 静态加密选项；PII 字段策略（标记/脱敏/不索引）。
- **F46（P1）多租户隔离**：集合级命名空间隔离，租户间资源/数据不串。

### 3.12 索引生命周期与零停机运维（完善产品必备，§4.5#8/#10）

- **F47（必备）零停机重建**：换分词器/换嵌入模型/改 schema 时，后台蓝绿重建新索引 + 原子切换，**在线查询不中断**（对比 Meilisearch 改 settings 全量重建、ParadeDB REINDEX）。
- **F48（必备）派生索引快照/快恢复**：因 PG 是真源，索引可重建；但要有**快照机制**让副本秒级拉起、崩溃后免全量重放（重放=兜底）。
- **F49（必备）索引格式版本化**：索引格式带版本号，升级时在线迁移或自动重建，不让旧索引卡住升级。
- **F50（必备）运维可观测**：Prometheus 指标（QPS、p50/p95/p99、召回、**CDC 同步 lag**、段数/索引大小、CPU/内存）、分布式追踪、结构化日志、慢查询日志、健康/就绪探针、优雅启停、限流 + admission control + 背压。

### 3.13 CDC 健壮性（完善产品必备，§4.5#7——把 F30 从"能同步"提到"生产可靠"）

- **F51（必备）一致性保证**：at-least-once + sink 幂等（按 LSN/PK）达到 exactly-once 效果；严格按行序应用；删除/`doc_id` 级替换正确捕获。
- **F52（必备）初始快照→增量无缝切换**：新集合先全量快照建索引，再从快照 LSN 切到增量流，不丢不重。
- **F53（必备）韧性**：slot 生命周期管理（`max_slot_wal_keep_size` + 心跳 + 监控防 WAL 撑爆）、引擎落后/断连自动续传、slot 失效自动重快照、DDL/schema 演进处理、多表/多发布。

### 3.14 集成与 SDK（完善产品必备，§4.5#11）

- **F54（必备）四张脸契约一致**：CLI / 库 / REST(+OpenAPI/JSON Schema) / MCP，语义与结果一致（对齐 docparse 四张脸哲学）。
- **F55（必备）客户端 SDK**：Python / TypeScript 零依赖客户端 + LangChain / LlamaIndex 检索器适配（对齐 docparse 的 clients/）。
- **F56（P1）Agent 集成**：MCP 工具（search / index / resolve_citation）+ Skill 包，拼成 docparse→fastsearch"解析→检索→引用"闭环。

## 4. 非功能需求

| 编号 | 需求 | 目标/指标 | 理由 |
|---|---|---|---|
| N1 | **单二进制引擎（依赖一个 Postgres）** | 引擎本身是一个 Rust 二进制，无 JVM/Python/C++ 运行时；**v2 起需要一个 Postgres 真源**（不再零依赖）；纯检索可零模型（外部传向量） | 对位 ES JVM 集群；用 PG 换托管/复制/可移植 |
| N1b | **托管 PG 可移植（硬约束）** | 只用 pgvector + 逻辑复制，**不在 PG 装原生扩展**；跑遍 RDS/Aurora/Supabase/Neon | 超越 ParadeDB/VectorChord 的结构性优势（§6.8 F31） |
| N1c | **最终一致可接受** | 派生索引与 PG 亚秒级最终一致（非同事务）；可观测同步延迟/lag | 等同 ParadeDB 搜索副本模型，RAG 灌库无碍 |
| N2 | **轻量 footprint** | 二进制几十 MB；冷启动 <100ms（对位 Quickwit 70ms，ES ~15s）；空载内存远低于 ES 最低 2GB | "比 ES 轻"的硬指标 |
| N3 | **低延迟检索** | 中等库（~百万 chunk）p95 全文 <50ms、向量 <50ms、混合 <100ms（指示性，需基准核实） | 对位 Meilisearch 即时搜索 |
| N4 | **确定性 / 可复现** | 同输入+同索引+同查询 → 稳定排序、可复现结果；并列项有确定 tie-break | 继承 docparse 确定性；便于 golden 测试 |
| N5 | **索引规模** | 单机目标几百万~千万级 chunk / 几十 GB；明确不追 100M+ 分布式 | 与 Meilisearch 同档；超出用 ES |
| N6 | **存储效率** | 控制索引膨胀（Meilisearch ~20× 是上限警示）；mmap/落盘可大于 RAM 工作集 | 避免 Typesense RAM 常驻上限 |
| N7 | **健壮性** | 坏 chunk/坏向量不中断整批；越界/缺失显式回退不 panic；近似兜底显式标注 | 继承 docparse 约定 |
| N8 | **契约稳定** | 索引文档/查询/结果三类 schema（JSON Schema draft 2020-12），golden 测试防漂移 | 继承 docparse；外部可 codegen 客户端 |
| N9 | **可观测** | 返回各路分数与融合分、命中解释（debug 模式）、索引/查询耗时统计 | 便于调相关性与排障 |
| N10 | **许可干净** | 全部依赖 MIT/Apache；注意 lindera 中文 CC-CEDICT 字典 CC-BY-SA，分发前审 | 继承 docparse Apache-2.0 边界 |
| N11 | **跨平台** | 引擎二进制 macOS / Linux / Windows；容器镜像 + Helm/k8s（CloudNativePG 式拓扑） | 生产部署形态 |
| N12 | **可用性/SLO（v3）** | 定义并守 SLO：检索可用性目标（如 99.9%）、p99 延迟目标、CDC 同步 lag 上界；只读副本滚动升级不掉服务 | 完善产品的服务承诺 |
| N13 | **容量信封（v3）** | 明确支持上限（chunk 数 / 集合数 / QPS / 库大小）与对应资源；越过上限有清晰"换 ES/Qdrant"指引；容量规划文档 | §4.5#12，诚实边界 |
| N14 | **可观测性是一等需求（v3）** | 指标/追踪/日志/慢查询/健康探针为必交付，不是事后加（F50） | 生产可运维 |
| N15 | **安全基线（v3）** | 默认安全（认证开、最小权限）、审计、加密选项；逐文档 ACL 不可绕过（F44） | 私有 RAG 合规 |

## 5. 范围与"不做什么"（边界）

> v2 修订：定位已从"单机嵌入式"改为"**外部单二进制引擎 + 托管 Postgres 真源**"，下表据此更新。v3 修订：以**完善产品**为目标，多项原"不做/远期"上提为必做（见 §4.5）。

**做（完善产品全集，§4.5 的 12 维度）：**
- 以托管 Postgres(pgvector) 为真源、引擎侧派生索引的**生产级**全文 / 向量 / 混合检索（§6.8）。
- 原生 docparse chunk 模型 + 引用溯源 + section 层级/auto-merging 检索。
- **完整查询能力**（typo/前缀/短语/分面/高亮/深分页/折叠/建议）、**相关性评测体系**、**向量两档 + 量化**。
- **安全与访问控制**（API Key/RBAC + 逐文档/集合级权限 + 审计）——私有 RAG 必备，**v3 提为必做**。
- **运维与可观测**（指标/追踪/慢查询/健康探针/优雅启停/限流）——**v3 提为必做**。
- **索引零停机生命周期**（蓝绿重建/快照恢复/格式迁移）、**CDC 健壮性**（exactly-once/自愈/DDL 演进）。
- **无状态只读副本 + LB 横向扩读**（F32）；四张脸 + Python/TS SDK + LangChain/LlamaIndex 适配。

**明确不做（真正的边界）：**
- ❌ **引擎自身的分布式分片/真复制协议（Raft 等）**——引擎无状态，扩展靠"多只读副本 + 同一 PG 真源"；真源 HA/复制/分区交给 Postgres（托管云已解决）。
- ❌ **做成 Postgres 扩展正面竞争 ParadeDB**（§6.8 路线 A 已否决）。
- ❌ **超大规模专用引擎（100M+ 向量、万级 QPS、跨节点分片）**——明确容量信封 + 越界线，超出建议 ES/Qdrant 集群（这是定义清楚的"边界"，不是"糊弄"）。
- ❌ **日志/可观测/时序/OLAP 分析框架**（ES 的另一战场，非我们用例）。
- ❌ **训练/微调嵌入或 rerank 模型**——只做推理，模型 opt-in。
- ❌ **取代 docparse 的解析**——只消费 chunk。
- ❌ **通用 SQL/脚本打分/parent-child join 查询语言**——只做检索 + 过滤 + 分面所需表达力。
- ⚠️ **多租户 / 认证 / 权限：从"不做"改为"做"**（§4.5 #9）——完善产品必须有 API Key/RBAC + 逐文档访问控制；不再甩给上层网关。REST 面向多人在线，默认安全（非裸绑 127.0.0.1）。

## 6. 技术选型建议与关键取舍

> **读法（重要）**：§6.1–6.5 是 v1"单文件嵌入式"假设下的选型分析，记录决策演进。**§6.6→6.7→6.8 是收敛过程，最终以 §6.8（v2 主决议）为准。**
> §6.1–6.5 哪些仍有效：**§6.3 向量索引（仅适用于 §6.8 子选择 b 引擎自建向量时）、§6.4 融合、§6.5 嵌入、分词/Tantivy 选型** 仍然有效。**已被取代**：§6.1/§6.2 的"存储 = redb/heed/LMDB 嵌入式 KV"——v2 真源是 Postgres，引擎侧只有 Tantivy 派生索引（mmap），不再需要嵌入式 KV 存储层。

### 6.1 主线技术栈（v1 嵌入式版，存储部分已被 §6.8 取代）

```
全文：Tantivy 0.26（BM25 + fast fields 过滤/排序/分面）
分词：lindera / charabia（中日韩 + 多语，语言自适应）
向量：hannoy 或 arroy（LMDB mmap，与存储同源）/ 备选 hnsw_rs（纯 Rust 内存+mmap）
融合：RRF 默认 + 分数归一化可选
嵌入：Candle（纯 Rust，include_bytes! 静态链 e5-small / bge-small）opt-in
存储：redb（纯 Rust）或 heed/LMDB（零拷贝读，与 arroy/hannoy 同源）
接口：clap(CLI) + axum(REST) + 手写 JSON-RPC(MCP)，复用同一查询核心
```

全部 MIT/Apache，唯一 C 依赖是选 heed/LMDB 时的 LMDB（可静态链，docparse 也接受 tract 这类原生依赖，先例在）。

### 6.2 取舍 A：存储 —— RAM 常驻 vs mmap 落盘

- **推荐 mmap 落盘（Tantivy mmap + LMDB/redb）**：工作集可大于 RAM，避免 Typesense RAM 常驻上限（28M 书 = 14GB RAM）；OS 页缓存管真实内存；对位 Meilisearch 的零拷贝读路径。文档 RAG 语料通常不大，这条让我们既轻又能扩。
- 取舍点：LMDB 长读事务在重写时会 pin MVCC 页导致文件膨胀（需注意）；redb 纯 Rust、无 mmap、规避了 LMDB 的地址预留/长读坑，但落盘略大。**若要与 arroy/hannoy 向量索引同源复用，选 heed/LMDB；若要纯 Rust 单文件最干净，选 redb（向量另用 hnsw_rs）。**

### 6.3 取舍 B：向量索引 —— 树(arroy) vs 图(HNSW/hannoy)

- **推荐图（hannoy 或 hnsw_rs）**：HNSW 搜索更快、召回更高，是事实工业标准（ES/Qdrant/usearch 都用）；hannoy 把 HNSW 建在 LMDB 上、与存储同源、号称比 arroy 搜索快 ~10×。
- 取舍点：树(arroy) 落盘友好、增量更新简单、内存可预测，但要很多树才够召回；图建索引更重、更吃内存。**MVP 可先用 hnsw_rs（纯 Rust、成熟、有 mmap）跑通，再评估 hannoy（与 LMDB 同源）。**

### 6.4 取舍 C：混合融合 —— RRF vs 分数归一化

- **推荐 RRF 做默认**：鲁棒、免归一化、对异构检索器稳健，与 ES Retriever API / Qdrant 对齐，用户心智一致；提供 `rank_constant`（默认 60）。
- **同时提供分数归一化/凸加权**（对位 Meilisearch 的 `distribution` 仿射变换 + `semantic_ratio`）作为可选——分数分布可比时它更优（Weaviate 实测 ~6% nDCG）。
- 决策：**两者都实现，RRF 默认**。这样既拿了 RRF 的稳健，又能在精调场景上 Meilisearch 式归一化。

### 6.5 取舍 D：嵌入 —— Candle vs ort/fastembed

- **推荐 Candle 做单文件主线**：纯 Rust，可 `include_bytes!` 静态链模型，真正单文件离线；CPU 略慢但可接受。默认模型 multilingual-e5-small（384 维，含中文）。
- **保留 ort/fastembed 作为高性能 opt-in 后端**：CPU 最快 + 一等 int8 + CoreML/CUDA，代价是带 native lib（非纯单文件）。**仿 docparse 的双后端思路（DocLayout-YOLO / PP-DocLayoutV2 按需选）。**

### 6.6 取舍 E：是否建在 Postgres + pgvector 上？（已决策）

> 这是一个**产品定位级**的岔路，不只是技术选型。专门调研了 Postgres + pgvector / ParadeDB 路线，结论与决策如下。

**Postgres 路线能做到什么**：pgvector 0.8.x（HNSW + halfvec + 二值量化）+ 原生 `tsvector`/GIN 或 **ParadeDB pg_search**（真 BM25，内嵌 Tantivy）+ SQL 里 RRF 融合 = 一套成熟的混合 RAG（Supabase 在主推）。单机 ~1–10M 向量没问题；**在线多人是 Postgres 的强项**（成熟 MVCC 并发、PgBouncer 连接池、流复制只读副本、RDS/Supabase/Neon/阿里云全托管）——上一轮"在线多人/性能"的问题在 PG 这边基本是已解决的。

**但针对本项目的四个关键坑**：
1. **原生 FTS 不是 BM25**：`ts_rank`/`ts_rank_cd` 无 IDF、无文档长度归一化（官方明说"不使用全局信息"），相关性明显弱于 Lucene/BM25。要真 BM25 得上 ParadeDB。
2. **中文分词是硬伤**：原生 PG 完全不会中文分词（整串当一词）。要 `zhparser`/`pg_jieba`（**C 扩展，托管 RDS/Aurora 上装不了**，只阿里云/字节云预装）；ParadeDB 内置 jieba/lindera 能解但要自托管 + AGPL。**对 localkb 中文为主 + docparse 中文 OCR 场景，这是要害。**
3. **pgvector 过滤/规模**：ANN 后过滤，选择性强的 `WHERE` 会饿死结果（AWS 实测仅返 ~1%），0.8 iterative scan 缓解、靠 partial index/分区才真解决；>10M 向量建索引数小时、HNSW 图要进 RAM。我们"按 doc/page/section/kind 过滤 + 向量"正撞这坑。
4. **连接模型**：进程级连接、`max_connections` 几百，突发流量必须 PgBouncer。

**最尖锐的洞察**：要在 PG 上拿真 BM25 + 中文分词，标准答案 ParadeDB pg_search **内嵌的正是 Tantivy**（fastsearch 主线本就要用的同一个库）——即"用 PG 做全文" = "在 PG 里跑 Tantivy"。fastsearch 直接用 Tantivy 就能省掉外面那层 Postgres 服务器。**一旦绑 Postgres，单文件零依赖的护城河和 requirement #1（"更轻量的 Rust 程序"）都不成立**——Postgres 是有状态 24/7 服务器，比单文件重得多。

**决策（2026-06-24 初版，⚠️ 已被 §6.8 取代）**：当时定为"默认单文件嵌入式 + 可选 PG 后端，MVP 不做 PG"。**此结论在深研 ParadeDB（§6.7）+ VectorChord/pg_textsearch（§2.5）后被推翻**——见 §6.8：最终改为"Postgres 作真源是默认架构，不是可选后端"。本节保留仅为记录决策演进，**不再有效**。

> 演进逻辑：§6.6 担心"绑 Postgres 会破坏单文件护城河"——这个担心在"做成 PG 扩展"（路线 A）时成立，但 §6.8 的"外部引擎 + 托管 PG 作真源"（路线 B）化解了它：引擎仍是单二进制，Postgres 是它的外部依赖而非把引擎塞进 PG，从而既得 Postgres 的托管/复制/可移植，又不掉进 ParadeDB 的 allowlist 陷阱。

### 6.7 ParadeDB 深研：结论与对 fastsearch 的影响（已决策修正）

> 专门对 [paradedb/paradedb](https://github.com/paradedb/paradedb) 做了源码级深研（`MVCCDirectory` / block storage / `LayeredMergePolicy` / custom scan）。ParadeDB 是"把 Tantivy 塞进 Postgres 做 BM25+混合"的参考实现，底座与 fastsearch 同为 Tantivy，因此既是**最近的对照系**，也是**最好的功能对标清单**。

**A. ParadeDB 的架构赌注**：把 Tantivy 存储**重写进 Postgres block storage**（8KB 页 + 共享缓冲池 + WAL + MVCC，v0.14.0 / 2025-01）。Tantivy 段以 8KB 块链表（`LinkedBytesList`）存在 PG 里，行可见性不进索引而是存 ctid fast field、扫描时回堆按快照核验。换来 WAL 崩溃恢复、物理复制、与堆表 ACID 一致——这是它相对所有"PG+外挂 ES"方案的根本差异。

**B. 这个赌注的代价**（源码/官方确认，全是 fastsearch 的机会）：
- 放弃了 Tantivy 的 **mmap 零拷贝**（fast field 改按需 deref + 缓存块）；
- **每个 segment + 每次 merge 都写 WAL** → 写放大；
- **社区版 BM25 索引锁死单节点**、**无原生分片**（横向扩展只能 Citus）；
- **装不了托管 RDS/Aurora/Cloud SQL**（`shared_preload_libraries` + 后台 worker 在 allowlist 外）→ 逼出"逻辑复制到自托管搜索副本"的旗舰部署；
- **AGPL-3.0**（网络条款，自托管改版要开源）；
- **直到 v0.24.0（2026-06）才把崩溃安全放进社区版**——之前 ≤0.23.x 持续并发写下有索引损坏/PANIC 高危 issue；
- **pg_search 不暴露 k1/b 调参**。

**C. 性能真相**（独立基准 vs 厂商）：ParadeDB 真正赢**建索引/摄入**（独立复现 ~2.5–4.5× 快于 ES）；但**独立基准（inevolin）显示 ES 在每个并发档搜索吞吐都赢**，且 **ParadeDB 索引磁盘占用更大**。厂商那些 20–1000× 是**对原生 PG FTS（常缺 GIN）**、非对 ES。结论：别被"PG 能比 ES 快几百倍"误导——那是 vs 羸弱基线。

**D. 核心洞察——"真源放哪"是这类引擎的根本抉择**：ParadeDB = **Postgres 是真源**（白嫖 ACID+复制，付写放大+单节点+allowlist）；Quickwit = **S3 是真源**（便宜弹性 PB 级，付最终一致+自建控制面）；fastsearch 单文件 = **自有存储是真源**（mmap 零拷贝+单文件+无 allowlist+自控 merge，付"自建持久化/复制/备份"+"放弃与源库 ACID 一致"）。三者都用 Tantivy 的 **`Directory` trait** 接存储——fastsearch 存储后端就插这里。

**E. 当时（§6.7）的倾向（⚠️ 部分被 §6.8 取代）**：此处一度倾向"standalone 自有存储是真源"（D 三选里的第三项），理由是 mmap 零拷贝 / 自控 merge / 无 allowlist。**但 §6.8 在同样的"真源三选"里最终改选了 Postgres-as-truth（D 的第一项，Quickwit 式外部引擎）**——因为它同样无 allowlist（pgvector 可移植）、还白嫖 PG 的复制/托管/备份，代价只是最终一致（对 RAG 灌库无碍）。所以：
- D 的"真源放哪"框架仍是理解全局的钥匙；**结论从"standalone"改成"Postgres 是真源、引擎在外"**（§6.8）。
- E 当时说的"自建持久化/WAL/复制是真实工作量"——**v2 已被化解**：PG 是真源，引擎侧派生索引可随时从 PG 重建，不必自建 WAL/复制/备份（见 §9 风险表对应修订）。
- F 的功能对标表不受影响，继续有效。

**F. 功能对标清单（fastsearch 要 match / 可 beat ParadeDB）**——两者同底座，这份清单直接细化 §3 全文/混合需求：
| 能力 | ParadeDB 做法 | fastsearch 目标 |
|---|---|---|
| 每字段多分词器 | cast 式 `col::pdb.ngram(3,3,'alias=...')` 同列多索引 | match：声明式 per-field tokenizer + alias |
| CJK 分词 | jieba / lindera(chinese/japanese/korean) / icu / chinese_compatible | match：至少 jieba + lindera + icu，质量排序 jieba>lindera≈icu>逐字 |
| 查询修饰 | `::pdb.fuzzy(n)`/`slop(n)`/`boost(f)`/`const(v)`，`pdb.parse`/`regex`/`more_like_this`/`range_term` | match 核心：fuzzy/phrase-slop/boost/bool/regex/parse；more_like_this 列 P1 |
| 相关性调参 | **k1/b 不可调**（硬伤） | **beat：暴露 k1/b + 自定义 Similarity** |
| 列存 fast fields | 单遍 faceting/聚合/排序，ES 兼容聚合 JSON | match：fast fields 做过滤/排序/分面（对应 F17/F19） |
| 高亮 | `pdb.snippet` + `snippet_positions`（byte 偏移） | match：文本片段高亮 + byte 偏移（对应 F20，与 bbox 互补） |
| 混合融合 | **无一等函数**，手写 RRF SQL（k=60） | match 即超越：把 RRF/归一化做成**一等内置**（F14），ergonomics 直接赢 |

> 一句话：ParadeDB 验证了"Tantivy 能扛生产 BM25"，也用它的全部约束（Postgres 绑定的代价）反向证明了 fastsearch 单文件路线的价值；它的功能面是我们最精确的对标尺，而它的两个软肋（k1/b 不可调、混合无一等融合）正是我们能立刻做得更好的地方。

### 6.8 最终架构决议：外部引擎 + 托管 Postgres(pgvector) 作真源（2026-06-24 已决，v2 主决议）

> 进一步深研了所有"用 Postgres 把检索做得比 ParadeDB 好"的现存项目（VectorChord-bm25、Tiger pg_textsearch、pg_bestmatch、ZomboDB）后，确认用 Postgres+pgvector 超越 ParadeDB 有**两条相反的路**，并最终选定路线 B。

**两条路的取舍**：
- **路线 A：做成 PG 扩展正面竞争**（VectorChord-bm25 / pg_textsearch 走的路）——自实现原生 `bm25` 索引 AM 配 pgvector。**致命问题：继承 ParadeDB 最大短板——同样需 `shared_preload_libraries` 原生扩展，进不了任何托管云 allowlist**；且要同时硬刚 ParadeDB(YC)、TensorChord、Tiger(Timescale) 三个有钱团队。**否决。**
- **路线 B：外部引擎 + 托管 PG/pgvector 作真源（选定）**——关键事实：**pgvector 在所有托管 PG 的 allowlist 上（RDS/Aurora/Supabase/Neon 全有），ParadeDB/VectorChord/pg_textsearch 一个都进不去**。让任意托管 Postgres 持有真源，fastsearch 作独立单二进制经 CDC 同步、自持派生索引、服务混合检索。**超越 ParadeDB 靠的就是钻这个它结构上过不去的缝。**

**为什么这条路最优（一并调和了整段对话）**：fastsearch 仍是单二进制（最初要的）；用 Postgres 拿到成熟并发/复制/托管（在线多人之忧）；pgvector 做持久化（现在要的）；引擎可作无状态只读副本横向扩展。架构原型 = **Quickwit 的控制面/数据面分离**（PG 当真源/metastore + 引擎当无状态服务层）。

**数据流**：
```
  docparse -f chunks ──▶ [写入] ──▶ ┌────────────────────────────┐
                                    │  Postgres（真源/持久层）    │
                                    │  chunks 表 + 元数据         │
                                    │  + pgvector 向量列          │
                                    └─────────┬──────────────────┘
                                              │ 逻辑复制 CDC（pgoutput / Sequin 式，亚秒、exactly-once）
                                              ▼
                          ┌──────────────────────────────────────┐
                          │  fastsearch（外部单二进制，可多副本） │
                          │  · 派生 BM25 倒排索引（Tantivy/mmap） │
                          │  · 向量 ANN：pgvector(SQL) 或 引擎自建 │
                          │  · 一等融合(RRF/归一化) + 预过滤 + rerank │
                          │  · 引用溯源(page+bbox+section)        │
                          └──────────────────────────────────────┘
                                  四张脸：CLI · 库 · REST · MCP
```

**超越 ParadeDB 的杠杆**（逐条坐实，细化 §3 需求）：
| 杠杆 | ParadeDB/pgvector 短板 | fastsearch |
|---|---|---|
| 托管 PG 可移植（最大的缝） | 装不了 RDS/Supabase/Neon | 外部引擎只需 pgvector，跑任意托管 PG |
| 一等混合融合 | 三家都无，手写 RRF SQL | RRF + 归一化内置（F14） |
| 真正预过滤 ANN | pgvector 后过滤召回崩（返~1%） | filter-aware 召回修掉 |
| BM25 可调 | pg_search k1/b 不可调 | 暴露 k1/b + 自定义 Similarity |
| 内置 rerank | 都没有 | 宽召回+分数保真+rerank（F15） |
| 引用原生 | 都没有 | page+bbox+section 端到端（F18） |
| PG 零写放大 | 每段+merge WAL 写进 PG | 倒排在引擎侧，保 mmap 零拷贝 |

**值得借鉴**：向量量化借 VectorChord 的 **RaBitQ**（1-bit、~192B/1536维、磁盘友好、自治重排）；BM25 top-k 用 **Block-Max WAND**；同步层借 **Sequin/Debezium** 逻辑复制（pgoutput 是托管 PG 上唯一可用插件；注意未消费 slot 会撑爆 WAL 磁盘，需 `max_slot_wal_keep_size` + 心跳 + 监控）。

**诚实的代价（写进文档让用户知情）**：
1. **派生 BM25 索引与 PG 亚秒级最终一致**（非同事务）——但这正是 ParadeDB 自己给托管用户推荐的"逻辑复制到搜索副本"同款模型，对 RAG 批量灌库无影响；我们只是把那台"带扩展的自托管 PG 副本"换成一个单二进制。
2. **要自建 CDC 同步层**（逻辑复制消费者，处理 slot 生命周期/断点续传/幂等）。模式成熟但是真实工作量。
3. **fastsearch 不再零依赖**——需要一个 Postgres。相对 v1 的"零运行时依赖单文件"是实质改变，已接受。

**仍待拍板的子选择——向量 ANN 在哪服务**：
- **(a) 用 pgvector 在 PG 里跑 ANN**：托管最省事、向量真源与服务同处；但继承 pgvector 后过滤短板 + 进程级连接并发；引擎只发 SQL 取向量臂、本地融合。
- **(b) 引擎侧自建向量索引**（从 PG 同步向量）：延迟/融合/预过滤/量化全可控（可上 RaBitQ），能真正"修掉 pgvector 的坑"；但要同步向量、pgvector 退化为纯持久层。
- 倾向（完善产品**两档都交付**）：**(b) 更能兑现"比 ParadeDB/pgvector 好"**；推进次序上可先 (a) 打通端到端再补 (b)，但 1.0 GA 两者都要、运行时可切。trait 边界须容纳两者。见 §11。

## 7. 架构草图（v2，待开发计划细化）

> 权威数据流见 §6.8。下图补充检索核心与分层。真源在 Postgres，引擎侧只持**派生索引**（可随时从 PG 重建）。

```
   docparse -f chunks ──▶ [写入 PG]
                              │
        ┌─────────────────────────────────────────┐
        │  Postgres（真源/持久层）                  │  ← 用户的托管 PG（RDS/Supabase/Neon）
        │  chunks 表 + 元数据 + pgvector 向量列      │
        └─────────────────┬───────────────────────┘
                          │ 逻辑复制 CDC（pgoutput, 亚秒, 幂等, 断点续传）
                          ▼
   ┌───────────────────────────────────────────────────────────┐
   │  fastsearch（外部单二进制，可多副本，无状态读）            │
   │  派生索引层：                                              │
   │   ┌────────────┐  ┌──────────────┐  ┌──────────────────┐  │
   │   │ 全文倒排    │  │ 向量 ANN      │  │ 元数据 fast field │  │
   │   │ Tantivy    │  │ (a)pgvector   │  │ + section docstore│  │
   │   │ BM25+CJK   │  │ (b)引擎自建    │  │  (过滤/分面)      │  │
   │   │ +Block-WAND│  │  +RaBitQ量化  │  │                   │  │
   │   └────────────┘  └──────────────┘  └──────────────────┘  │
   │  检索核心：keyword ∥ semantic 召回                          │
   │   → 预过滤(filter-aware, doc/page/section/kind)            │
   │   → 一等融合(RRF / 归一化, semantic_ratio)                 │
   │   → [auto-merging 上提 section] → [rerank 钩子]            │
   │   → 命中:{chunk_id,doc_id,page,bbox,heading_path,score,hl} │
   └───────────────┬───────────────────────────────────────────┘
                   ▼
   四张脸：CLI · 库 · REST(axum) · MCP(stdio)  ——共用检索核心
                   ▼
   答案层(外部 LLM)：resolve_citation(id) → {page,bbox} → 深链/高亮
```

**Cargo workspace 分层（v2）：**
- `fastsearch-core`：文档模型、查询 AST、过滤、融合、排序——不依赖具体后端（trait 边界）。
- `fastsearch-sync`：**（v2 新增）** CDC 同步层——pgoutput 逻辑复制消费者、初始快照、断点续传(LSN)、slot 生命周期、`doc_id` 级替换。
- `fastsearch-text`：Tantivy（BM25 + Block-Max WAND，暴露 k1/b）+ CJK 分词（lindera/jieba/icu）。
- `fastsearch-vector`：向量后端 **trait**——实现 (a) pgvector(SQL 直查) 与 (b) 引擎自建(hnsw_rs + RaBitQ)，两者可切。
- `fastsearch-embed`：嵌入后端（Candle / ort）trait + 实现，opt-in。
- `fastsearch-pg`：Postgres 接入——schema 约定、写路径、pgvector 读写、复制连接。
- `fastsearch-eval`：**（v3）** 相关性评测体系——golden 集、nDCG/recall 算分、CI 回归门禁（F39）。
- `fastsearch-server`：**（v3）** REST(axum) + MCP + 认证/RBAC/ACL（F43–F46）+ 可观测（指标/追踪/限流，F50）+ 副本编排。
- `fastsearch-cli`：CLI，复用 core/server。
- 客户端：`clients/python`、`clients/typescript`（+ LangChain/LlamaIndex 适配，F55），仿 docparse。

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| **相关性调优是长期工程**（融合/归一化/分词权重） | 检索质量不达预期 | 早建 golden 评测集（用 docparse 真实 chunk）、可观测分数、RRF 稳健兜底 |

| **索引膨胀**（Meilisearch ~20×） | 磁盘占用 | 监控索引/文档比，按需关前缀/邻近库，提供精简档 |
| **docparse chunk schema 演进** | 集成漂移 | 对 chunk schema 做 golden 测试，跟随 docparse schema 版本 |
| **派生索引崩坏/不一致**（v2：但 PG 是真源） | 索引坏 | 因 PG 是真源，派生 BM25 索引**可随时从 PG 重建**（F33）——崩溃恢复=重放复制流，门槛远低于 standalone 自建持久化 |
| **复制 slot 撑爆 WAL 磁盘**（v2 新增，CDC 头号坑） | 拖垮用户的 PG | `max_slot_wal_keep_size` + 心跳 + 监控 `pg_replication_slots`；引擎挂掉/落后要告警；slot 失效要能重新快照 |
| **CDC 同步层复杂度/延迟**（断点续传、幂等、删除捕获、初始快照与增量切换） | 漏数据/重复/不一致 | 参照 Sequin/Debezium 成熟模式；at-least-once + sink 幂等（按 LSN/PK）；同步延迟可观测；端到端一致性测试入回归 |
| **托管 PG 可移植被破坏**（误引入需原生扩展的依赖） | 丢掉对 ParadeDB 的核心优势 | 把"PG 侧只用 pgvector + 逻辑复制"作为 CI 守护的硬约束（N1b/F31）；任何 PG 侧扩展依赖需评审 |
| **pgvector 后过滤召回坑**（若选向量 ANN 在 PG 跑，§6.8 子选择 a） | 强过滤时召回崩 | 用 0.8 iterative scan / partial index / 分区；或选子方案 (b) 引擎侧自建向量索引做 filter-aware 召回 |
| ~~自建持久化/并发写损坏~~（v1/standalone 风险，**v2 已被 PG 真源化解**） | — | 保留记录：选 PG-as-truth 后，持久化/复制/备份交给 PG，引擎侧索引可重建 |

## 10. 验收标准（1.0 GA 产品级 · 对齐 §4.5 十二维）

> v3：从"MVP 验收"升级为"**完善产品验收**"。1.0 GA 须全部通过；阶段（§8）逐项点亮。

**A. 检索质量**
1. 三检索（keyword/vector/hybrid）各自生产级；混合在 golden 集上 nDCG@10 ≥ 单路最优，且融合 RRF/归一化/加权三法可切。
2. **相关性回归门禁**：golden 集 nDCG/recall@k 进 CI；改分词/融合/权重导致掉点会卡住合并（F39）。
3. 查询能力完备：typo/前缀/短语/邻近/布尔/范围/正则/分面/高亮/深分页/折叠全部可用并测过（F35–F37）。
4. CJK：中文 jieba/lindera/icu 可选 + 用户词典生效；分词质量有回归用例。

**B. 数据与一致性**
5. **PG 真源 + 健壮 CDC**：写 PG 后亚秒可检索；删/改 `doc_id` 正确反映；exactly-once 效果、严格行序；引擎重启从 LSN 续传或快照重建一致；slot 不泄漏 WAL（F51–F53）。
6. **托管 PG 可移植**：全流程跑在"未装任何原生扩展、仅 pgvector + 逻辑复制"的 PG 上（等价 RDS/Supabase 约束，N1b）。
7. **零停机生命周期**：换分词器/模型/schema 时蓝绿重建 + 原子切换，在线查询不中断（F47）；索引格式版本化可在线迁移（F49）。

**C. 安全（私有 RAG 硬门槛）**
8. REST 默认要求认证（F43）；**逐文档/集合 ACL 不可绕过**——构造越权查询无法命中无权 chunk（F44）；审计日志完整（F45）。

**D. 运维与服务**
9. 可观测：指标（QPS/p50-p99/召回/CDC lag/资源）、追踪、慢查询、健康探针、优雅启停、限流全部就绪（F50/N14）。
10. SLO/HA：定义并实测可用性/p99/lag SLO；只读副本滚动升级不掉服务（N12/F32）；容量信封 + 越界指引文档（N13）。

**E. 集成与契约**
11. 四张脸（CLI/库/REST+OpenAPI/MCP）契约一致、JSON Schema + golden 防漂移（F54/N8）；Python/TS SDK + LangChain/LlamaIndex 适配可用（F55）。
12. 引用闭环：命中 `page+bbox+section` 全程不丢、ID 可反解，与 docparse `locate` 对得上（F18）。

**F. 工程基线**
13. 健壮：坏 chunk/向量不中断整批、不 panic；越界/缺字段显式回退；并发写/同步压测无损坏。
14. 确定性：同稳定索引 + 同查询 → 同结果、稳定 tie-break。
15. **越 ParadeDB 实证**：公开方法的基准（vs ParadeDB/ES/Meilisearch），在可移植性、融合、k1/b 可调、预过滤、引用溯源上展示优势。
