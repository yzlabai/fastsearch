## 0. TL;DR

- **四个场景**（来自原始需求）：
  1. **文档内嵌图**：检索命中后，生成的答案里要能把图**展示**出来（图 ↔ 上下文绑定 + 引用溯源到图区域）。
  2. **图片按内容搜**：以文搜图 / 以图搜图（跨模态语义）。
  3. **语音**：入库后按**转录文本**检索，命中要能定位到**音频片段**（时间区间）。
  4. **视频**：搜到**相关片段**（时间区间 + 关键帧/转录）。
- **一句话结论**：四个场景**不需要新架构，是现有架构的自然延伸**。现有的 `Chunk`/`GlobalId`/`Citation`/混合检索管线/CDC 已经把"可溯源的内容单元"抽象好了；多模态 = ①把"片段(segment)"的语义从"页+bbox"扩展到"**媒资指针 + 时间区间/区域**"，②把"嵌入"从"纯文本一种"扩展到"**按模态选 embedder**"，③把"召回"从"BM25∥文本向量"扩展到"**BM25(转录/caption) ∥ 跨模态向量**"。
- **三种入库路线，按"语义损失/成本"递增，可叠加**：
  1. **转录/描述派生文本**（语音→转录、图/视频→caption）——**零新依赖**，直接复用现有 BM25 + 文本向量管线，**P0 首选**，已能覆盖"语音按转录检索""文档图按上下文展示"。
  2. **跨模态单向量嵌入**（CLIP/SigLIP-2/JinaCLIP-v2/jina-v4 单向量）——经现有 `HttpEmbedder` 多模态扩展接入，**P1**，覆盖"以文搜图/以图搜图""视频关键帧检索"。
  3. **视觉文档/多向量后期交互**（ColPali/ColQwen2，OCR-free）——重，引擎侧多向量 MaxSim，**P2 opt-in**。
- **硬约束的关键影响（必须先想清楚）**：
  - **不变量 #1（托管 PG 只用 pgvector + 逻辑复制，不装原生扩展）**：pgvector 核心**不支持多向量/MaxSim 后期交互**（那需要 VectorChord 等原生扩展，违约）。→ **ColPali 式多向量检索只能放在引擎侧派生索引里做，PG 真源只存原始向量（jsonb/数组）或不存**。这反而印证了"PG 真源 + 引擎派生"的架构是对的。
  - **大二进制媒资（图片/音/视频文件）不进 PG bytea**：PG 只存**媒资指针(URI/对象存储 key) + 元数据 + 向量**，原始字节走对象存储。这是一个需拍板的新决策（见 §6.1）。
  - **不变量 #3（ACL 不可绕过）**：媒资深链 / 缩略图 / 字节回放必须同样经服务端 ACL，不能给客户端裸 URL 绕过权限（见 §5.7）。
- **现状**：`ChunkKind::Image` 与 `ImageMeta`（含 `caption`/`file`/`media_type`）**已存在**于 [`core/src/model.rs`](../../crates/fastsearch-core/src/model.rs)，是一个"半成品锚点"——已为图预留，但 `text` 字段强制非空、无 audio/video、无时间区间、嵌入按模态选择未做。

## 1. 背景与范围

### 1.1 现状：引擎已经把"可溯源内容单元"抽象好了

fastsearch 的核心抽象是 **`Chunk`（内容单元）+ `GlobalId`（稳定标识）+ `Citation`（溯源坐标）**，检索管线对所有 chunk 一视同仁：`ACL 注入 → keyword∥vector 召回 → 融合 → rerank → 高亮/分面 → top-K`。多模态的本质诉求——"把一段图/音/视频当作可被检索、可被引用、可被展示的内容单元"——**正好落在这个抽象上**。所以本文的基调是**扩展，不是重写**。

关键代码锚点（改造清单见 §5，签名以代码为准）：

| 关注点 | 位置 | 现状 |
|---|---|---|
| `Chunk` / `ChunkKind` / `ImageMeta` | [`fastsearch-core/src/model.rs`](../../crates/fastsearch-core/src/model.rs) | `ChunkKind` 已含 `Image`；`ImageMeta{caption,caption_source,file,media_type}` 已存在；`text: String` **强制非空**；无 audio/video、无时间区间 |
| `GlobalId` / `Citation` / `citation_id` | 同上 | `(collection,doc_id,chunk_id)`；`Citation{page,bbox,heading_path,section_id}` —— **bbox 是 PDF 2D 坐标，无时间维** |
| PG 真源 DDL / 行映射 | [`fastsearch-pg/src/sql.rs`](../../crates/fastsearch-pg/src/sql.rs) | 列含 `kind,text,page,bbox,heading_path,section_id,image_meta(jsonb),tenant,acl`；单 `embedding vector(dim)/halfvec(dim)` 列 |
| `Embedder` trait / `HttpEmbedder` | [`fastsearch-embed/src/`](../../crates/fastsearch-embed/src/) | `embed(&[String], EmbedKind)` —— **只吃文本**；HTTP 后端走 Ollama / OpenAI 兼容 `{model,input}` |
| 向量后端 trait / `MemVectorIndex` | [`fastsearch-vector/src/lib.rs`](../../crates/fastsearch-vector/src/lib.rs) | 单向量/chunk；filter-aware 真预过滤；`VecMeta` 无模态字段 |
| 文本/BM25 schema | [`fastsearch-text/src/`](../../crates/fastsearch-text/src/) | Tantivy `text` 字段 STORED+INDEXED；无 caption/转录专列 |
| 引擎管线 / CDC sink | [`fastsearch-engine/src/lib.rs`](../../crates/fastsearch-engine/src/lib.rs) | `run()`；`SearchRequest.vector: Option<Vec<f32>>`（单向量）；CDC `apply_upsert` 只嵌 `chunk.text` |
| CDC 变更模型 | [`fastsearch-sync/src/lib.rs`](../../crates/fastsearch-sync/src/lib.rs) | `Change::{Upsert,Delete,DeleteDoc}`；`IndexSink`；LSN 水位幂等 |

### 1.2 范围

**纳入**：图（文档内嵌图、独立图片）、语音、视频四类的**入库表示、检索召回、融合排序、溯源/展示**。
**暂不纳入（标注但不展开）**：实时流媒体检索、媒资本身的转码/缩略图生成流水线（属上游 docparse / 媒资服务职责）、端到端 LLM 答案生成（属答案层）。fastsearch 的边界仍是"**检索 + 溯源**"，多模态不改变这个边界。

## 2. 四个场景 → 需求拆解

> 记号：以下"片段(segment)"指一个 `Chunk`，"媒资(asset)"指底层图片/音/视频文件。

### 2.1 场景 A：文档内嵌图，答案里要能展示

- **入库**：docparse 解析文档时，图作为一个 `kind=Image` 的 chunk，携带 ①所在 `page`+`bbox`（图在原文档的位置）、②`caption`/周边文字、③媒资指针（裁出的图片 key 或"父文档页+bbox"足以渲染）。图 chunk 通过 `heading_path`/`section_id`/相邻 `chunk_id` 与上下文绑定。
- **检索**：用户查询文字时，图 chunk 经 **caption/周边文字的 BM25 + 文本向量**被召回（P0 即可）；进一步可加跨模态向量让图本身的视觉内容也可被文搜（P1）。
- **展示/溯源**：命中返回的 `Citation` 要能让答案层**渲染这张图**——即 `Citation` 需带"如何取到这张图的字节/深链"（媒资 key 或 `doc_id#page&bbox`）。这是场景 A 的**核心新需求**：溯源从"跳转到页"升级到"内联渲染媒资"。
- **验收**：查"2023 年毛利率趋势" → 命中文档里的趋势图 chunk → 答案层能内联显示该图并标注来源页。

### 2.2 场景 B：图片按内容搜（跨模态）

- **入库**：独立图片作为 `kind=Image` chunk，存**跨模态视觉向量**（CLIP/SigLIP-2/JinaCLIP-v2…）；可选附 caption。
- **检索**：**以文搜图**（query 文本 → 文本编码到同一空间 → ANN 召回图向量）与**以图搜图**（query 图 → 视觉编码 → ANN）。要求文本与图编码进**同一向量空间**（统一/对齐模型，见 §3.1）。
- **关键技术点**：`SearchRequest` 需支持"以图为 query"（当前只有文本/单向量 query），且需让某个模态的查询只在对应模态/或跨模态子集里召回（modality-aware 召回）。
- **验收**：传一张"红色跑车"图或文字 → 召回库里视觉相似的图片，按相似度排序。

### 2.3 场景 C：语音，按转录检索 + 定位到音频片段

- **入库**：上游（Whisper/WhisperX 类）把音频转录成**带毫秒时间戳的句/段**；每段成为一个 chunk：`text=转录文本`、新增 `time_start_ms/time_end_ms`、媒资指针指向音频文件。
- **检索**：转录文本走**现有 BM25 + 文本向量**（**零新依赖，P0 直接可用**）。可选叠加音频自身的声学向量（说话人/音色检索，远期）。
- **溯源/回放**：`Citation` 带 `media_offset_ms`/`duration_ms`，答案层据此**深链到音频该时间点回放**（`audio.mp3#t=83.2,91.5`）。
- **验收**：查"客户提到的违约条款" → 命中某通话转录段 → 可一键跳到音频 1:23 处播放。

### 2.4 场景 D：视频，搜相关片段

- **入库**：视频先**场景分割**（TransNetV2 类）+ **关键帧抽取**，每个场景/片段成为一个 chunk，携带：①时间区间、②关键帧的跨模态视觉向量、③该片段的转录文本（ASR）、④可选 caption/字幕。这是图(关键帧)+语音(转录)+时间区间三者的**复合**。
- **检索**：转录 BM25/文本向量 ∥ 关键帧跨模态向量，融合后定位到**片段时间区间**；可选"时刻检索(moment retrieval)"做边界细化（远期）。
- **溯源**：同 C，`Citation` 带时间区间 + 可选关键帧引用，深链到 `video.mp4#t=...`。
- **验收**：查"演示新功能的那段" → 召回视频对应片段并能从该时间点播放。

### 2.5 四场景的需求公因子

抽象出来，四个场景只引入**三个正交的新概念**，全部可加性扩展现有抽象：

1. **时间区间维**（C/D）：`Citation`/`Chunk` 增加 `time_start_ms/time_end_ms`（与现有 `page/bbox` 并列，bbox 是空间维、time 是时间维）。
2. **媒资指针 + 服务端可控的字节/深链解析**（A/B/C/D）：`resolve_citation` 升级为能返回"如何安全取到这段媒资"。
3. **按模态选 embedder + 跨模态向量空间**（B/D，及 A 的增强）：嵌入管线从"一种文本 embedder"变为"按 `chunk.modality` 路由到对应 embedder，且文本/视觉共享空间"。

## 3. 技术选型调研（2026-06）

### 3.1 跨模态嵌入：统一空间 vs 分模态

- **结论先行**：场景 B/D 的跨模态检索**要求文本与图进同一向量空间**；分模态各建索引再拼结果"几乎做不到真正的跨模态检索"。因此选**统一/对齐的多模态 embedder**，而非每模态独立模型。
- **候选模型**（均可经 HTTP 后端接入，符合"不进程内推理"的现有决策）：
  - **SigLIP-2**（Google，sigmoid loss）：2026-06 最强开源图文相似模型，细粒度图检索强；纯图文。
  - **JinaCLIP-v2**：多语(89 语) + Matryoshka(可截断维度) + 8192 token 长上下文，单向量图文。
  - **jina-embeddings-v4 / v5-omni**：**通用多模态多语**，同时支持**单向量与多向量(后期交互)**两种模式——对本项目很关键，可单模型覆盖 P1(单向量) 与 P2(多向量) 两档。
  - **Voyage-multimodal-3.5**（MongoDB）/ **Cohere Embed-v4**：商用，文本/图/(视频)，托管 API。
  - **音频**：声学嵌入(CLAP 类)为远期；**P0/P1 的语音检索走转录文本**，不需要音频专用 embedder。
- **维度与 pgvector 约束**：pgvector `vector` 类型 HNSW 索引上限 **2000 维**，`halfvec`（半精度，2 字节/维）把上限提到 **4000 维**且省一半存储、质量损失小。多模态向量常 768~1536 维，落在 `vector`/`halfvec` 范围内 → **真源存储无障碍**。Matryoshka 模型可按需截断维度控成本。

### 3.2 视觉文档检索（OCR-free）：ColPali / ColQwen2

- **是什么**：把 PDF 页**整页渲染成图**直接嵌入，用**后期交互(late interaction, MaxSim)** over patch 级多向量打分（ColBERT 思想搬到视觉）。ColPali(PaliGemma 3B)→ ColQwen2(Qwen2-VL) 为 2026 ViDoRe SOTA，**跳过 OCR 也能赢最强文本 RAG**，尤其在表格/图表/版式重的页面。ColPali 每页 1024 个 patch 向量(各 128 维)。
- **对本项目的价值与代价**：
  - 价值：场景 A 的"文档内嵌图/图表"和"版式重文档"召回质量天花板更高，且省去对图的 caption 依赖。
  - **代价/约束冲突**：多向量 MaxSim **pgvector 核心不支持**（需 VectorChord 原生扩展 → **违反不变量 #1**）。
  - **化解**：本项目"**PG 真源 + 引擎派生索引**"的架构天然适配——**MaxSim 多向量检索放在引擎侧 `vector` crate 实现**（patch 向量集 + MaxSim 打分），PG 真源只存"页图指针 + patch 向量(jsonb 或多行)"或干脆不存派生向量。这样既拿到 ColPali 的质量，又不在 PG 装任何原生扩展。存储优化(patch 剪枝/量化/Prune-then-Merge)有现成研究可循。
  - **定位**：**P2、opt-in、重依赖**，按 §6 诚实标注"下一迭代/待运行验证"。

### 3.3 语音检索：转录优先

- **主路线**：上游 WhisperX 类输出**毫秒级时间戳的结构化 JSON**（句/段边界），按 ~3 句切 chunk，转录文本进现有文本管线。**这是 zero-new-dependency 的 P0**，覆盖场景 C 主诉求。时间戳→音频深链是纯元数据透传。
- **远期**：声学嵌入(说话人/情绪/非语音音频事件检索)、长音频事件级 QA(LongAudio-RAG 类) —— 标注为远期，不在本期。

### 3.4 视频检索：场景分割 + 关键帧 + 转录复合

- **流水线**（上游/媒资服务职责，fastsearch 消费其产物）：场景分割(TransNetV2) → 每场景抽 4 关键帧(首/中×2/尾) → 关键帧跨模态向量 + 该段 ASR 转录 → 去重(cosine) → 片段 chunk。
- **检索**：转录文本 ∥ 关键帧视觉向量，融合定位片段；**时刻检索/边界细化(temporal moment retrieval)** 做精定位为远期增强。SceneRAG 等把视频按"事件场景"切分的思路与本项目"chunk=可溯源单元"一致。
- **定位**：P1(关键帧向量 + 转录) 即可覆盖"搜到相关片段"；精细时刻边界 P2。

### 3.5 选型小结（按场景 × 路线）

| 场景 | P0（转录/描述派生文本，零新依赖） | P1（跨模态单向量，HTTP 后端） | P2（多向量/精定位，opt-in） |
|---|---|---|---|
| A 文档图展示 | caption/周边文字 BM25+文向量 ✓ + 媒资展示溯源 | 图视觉向量增强召回 | ColPali 整页视觉检索 |
| B 图片搜 | （caption 兜底） | **SigLIP-2/Jina-v4 以文搜图/以图搜图** ✓ | — |
| C 语音 | **WhisperX 转录 BM25+文向量 + 时间深链** ✓ | — | 声学/说话人嵌入 |
| D 视频 | 转录 BM25+文向量 + 时间深链 | **关键帧视觉向量 + 转录融合** ✓ | 时刻边界细化 |

## 4. 与现有架构的契合度（守不变量）

逐条对照 [CLAUDE.md](../../CLAUDE.md) 的关键不变量，确认多模态改造**不破坏**它们：

1. **#1 托管 PG 只用 pgvector + 逻辑复制**：✓ 守住。多模态向量是普通稠密向量(`vector`/`halfvec`)，pgvector 原生支持；唯一会越界的"多向量 MaxSim(ColPali)"**放到引擎侧派生索引**，PG 不需要任何原生扩展。媒资字节走对象存储，不进 PG。
2. **#2 PG 真源 / 引擎派生**：✓ 强化。媒资字节 = 对象存储；媒资**元数据 + 向量真源**在 PG；引擎侧的 BM25 / ANN / ColPali 多向量索引全部"可重建"。崩溃恢复仍是重放复制流 / 从快照重建。**不引入"只在引擎有的权威媒资数据"**。
3. **#3 ACL 不可绕过**：⚠ 需主动设计。媒资深链/缩略图/字节回放是**新的数据出口**，必须同样经服务端 ACL（见 §5.7）——不能签发绕过 `engine.search` ACL 注入的裸 URL。
4. **#4 确定性**：✓ 同分仍按 `GlobalId` tie-break；跨模态融合复用现有 `fuse`（RRF/归一化），多召回源只是多一路输入。
5. **#5 预过滤策略**：✓ 复用。模态过滤(`modality in [...]`) 作为 `Filter` 子句，翻译成后端 SUPERSET 查询 + `core::Filter::eval` 精确后过滤，与现有 text/vector 预过滤同构。
6. **#6 重依赖 opt-in、诚实记账**：✓ 本文已按此标注（P2/ColPali/声学嵌入/时刻检索 = 下一迭代/待验证）；真语义嵌入仍走外部 HTTP，不引进程内模型推理。

**结论：四个场景与现有架构/不变量高度契合，无需架构级返工。** 主要工作量在数据模型加字段、嵌入管线按模态路由、溯源升级为媒资感知、以及一个新的"媒资字节/深链的 ACL 网关"。

## 5. 改造清单（按 crate，自底向上）

> 下列为**需求级改造方向**，非最终签名；落地时以代码为准并回写 specs。原则：**加字段优先于改语义**、**`Option` 化保后向兼容**、**trait 边界不破**。

### 5.1 `core`（数据模型）— [model.rs](../../crates/fastsearch-core/src/model.rs)

- `Chunk.text: String` → `Option<String>`（纯媒资 chunk 可无文本）；管线里凡假设文本非空处需审计。
- `ChunkKind` 增 `Audio`、`Video`（`Image` 已有）。
- 引入 `modality` 概念（`text|image|audio|video`），用于嵌入路由与召回过滤——可由 `ChunkKind` 派生或显式存。
- `ImageMeta` 泛化为 `MediaMeta`（保留 `caption/file/media_type`，新增音视频所需）：媒资指针(对象存储 key/URI)、`time_start_ms/time_end_ms`、`duration_ms`、关键帧引用、缩略图 key。
- `Citation` 增时间维与媒资字段：`time_start_ms/time_end_ms`、媒资指针、可选"区域/帧"坐标；`page/bbox` 保持向后兼容（PDF 路径不变）。
- `citation_id` 格式不变（`collection:doc_id:chunk_id`），时间/媒资信息走 `Citation` 字段而非 id 编码。

### 5.2 `pg`（真源 DDL / 行映射）— [sql.rs](../../crates/fastsearch-pg/src/sql.rs)

- DDL 增列：`media_meta jsonb`（替/扩 `image_meta`）、`modality text`、`time_start_ms int`、`time_end_ms int`、媒资指针列。
- 大二进制**不进 PG**：只存对象存储 key/URI（见 §6.1）。
- 单 `embedding` 列的多模态策略（见 §6.2）：最小改动是"按模态选模型但仍单列(同一空间)"；如需多向量(ColPali)则用 `jsonb`/旁表，且**该派生向量可不进 PG 真源**。
- `ChunkRow` 映射 + `insert_sql`/`fetch_doc_sql` 同步加参数；更新 DDL 测试。

### 5.3 `embed`（嵌入管线）— [fastsearch-embed/src/](../../crates/fastsearch-embed/src/)

- `Embedder::embed` 从"只吃 `&[String]`"扩展为可吃**多模态输入**（文本 / 图字节 / 音视频特征），或新增 `embed_multimodal`，按 `modality` 路由。
- `HttpEmbedder` 支持多模态后端：图走 base64 / OpenAI 兼容多模态 `input`，对接 SigLIP-2 / JinaCLIP-v2 / jina-v4 / Voyage / Cohere 的 HTTP API。
- `EmbedderConfig` 增模型能力声明（支持哪些模态、维度、是否同一空间），引擎据此校验"query 模态 ↔ 索引模态"是否可跨。
- 确定性基线(HashEmbedder/LexicalOverlap)对图/音仍可退化为"对 caption/转录做文本嵌入"，保 CI/离线可跑。

### 5.4 `vector`（向量后端）— [lib.rs](../../crates/fastsearch-vector/src/lib.rs)

- `VecMeta` 增 `modality`、时间区间字段，使召回结果自带模态与时间信息。
- 召回支持**模态过滤**（只在 image / 跨模态子集里搜），复用现有 filter-aware 预过滤。
- 单向量路径(P1)无需改打分；**多向量 MaxSim(P2/ColPali)** 在此 crate 内新增一种后端/打分模式（patch 向量集 + MaxSim），与现有 `MemVectorIndex` 并列，opt-in。

### 5.5 `text`（BM25）— [fastsearch-text/src/](../../crates/fastsearch-text/src/)

- `text` 字段 `Option` 化；新增 caption / 转录的索引字段（或复用 `text` 字段灌入转录/caption，再用 `kind`/`modality` 区分）——倾向**复用 `text` 字段**（转录/caption 本就是该 chunk 的"文本表示"），最小化 schema 变更。
- 高亮对转录文本同样可用（场景 C/D 高亮命中句）。

### 5.6 `engine`（管线 + CDC sink）— [lib.rs](../../crates/fastsearch-engine/src/lib.rs)

- `SearchRequest`：支持**以图/媒资为 query**（非纯文本 query）、`modality_filter`、必要时多路向量 query。
- `run()`：召回阶段按模态可能并发多路向量召回 + BM25(转录/caption)，融合复用 `fuse`；`SearchHit`/`Citation` 透出模态与时间区间。
- CDC `apply_upsert`：从"只嵌 `chunk.text`"改为"**按 `modality` 选 embedder**"——image→视觉向量、audio/video→转录文本向量(+可选关键帧向量)、text→文本向量。**这是 CDC 落地嵌入主循环的关键改动点**（现主循环见 commit 3863dee）。

### 5.7 媒资字节 / 深链的 ACL 网关（**新**，跨 server/engine）— [fastsearch-server](../../crates/fastsearch-server/src/)

- 新增受 ACL 保护的端点：`GET /asset/{citation_id}`（按时间区间裁剪音视频 / 返回图字节 / 缩略图），**复用 `principal_from_headers → acl_for → 可见性校验**，命中才放行。
- 绝不给客户端可绕过权限的对象存储裸 URL；如需直链对象存储，用**短时签名 URL 且仅在 ACL 校验通过后签发**。
- `resolve_citation` 升级为"返回**安全的媒资获取方式**（受控端点 / 短签 URL）+ 时间区间 + 渲染所需元数据"，供答案层内联展示（场景 A 核心）。

### 5.8 `sync` / CDC — [lib.rs](../../crates/fastsearch-sync/src/lib.rs)

- `Change::Upsert` 语义不变（chunk 携带媒资指针即可，**不在 CDC 流里搬大二进制**——字节已在对象存储)。
- 幂等/LSN 续传不受影响；删除 doc 时需联动媒资清理策略（可异步 GC，标注远期）。

## 6. 待拍板的关键决策

### 6.1 媒资字节存哪？（**优先级最高**）
- **倾向**：对象存储(S3/MinIO/兼容)，PG 只存 key/URI。理由：PG bytea 存大二进制有 TOAST/WAL/复制放大代价，且违背"PG 轻量真源"基调；逻辑复制搬 GB 级二进制不现实。
- **取舍**：引入对象存储依赖（与"单二进制 + 一个 PG"的轻量叙事有张力）。需明确：自带/可选/由用户提供 bucket。
- **替代**：小图(如裁出的图表，KB 级)可考虑进 PG bytea 或 jsonb base64；音视频必走对象存储。

### 6.2 多模态向量在 PG 真源怎么存？
- **选项 A（最小）**：仍单 `embedding` 列，按模态选模型但都映射到**同一空间/同一维度**。简单、pgvector 友好。要求选统一模型(jina-v4 类)或维度对齐。
- **选项 B**：每模态一列(`embedding_text/embedding_image…`)或旁表。灵活但复杂、稀疏。
- **选项 C（ColPali 多向量）**：patch 向量集存 `jsonb`/旁表，且**可只存在引擎派生层、不入 PG 真源**（因其是纯派生、可由页图重算）。
- **倾向**：P1 用 A；P2 ColPali 用 C（不污染 PG 真源）。

### 6.3 由谁产出转录/caption/关键帧？
- **倾向**：上游 docparse / 专门媒资预处理服务负责 ASR、场景分割、关键帧、caption；fastsearch **消费其结构化产物**（带时间戳的 chunk）。这保持 fastsearch "检索 + 溯源"的边界。需与 docparse 对齐 chunk schema（时间戳、媒资指针字段）。

### 6.4 query 端跨模态怎么暴露？
- 以图搜图需 REST 接受图上传/URL；需定义 `modality_filter` 语义（"只搜图""图+文都搜"）。

## 7. 演进路线（通往 GA，可叠加，无"永久砍掉"）

> 与 [需求分析报告 §4.5](2026-06-24-需求分析报告.md) 的"完善产品而非 MVP"一致：P0/P1/P2 是**次序**不是取舍，终态全交付。

- **P0 — 转录/描述派生文本路线（零新嵌入依赖）**
  - 数据模型加时间区间 + 媒资指针 + `MediaMeta`；`text` `Option` 化；`ChunkKind` 加 Audio/Video。
  - PG DDL + 行映射 + 文本 schema 跟进；CDC sink 对转录/caption 走现有文本嵌入。
  - 媒资 ACL 网关 + `resolve_citation` 媒资感知（覆盖场景 A 展示、C 语音、D 转录检索）。
  - **产出**：语音按转录检索 + 音频深链；文档图按上下文召回 + 内联展示；视频按转录召回。**全程不需新模型依赖。**
- **P1 — 跨模态单向量（HTTP 后端，opt-in）**
  - `Embedder`/`HttpEmbedder` 多模态扩展，接 SigLIP-2 / JinaCLIP-v2 / jina-v4 之一；`vector`/`engine` 支持模态过滤 + 以图为 query。
  - **产出**：以文搜图 / 以图搜图（场景 B）；视频关键帧视觉召回（场景 D 增强）。
- **P2 — 视觉文档多向量 + 精定位（重，opt-in，待运行验证）**
  - 引擎侧 ColPali/ColQwen2 多向量 MaxSim 后端（不动 PG 真源）；视频时刻边界细化。
  - **产出**：版式/图表重文档的 OCR-free 高质量召回；视频精片段。

每阶段都要带：相关性评测样本（[eval](../../crates/fastsearch-eval/src/) 的 nDCG/recall 扩到多模态 golden 集）、ACL 不可绕过测试、确定性测试。

## 8. 风险与未决

- **ACL 新出口**：媒资字节/深链是新数据出口，是本期最大安全风险点；§5.7 的网关必须先于任何媒资展示落地，并有专门测试。
- **嵌入空间一致性**：跨模态检索质量高度依赖"文本/图真的在同一空间"；模型换代/维度变更需走索引重建(蓝绿)，靠 [eval](../../crates/fastsearch-eval/) 守回归。
- **对象存储依赖**与"轻量单二进制"叙事的张力（§6.1）——需产品定位上明确表态。
- **ColPali 存储/算力成本**：每页千级 patch 向量，需剪枝/量化（有现成研究），P2 前先做容量信封估算。
- **诚实记账**：P1/P2 落地前，文档与状态一律标"下一迭代/待运行验证"，禁止写"已完成"。
- **上游耦合**：转录/场景分割/关键帧的质量决定检索上限；需与 docparse 对齐 chunk schema 并明确职责边界。
