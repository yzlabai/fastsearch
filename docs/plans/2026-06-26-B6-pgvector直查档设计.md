> 按迭代循环步骤②：B6 是复杂项（新后端 + 触不变量 #5 + 嵌入归属设计问题），先定方案再施工。代码是真源，落地以代码为准并回写。

## 1. 是什么 / 为什么

第三种向量后端档：**不在引擎侧维护向量索引**，而是把 ANN 查询直接下推到 **pgvector**（PG 内
`ORDER BY embedding <=> $q LIMIT k`）。与暴力 `MemVectorIndex`、近似 `HnswVectorIndex`（引擎侧）
并列，经同一 `VectorBackend` trait 接入。

**何时用**：部署方不想在引擎进程里吃向量索引内存 / 已重度用 PG / 向量规模适中。代价：每查
一次 PG 往返延迟 + 依赖 pgvector 的 filter-aware 能力。

**可移植不变量 #1 仍守**：只用 **pgvector 扩展 + （HNSW 索引）**，不装任何
`shared_preload_libraries` 原生扩展。pgvector 0.8.2（本环境 fs-pg 实测）自带 **iterative scan**，
正是解决"ANN + WHERE 后过滤召回崩"的关键。

## 2. 设计决策：嵌入归谁算/写（核心问题）

引擎常规模式：`apply_upsert` 用 HTTP Embedder 嵌入正文 → 写**引擎侧**向量索引。pgvector 直查
要求向量在 **PG 的 `embedding` 列**。谁写？

**决策：写穿（write-through）**——直查模式下 `PgVectorBackend::upsert(gid, embedding, meta)` =
`UPDATE {table} SET embedding=$1 WHERE (collection,doc_id,chunk_id)=...`。即：
- 嵌入仍由**引擎**算（HTTP Embedder，不在 PG 内做模型推理——守"不在 PG 装原生扩展"）。
- 算出的向量**写回 PG 的 embedding 列**；检索时 pgvector 直接用它做 ANN。
- PG 成为向量的单一存储（embedding 是 chunk 的**派生列**，可重算——不违反 #2，#2 禁的是
  "只在引擎有、PG 没有的权威数据"，派生列在 PG 完全 OK）。

`delete`/`delete_doc` = `UPDATE ... SET embedding=NULL`（或随 chunk 行删除自然消失）。
引擎侧无 `vector.bin`，`persist` 跳过向量落盘，`open` 直连 PG。

## 3. 架构：第三个 VectorStore 变体

```
VectorBackend (trait)
  ├── MemVectorIndex    暴力精确（默认，确定）
  ├── HnswVectorIndex   引擎侧 HNSW（近似，A9）
  └── PgVectorBackend   pgvector 直查（ANN 在 PG，B6）—— 持 PG 连接/池
```

`VectorBackendKind` 加 `PgVector { conn, table }` 变体；`VectorStore` 加分支。engine 排序管线零改动
（trait 后面换实现）。**注意**：trait 当前是同步 `search`，而 PG 查询是 async → 见 §6 集成。

## 4. filter-aware：下推 SQL + iterative scan + 精确后过滤（守不变量 #5）

| 维度 | 保证 | 手段 |
|---|---|---|
| 精度/ACL | **精确、不越权** | ACL + 可翻译 filter 子句 → **精确 SQL WHERE**；返回行再用 `AclFilter::visible`/`Filter::eval` 复核（defense-in-depth） |
| 召回（选择性过滤） | **filter-aware**（pgvector 0.8 iterative scan） | `SET LOCAL hnsw.iterative_scan = relaxed_order`：HNSW 持续扫描直到够 k 行过 WHERE，避免后过滤召回崩 |
| 召回（ANN 近似） | 近似（明示） | over-fetch（取 `k×factor`）+ ef_search（`SET LOCAL hnsw.ef_search`）；全精度由 pgvector 距离保证 |

**filter→SQL 翻译**（`core::Filter` AST → 参数化 WHERE）：
- 可翻译子句（落具体列）：`kind/modality/doc_id/page/section_id/tenant/time_start_ms/time_end_ms`
  上的 `Eq/Ne/In/Gt/Gte/Lt/Lte/Exists` + `And/Or/Not`。**绝大多数真实过滤都可翻译**（字段即列）。
- 不可翻译子句 → 译为 `TRUE`（SUPERSET，不漏召回）+ over-fetch + **Rust 精确后过滤**
  （`Filter::eval`）。与 text/vector 后端同一 SUPERSET+精确后过滤范式（#5）。
- ACL：`tenant 匹配 AND acl && $tags::text[]`（数组重叠）下推为精确 SQL。

**SQL 骨架**（参数化，jsonb/vector 用 `::` 转换避 tokio-postgres 类型推断坑）：
```sql
SET LOCAL hnsw.iterative_scan = relaxed_order;
SET LOCAL hnsw.ef_search = <ef>;
SELECT collection, doc_id, chunk_id, kind, modality, page, section_id, tenant, acl,
       1 - (embedding <=> $1::vector) AS score
FROM {table}
WHERE embedding IS NOT NULL AND <acl_superset> AND <filter_superset>
ORDER BY embedding <=> $1::vector
LIMIT <k * over_fetch>;
```
返回行 → 建轻量 `FieldSource` → `Filter::eval`+`AclFilter::visible` 精确后过滤 → `Scored` → 截 k。

## 5. DDL：ANN 索引（可选、幂等）

直查需 embedding 上的 HNSW 索引（否则退化为全表 KNN 扫描）：
```sql
CREATE INDEX IF NOT EXISTS {table}_emb_hnsw ON {table}
  USING hnsw (embedding vector_cosine_ops);
```
加进 `sql::ddl`（仅当选直查档时建？或恒建——空表建索引成本低）。`<=>` 是 cosine 距离算子。

## 6. 集成的同步/异步张力

`VectorBackend::search` 当前**同步**；PG 查询**异步**。两条路：
1. **后端内阻塞跑 async**：`PgVectorBackend` 持一个 `tokio::runtime::Handle` 或用
   `tokio::task::block_in_place` + `Handle::block_on`。engine 的 search 在 server 里本就在 async
   上下文（`spawn_blocking` 包检索），故 `block_in_place` 可行。
2. **trait 加 async 变体**：改动大，暂不。

选 ①（局部、非侵入）：`PgVectorBackend::search` 内用持有的 `Handle::block_on(async { ... })`。
upsert 同理（写穿是 async）。CDC apply 路径已在 async，注意不要嵌套 runtime（用 block_in_place）。

## 7. 持久化 / 确定性

- 持久化：**无引擎侧向量落盘**（PG 是向量存储）。`Checkpoint.vector_backend="pgvector"`；
  `open` 据此连 PG。重建=重算嵌入写回 PG（`rebuild_from` 写穿）。
- 确定性：pgvector HNSW + iterative scan **近似且非确定**（同 A9 档）。默认暴力档仍确定。诚实标注。

## 8. 测试计划（需 PG，Docker 实跑）

1. **基础 KNN**：插入 N 行 + 写 embedding + 建 hnsw 索引，查 → 最近邻正确（对暴力 ground-truth recall）。
2. **ACL 下推**：team-a/team-b 混插，team-a 主体查 → 只见 team-a（精确、不越权）。
3. **filter-aware 召回**：选择性过滤（modality=image，image 占比低）→ iterative scan 仍召回到
   足够 image（对比关 iterative scan 的召回崩，证明其价值）。
4. **写穿**：upsert 后 PG embedding 列被更新；delete 后该行不再命中。
5. env-gated（`DATABASE_URL`），纳入 CI（CI 已有 wal_level=logical 的 pgvector）。

## 9. 施工顺序（每步收口三绿 + 回写）

1. ✅ `sql.rs`：filter→SQL 翻译器（`Filter`+`AclFilter` → 参数化 WHERE：可翻译子集精确下推
   /否定·未知列·类型不符·文本大小比较→TRUE 超集）+ `pgvector_search_sql`（SELECT 生成，`$1`=查询
   向量、filter/acl 从 `$2`）+ `ann_index_sql`（HNSW cosine 索引）。纯函数，3 单测，**不需 PG**。
   `SqlParam` 枚举承载类型化参数供 step2 bind。
2. ⏳ `PgVectorBackend`（持 PG client + runtime Handle）实现 `VectorBackend`：search（§4）+ 写穿
   upsert/delete + Rust 精确后过滤。**层级决策**：PgVectorBackend 需同时依赖 vector(trait) + pg
   (client) → 落 engine 或新 crate（vector/pg 是兄弟层，避免互依）。env-gated 集成测试（§8）。
3. 接 `VectorStore`/`VectorBackendKind`/engine `open`/`Checkpoint` + server env
   `FASTSEARCH_VECTOR_BACKEND=pgvector`（+ `DATABASE_URL`）。
4. 文档 + 诚实标注近似/非确定。

## 10. 风险与回退

- 同步/异步张力（§6）：`block_in_place` 不当会 panic（非 multi-thread runtime）→ 测试覆盖 +
  fallback 文档；最坏退为"直查档需 multi-thread runtime"。
- iterative scan 行为依赖 pgvector ≥0.8（本环境 0.8.2 ✓）→ 版本探测 + 文档标注最低版本。
- 默认档不变（暴力），直查全程 opt-in → 上线零风险。
