# fastsearch · 产品设计文档（技术架构 · 功能设计 · 开发计划）

> 状态：设计草案 v1｜日期：2026-06-24｜上游：[需求分析报告 v3](2026-06-24-需求分析报告.md)
>
> 本文把需求报告落成可实现的设计：**技术架构**（组件/数据模型/数据流/部署）、**功能设计**（每项能力怎么做）、**API 设计**、**开发计划**（阶段/任务/出口）。目标是**完善产品（1.0 GA）**，非 MVP（见需求报告 §4.5 十二维判准）。
>
> 一句话定位：**fastsearch 是一个外部单二进制混合检索引擎，以托管 Postgres(pgvector) 为真源，经逻辑复制 CDC 同步、自持派生 BM25 索引，提供安全/可观测/可零停机运维的生产级全文+向量+混合检索，并把 page+bbox 引用端到端带到答案层。** 超越 ParadeDB 的关键：能跑在任意托管 Postgres 上（ParadeDB/VectorChord/pg_textsearch 因需原生扩展都做不到）。

## 1. 设计目标与原则

**设计目标（对齐需求报告 §4.5）**：在任意托管 Postgres 上提供生产级混合检索；相关性有评测护栏；查询能力对齐 ES/ParadeDB，在融合/可调性/可移植/引用溯源上超过它们；安全（逐文档 ACL）、可观测、可零停机运维；四张脸 + SDK 完整集成。

**设计原则**：
1. **Postgres 是真源，引擎索引是派生且可重建**——降低持久化/复制/备份的自建成本；崩溃恢复=重放复制流或拉快照。
2. **托管 PG 可移植是硬约束**——PG 侧只用 pgvector + 逻辑复制，**绝不要求装原生扩展**。这是护城河，CI 须守。
3. **trait 边界隔离可替换件**——存储后端、向量后端(a/b)、嵌入后端、rerank 后端、分词器都是 trait。
4. **确定性 + 契约**——同稳定索引+同查询→同结果；查询/结果/配置有 JSON Schema，golden 防漂移。
5. **安全默认开**——认证默认要求，逐文档 ACL 在过滤期强制注入、客户端不可绕过。
6. **健壮不糊弄**——坏数据不中断整批、不 panic；CDC exactly-once 效果；并发写不损坏。

## 2. 技术架构

### 2.1 系统总览（控制面/数据面分离，Quickwit 式）

```
┌────────────────────────────────────────────────────────────────────────┐
│                         用户的托管 Postgres（真源）                       │
│   fastsearch_chunks 表：chunk + 元数据 + ACL + pgvector 向量列            │
│   逻辑复制 publication + replication slot（pgoutput）                     │
└───────────────┬──────────────────────────────────────────────────────────┘
                │ ① 写路径：docparse chunks → 直写/代理写
                │ ② CDC：逻辑复制流（insert/update/delete + LSN）
                ▼
┌────────────────────────────────────────────────────────────────────────┐
│  fastsearch 引擎（单二进制，可多副本，无状态读）                          │
│  ┌──────────────┐   ┌─────────────────────────────────────────────────┐ │
│  │ fastsearch-  │   │  派生索引（每集合一份，可蓝绿双版本）            │ │
│  │ sync (CDC)   │──▶│  · BM25 倒排（Tantivy/mmap，fast fields 过滤）   │ │
│  │ 快照+增量    │   │  · 向量 ANN：(a) pgvector 直查 或 (b) 引擎自建   │ │
│  │ LSN/幂等/自愈│   │  · ACL/元数据 fast fields + 引用 stored fields   │ │
│  └──────────────┘   └─────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────────────────┐│
│  │ 检索核心：解析→ACL强制过滤→keyword∥semantic 召回→融合→[merge]→[rerank]││
│  │           →组装命中(引用+高亮+分数)                                  ││
│  └─────────────────────────────────────────────────────────────────────┘│
│  fastsearch-server：REST(+OpenAPI) · MCP · 认证/RBAC · 可观测 · 限流     │
└───────────────┬──────────────────────────────────────────────────────────┘
                ▼  CLI · 库 · REST · MCP（四张脸契约一致）
        答案层(外部 LLM)：resolve_citation(id)→{page,bbox}→深链/高亮
```

**控制面/数据面**：Postgres 是真源（数据面 + 元数据/控制状态如 LSN 检查点）；fastsearch 引擎是**无状态服务层**（派生索引可丢可重建）。这让"在线多人"靠多只读副本 + LB 横向扩展。

### 2.3 数据模型

**Postgres 真源表（单表 multi-collection，建议）**：

```sql
CREATE TABLE fastsearch_chunks (
  collection    text        NOT NULL,           -- 集合名（多知识库隔离）
  doc_id        text        NOT NULL,           -- 来源文件标识
  chunk_id      bigint      NOT NULL,           -- docparse chunk 文档内序号
  kind          text        NOT NULL,           -- paragraph|table|image|heading|...
  text          text        NOT NULL,           -- 检索正文（可含 heading_path 前缀，F27）
  page          int         NOT NULL,
  bbox          jsonb       NOT NULL,           -- {x0,y0,x1,y1} PDF 用户空间
  heading_path  text[]      NOT NULL DEFAULT '{}',
  section_id    bigint      NOT NULL DEFAULT 0,
  char_len      int         NOT NULL,
  image_meta    jsonb,                          -- image chunk 的 caption/file/...
  -- 访问控制（F44）
  tenant        text,                           -- 租户隔离
  acl           text[]      NOT NULL DEFAULT '{public}',  -- 可见性标签/允许的 principal
  -- 向量（pgvector，真源；维度随 embedder）
  embedding     halfvec(384),                   -- halfvec 省一半、几乎不掉召回
  embed_model   text,                           -- 标记用哪个模型/版本，便于重嵌入
  -- 簿记
  updated_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (collection, doc_id, chunk_id)
);
-- doc_id 级替换：DELETE WHERE (collection,doc_id)=... 再批量插入新 chunk
-- REPLICA IDENTITY 用默认(PK)即可捕获删除键；逻辑复制 publication 覆盖本表
CREATE PUBLICATION fastsearch_pub FOR TABLE fastsearch_chunks;
```

> 设计要点：① 向量列在 PG（真源），无论 ANN 在 (a) 还是 (b) 都有据可依/可重嵌入；② ACL 是数组列，便于"标签 ∩ 调用者授权"判定；③ `doc_id` 级替换是天然的"重解析一个文件即更新它全部 chunk"语义，CDC 捕获为 delete+insert。

**引擎侧派生 Tantivy 索引 schema（每集合一份）**：

| 字段 | Tantivy 类型 | 作用 |
|---|---|---|
| `global_id`(=collection+doc_id+chunk_id) | u64/bytes, STORED+FAST | 主键，回指 PG / 去重 / 反解引用 |
| `text` | text, indexed(positions) | BM25 主字段，按集合配置的分词器 |
| `heading_path` | text, indexed, **boost>正文** | 标题面包屑入检索（廉价 Contextual） |
| `doc_id/kind/section_id/page` | FAST | 过滤/分面/折叠 |
| `tenant/acl` | FAST(多值) | **ACL 强制过滤** |
| `page/bbox/heading_path/doc_id/chunk_id` | STORED | 命中组装引用，免回 PG |

**引擎侧向量索引（子选择 b 时）**：`hnsw_rs` 图索引，节点 = `global_id`，向量从 PG 同步而来，可选 RaBitQ/int8 量化 + 全精度重排；过滤靠 filter-aware 遍历（持有 ACL/元数据位图）。

### 2.4 数据流

**① 写路径**（两种，可配）：
- 直写 PG：`docparse ... -f chunks` → 客户端/SDK 批量 `INSERT`（或 `COPY`）入 `fastsearch_chunks`（含 ACL）。引擎不在写路径，最简。
- 代理写：`fastsearch index` 收 chunk → 校验/补 ACL/可选生成嵌入 → 写 PG。统一入口，便于校验。
- 两者写完都**只动 PG**；索引更新由 CDC 异步驱动（F29）。

**② CDC 同步**（fastsearch-sync 状态机，F51–F53）：
```
[启动] → 有该集合的 LSN 检查点? 
  否 → [初始快照] 在快照 LSN 下 COPY 全表 → 建派生索引 → 记 snapshot_lsn
  是 → 从检查点 LSN
       ↓
[流式] 连接复制 slot，从 LSN 起接 pgoutput
   → 解码 INSERT/UPDATE/DELETE（按 PK）
   → 幂等应用到 Tantivy(+向量索引)：upsert/delete by global_id
   → 周期性持久化 applied_lsn（幂等点）
   → 心跳推进 slot，监控 lag；slot 失效→自动重快照
```
- **exactly-once 效果**：应用幂等（按 global_id upsert/delete）+ 持久化 applied_lsn，重启从该 LSN 续传，重复消息无副作用。
- **slot 韧性**：心跳 + 监控 `pg_replication_slots`，建议用户设 `max_slot_wal_keep_size` 防 WAL 撑爆；引擎落后告警。

**③ 查询路径**（检索核心，见 §3.4 排序管线）：解析 → ACL 注入 → keyword∥semantic 召回 → 融合 →[auto-merge]→[rerank]→ 组装命中。

### 2.5 部署架构

- **单机**：一个引擎进程 + 用户的 Postgres。最简，适合中小库/单团队。
- **在线多人（推荐拓扑）**：**1 个索引 leader**（跑 CDC、建索引、产出索引快照）+ **N 个无状态只读副本**（拉 leader 的快照、tail 同一复制流或从快照增量），前挂 LB。读随副本数近线性扩。PG 的 HA/复制/分区交给托管云。
  - 备选：每副本各持一个复制 slot 独立消费（简单但多占 WAL 保留）。leader+快照更省 PG 负载，推荐。
- **容器/K8s**：提供镜像 + Helm；副本拓扑可借 CloudNativePG 式编排。
- **托管 PG 适配**：RDS/Aurora 开 `rds.logical_replication`、Supabase/Neon 开逻辑复制；引擎在 VPC 内连 PG。**全程不需在 PG 装扩展**（N1b）。

## 3. 功能设计

### 3.1 索引与文档模型

- 集合(collection)为隔离单元，各有独立 schema/分词器/嵌入模型配置。
- 文档模型 = docparse chunk（§2.3）。`doc_id` 级原子替换 = 重解析一个文件更新其全部 chunk。
- 通用接入：非 docparse 的任意 PG 表也可映射（配主键/文本列/向量列/过滤列/ACL 列），把 fastsearch 当"任意 Postgres 的检索加速层"（F34）。

### 3.2 全文检索（fastsearch-text）

- **BM25**：Tantivy 打分，**暴露 `k1`/`b`** 与自定义 Similarity（超 ParadeDB）；字段权重（`heading_path` > `text`）。
- **Block-Max WAND** top-k 跳过（用带 BMW 的 Tantivy 路径）。
- **CJK/多语分词**：每字段可配 `jieba`/`lindera(chinese/japanese/korean)`/`icu`/`ngram`/`unicode`；用户词典；繁简；混排脚本自适应。质量排序 jieba>lindera≈icu>逐字。
- **查询能力（F35）**：term、phrase(+slop)、邻近(有序/无序)、布尔(must/should/must_not)、范围、正则/通配、term_set、fuzzy(Levenshtein+转置)、boost/const、more_like_this。
- **高亮（F37）**：返回片段 + 文本 offset + byte 位置（与 bbox 互补）。

### 3.3 向量检索（fastsearch-vector，两档都交付）

- **嵌入（F12）**：内置 embedder（Candle 主、ort 可选），默认 `multilingual-e5-small`（384 维，含中文）；查询/文档侧分别加 `query:`/`passage:` 前缀；模型 opt-in、按需下载；也接受外部向量。
- **(a) pgvector 直查档**：ANN 走 PG（HNSW/halfvec）；过滤用 0.8 iterative scan / partial index / 分区缓解后过滤召回坑；托管最省事。
- **(b) 引擎自建档**：`hnsw_rs` 图索引 + **RaBitQ/int8/二值量化** + 全精度重排；**filter-aware 预过滤**（真正修掉 pgvector 后过滤召回崩）。延迟/融合/过滤全可控。
- 运行时按集合/查询可切；trait 统一接口。

### 3.4 混合检索与排序管线（fastsearch-core，产品核心）

统一排序管线（每阶段可配/可关）：
```
查询 q + 过滤 F + 调用者身份 P
  │
  1. 解析与改写：分词、同义词、（可选）heading 前缀
  2. ACL 强制注入：F' = F ∧ acl(P)        ← 不可被客户端覆盖（§3.6）
  3. 并行召回（候选 N，宽，默认~150）：
       keyword: Tantivy BM25 + 过滤 F'  → 候选+分数
       semantic: embed(q) → ANN(a|b) + 过滤 F' → 候选+分数
  4. 融合（F14，一等内置，超三家手写 SQL）：
       · RRF（默认，rank_constant=60）
       · 分数归一化（min-max/z-score，distribution 仿射，semantic_ratio）
       · 加权凸组合（alpha）
  5. [可选] auto-merging：同 section 多数子块命中 → 上提 section 父块（F16）
  6. [可选] rerank：cross-encoder 对 top-K（默认 bge-reranker-v2-m3）
  7. 组装命中：引用(page+bbox+section+heading_path)+高亮+各路分数+融合分
  8. 分面(F36) / 深分页(search_after) / 折叠(F37) / 截断 top_k
```
- 宽召回 + 分数保真（F15）：返回足够候选、保留各路分数，便于外接/内置 rerank。

### 3.5 过滤、分面、溯源

- **过滤（F17）**：类型化可嵌套 `AND/OR/NOT`，操作符 `= != > >= < <= IN EXISTS`；维度 `doc_id/page(范围)/section_id/heading_path 前缀/kind/tenant`。引擎侧走 Tantivy fast fields（快、与检索一次完成）。
- **分面（F36）**：terms/histogram/range 桶 + avg/sum/min/max/count/cardinality，单遍随检索返回。
- **溯源（F18）**：每命中稳定返回 `citation_id + doc_id + page + bbox + heading_path + section_id`；`resolve_citation(id)` 反解到 `{page,bbox}`，与 docparse `locate` 闭环。

### 3.6 安全与访问控制（fastsearch-server，私有 RAG 硬门槛）

- **认证（F43）**：API Key/Token；REST 默认要求认证。
- **逐文档 ACL（F44）——核心设计**：
  - 每 chunk 带 `tenant` + `acl[]`（标签或 principal）。
  - 调用者身份 P → 解析出其 `tenant` 与授权标签集合 `tags(P)`。
  - **过滤期强制注入**：服务端把 `acl_filter = (tenant = P.tenant) AND (acl && tags(P) OR 'public' = ANY(acl))` 与用户过滤 **AND** 起来，注入在 ACL 检查点（管线第 2 步），**客户端无法覆盖/绕过**。
  - 引擎侧用 Tantivy 多值 fast field `acl` 求交；pgvector 档在 SQL WHERE 注入同等条件。
  - 验收：构造越权查询无法命中无权 chunk（§10-C8）。
- **审计（F45）**：检索/写入审计日志（谁/何时/查了什么/命中什么）；TLS + 静态加密选项；PII 字段策略（标记/脱敏/不索引）。
- **多租户（F46）**：集合级命名空间 + tenant 列双层隔离。

### 3.7 相关性质量体系（fastsearch-eval，MVP 最爱砍）

- **评测集**：golden 集 = query → 相关 chunk 标注（docparse 真实语料）；支持人工标注 + 合成 + 点击反馈三来源。
- **指标 + 门禁（F39）**：nDCG@k / recall@k / MRR；**进 CI**——改分词/融合/权重掉点则卡合并。
- **调优面板（F40）**：`k1/b`、字段权重、`semantic_ratio`/`rank_constant`/alpha、rerank 开关；配置版本化 + A/B。
- **词典治理（F41）**：同义词/停用词/用户词典增删 + 热更新，不全量重建。
- **可解释（F42）**：debug 模式返回打分明细（各路分、融合分、命中词、字段贡献）。

### 3.8 索引生命周期与运维（fastsearch-server）

- **零停机重建（F47）**：换分词器/嵌入模型/schema → 后台蓝绿建新版本索引 → 原子指针切换 → 丢旧版本；在线查询不中断。
- **快照/快恢复（F48）**：周期持久化 Tantivy 段到本地/对象存储；副本秒级拉起、崩溃免全量重放（重放兜底）。
- **格式版本化（F49）**：索引带版本号，升级在线迁移或自动重建。
- **可观测（F50/N14）**：Prometheus 指标（QPS、p50/p95/p99、召回、**CDC lag**、段数/索引大小、CPU/内存）、分布式追踪、结构化日志、慢查询日志、健康/就绪探针、优雅启停、限流 + admission control + 背压。

### 3.9 接口设计（四张脸，契约一致）

- **CLI**：`fastsearch collection|index|search|reindex|serve|mcp ...`。
- **库**：`fastsearch` crate 直接嵌入。
- **REST**：见 §4。
- **MCP**：工具 `search`/`index_document`/`resolve_citation`/`get_chunk`，供 agent 直连，与 docparse MCP 拼"解析→检索→引用"闭环。
- **SDK（F55）**：Python/TS 零依赖客户端 + LangChain/LlamaIndex 检索器适配。

## 4. API 设计（REST，OpenAPI + JSON Schema 契约）

**端点**：
```
POST   /v1/collections                 创建集合（schema/分词器/嵌入模型/向量后端 a|b）
GET    /v1/collections                 列出
DELETE /v1/collections/{c}             删除
POST   /v1/collections/{c}/documents   写 chunk（代理写路径，含 ACL）
POST   /v1/collections/{c}/search      检索（主入口）
POST   /v1/collections/{c}/reindex     蓝绿重建
GET    /v1/collections/{c}/chunks/{id} 取单 chunk / resolve_citation
GET    /healthz /readyz /metrics       运维
GET    /openapi.json /schema/{name}    契约
```

**检索请求**（`POST /v1/collections/{c}/search`）：
```json
{
```

## 5. 关键技术决策（汇总）

| 决策点 | 选择 | 理由 / 出处 |
|---|---|---|
| 总体架构 | 外部引擎 + 托管 PG(pgvector) 真源 + CDC | 需求报告 §6.8；唯一能跑遍托管 PG、绕过 ParadeDB allowlist 短板 |
| 真源存储 | 用户的 Postgres + pgvector(halfvec) | 白嫖托管/复制/备份；向量真源在 PG |
| 全文引擎 | Tantivy（BM25 + Block-Max WAND，暴露 k1/b） | Rust Lucene；beat pg_search 的 k1/b 不可调 |
| 分词 | lindera/jieba/icu 可选 + 用户词典 | CJK 质量；注意 lindera 中文字典 CC-BY-SA |
| 向量 ANN | 两档：(a) pgvector、(b) hnsw_rs+RaBitQ | 托管省事档 + 高性能/预过滤/量化档，运行时可切 |
| 嵌入 | Candle 主 + ort 可选；e5-small 默认 | 本地离线；含中文 |
| 融合 | RRF 默认 + 归一化 + 加权，一等内置 | beat 三家手写 SQL |
| rerank | cross-encoder（bge-reranker-v2-m3）opt-in | +nDCG，宽召回后重排 |
| CDC | pgoutput 逻辑复制 + 幂等 + LSN 检查点 | 托管 PG 唯一可用插件；Sequin/Debezium 模式参照 |
| 索引生命周期 | 蓝绿重建 + 快照 + 格式版本 | 零停机；副本快起 |
| 安全 | API Key/RBAC + 逐文档 ACL 强制注入 | 私有 RAG 硬门槛 |

## 6. 开发计划（通往 1.0 GA，对齐需求报告 §8 P0–P5）

> 原则：每阶段交付**生产级切片**（带测试 + 可观测 + 文档），不牺牲健壮性换速度。"必备"项 1.0 GA 不可缺；阶段是推进次序。

### P0 · 奠基与去风险（地基）
**目标**：验证关键不确定性，立骨架。
- 工作流：建 Cargo workspace（§2.2 各 crate 空壳）+ CI（fmt/clippy/test 零 warning）+ 评测框架骨架。
- **去风险 spike（最关键）**：`PG → pgoutput → Tantivy 索引 → 检索` + `pgvector ANN 查询` 端到端各跑通；验证向量两档可切。
- 定基础契约：PG schema（含 ACL 列）、索引格式版本、集合配置 schema、查询/结果 JSON Schema 草案。
- **出口**：CDC 可行性 + 向量两档可切 已验证；端到端一条命令能建库能查。
- 风险/对策：CDC 是最大不确定性 → 此阶段必须打穿，否则架构不成立。

### P1 · PG 真源 + 生产级全文 + 健壮 CDC
**目标**：全文检索可上线，CDC 可靠，相关性有护栏。
- `fastsearch-pg`：写路径（直写 + 代理写）、schema、ACL 列。
- `fastsearch-sync`：**初始快照 → 增量切换、LSN 检查点、幂等 upsert/delete、slot 自愈、lag 监控**（F51–F53，非批量糊弄）。
- `fastsearch-text`：BM25(+k1/b)、CJK 分词、完整查询能力（F35）、高亮（F37 文本部分）。
- `fastsearch-core`：查询 AST、过滤求值（fast fields）、引用组装（F18）。
- `fastsearch-eval`：golden 集机制 + nDCG/recall + **CI 回归门禁**（F39 起步）。
- **出口**：全文检索生产可用；CDC 重启可续传/重快照一致；相关性有回归门禁；引用闭环。

### P2 · 向量 + 一等混合融合
**目标**：三检索齐活，混合质量 ≥ 单路，超 pgvector 后过滤坑。
- `fastsearch-embed`：Candle + e5-small，按需下载；外部向量通道；多 embedder。
- `fastsearch-vector`：**(a) pgvector 直查 + (b) 引擎自建(hnsw_rs+RaBitQ) 两档**（F10–F13），filter-aware 预过滤。
- `fastsearch-core`：融合 RRF/归一化/加权（F14，一等内置）+ `fastsearch-rerank`（F15）。
- **出口**：混合在 golden 集 nDCG ≥ 单路最优；向量两档可切；预过滤召回不崩。

### P3 · 文档 RAG 深配 + 质量
**目标**：RAG 流水线完整，相关性可调可解释。
- auto-merging（F16）、section docstore（F5）、Contextual 前缀（F27）。
- 分面/聚合（F36）、深分页 search_after/折叠（F37）。
- 调优面板（F40）、词典治理（F41）、打分可解释（F42）。
- docparse 端到端流水线（F25）、OKF/outline 感知（F26）。
- **出口**：RAG 闭环完整；相关性可调、掉点可定位。

### P4 · 安全 + 运维（生产硬门槛）
**目标**：可安全地多人在线、可运维、可观测。
- `fastsearch-server`：认证/RBAC（F43）、**逐文档 ACL 强制注入**（F44）、审计（F45）、多租户（F46）。
- 索引生命周期：蓝绿重建（F47）、快照/快恢复（F48）、格式版本（F49）。
- 可观测全套（F50）：指标/追踪/慢查询/健康探针/优雅启停/限流；无状态副本（F32）+ leader 拓扑。
- SLO/容量信封/安全基线（N12–N15）。
- **出口**：ACL 不可绕过；可观测/限流就绪；副本滚动升级不掉服务；SLO 实测达标。

### P5 · 集成 + 打磨 → 1.0 GA
**目标**：完整集成，实证超越 ParadeDB，§4.5 十二维全达标。
- 四张脸契约一致 + OpenAPI（F54）；Python/TS SDK + LangChain/LlamaIndex 适配（F55）；MCP + Skill（F56）。
- 查询辅助：autocomplete/did-you-mean（F38）。
- **基准 vs ParadeDB/ES/Meilisearch**（公开方法）：可移植/融合/k1·b 可调/预过滤/引用溯源的优势实证。
- 容量规划 + 运维文档；跨平台二进制 + 容器/Helm。
- **出口**：§10 全部验收通过 → **发布 1.0 GA**。

### 阶段依赖与并行
- 强依赖链：P0(spike) → P1(CDC+全文) → P2(向量+融合) → P3(质量)。
- 可并行：P4 安全/运维的部分（认证、可观测、生命周期）可与 P2/P3 并行起步；`fastsearch-eval` 从 P1 起持续演进；SDK（P5）可在 REST 稳定后提前。
- 贯穿全程：评测护栏（F39）、可观测、文档、CI——不是某阶段的事。

## 7. 风险与对策（设计层，补充需求报告 §9）

| 风险 | 设计对策 |
|---|---|
| CDC 同步层是最大未知（exactly-once/快照切换/slot 自愈） | P0 先 spike 打穿；幂等 upsert + LSN 检查点；参照 Sequin/Debezium |
| 数据副本放大（PG + Tantivy + 子选择 b 向量索引 = 3 份） | 子选择 (a) 只 2 份；(b) 用量化压缩；按集合选档 |
| ACL 绕过（私有 RAG 致命） | 强制注入在管线固定点、客户端不可达；越权用例进 CI |
| 托管 PG 可移植被破坏（误引原生扩展依赖） | CI 守"PG 侧只用 pgvector+逻辑复制"；依赖评审 |
| 相关性长期工程 | 评测护栏从 P1 起；RRF 稳健兜底；可解释打分 |
| 蓝绿重建/快照的存储与切换正确性 | 原子指针切换 + 版本号；切换前后 golden 比对 |
