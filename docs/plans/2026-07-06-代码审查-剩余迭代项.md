## A. CDC 崩溃安全（需 Change / IndexSink 手术）

> 共性：都需给 `Change` 枚举加变体或给 `IndexSink` trait 加方法 + 引擎实现，且落在崩溃安全的 CDC 核心路径——不宜仓促，值得专门一轮 + Docker e2e。R4 已把**确定性毒丸死信**（decode/map 失败跳过）补上（[R4.1b](../devlog/2026-07-05-代码审查修复-R4-CDC-PG.md)），以下是剩余的。

### M14 — 改主键 UPDATE 留幽灵行
- **落点**：[sync/replication.rs:248-250](../../crates/fastsearch-sync/src/replication.rs#L248)（`PgMessage::Update { new: tuple, .. }` 只取新元组）。
- **问题**：外部 `UPDATE ... SET chunk_id=99`（或改 doc_id/collection）时 pgoutput 带 `key`/`old` 旧键，代码丢弃 → 旧 `GlobalId` 行永不删 → 检索命中已不存在的 chunk。
- **为何延后**：① **极罕见**——本系统 chunk 一律经 `upsert_doc`（DELETE+INSERT）替换，从不 PK-UPDATE；仅外部直接改 PG 主键才触发。② 需一条 Update 同时发 `Delete(old)`+`Upsert(new)`，与 H3 的 `Refetch` 路径交织，要重构 `map` 的多变更发射（目前 `map` 返回单个 `Mapped`）。
- **建议做法**：`map` 的 Update 分支取 `key.or(old)` 算旧 gid，若 ≠ 新 gid 则先产 `Delete(old_gid)` 再产 upsert；`decode_batch_lenient` / `fetch_changes` 已能承接多变更（改成 `Vec<Mapped>` 或前置一个 delete）。带"改主键"的 Docker e2e。

### M13 — TRUNCATE 事件被吞
- **落点**：[sync/replication.rs:279](../../crates/fastsearch-sync/src/replication.rs#L279)（`_ => Ok(None)` 吞掉 `Truncate`）。
- **问题**：运维对 `fastsearch_chunks` 执行 TRUNCATE 后，PG 已空而引擎索引仍返旧结果，永久发散直到全量重建。
- **为何延后**：需新增 `IndexSink::apply_clear`（清整表派生索引）+ `Change::Truncate` 变体 + 引擎实现（text/vector clear，已有 `clear()` 可复用）。TRUNCATE 是罕见运维动作。
- **建议做法**：`map` 把 `Truncate` → `Change::Truncate`；`IndexSink` 加 `apply_truncate`（引擎 `text.clear()+vector.clear()+commit`）。

### M10 — IndexSink 无原子性（半状态窗口）
- **落点**：engine `apply_upsert`（[engine/lib.rs](../../crates/fastsearch-engine/src/lib.rs) 的 `IndexSink` 实现）。
- **问题**：`text.upsert` 成功后嵌入（HTTP）或 `set_embedding`（PG）失败即返回 Err，text writer 留未提交半状态；`commit` 是引擎全局的，其他写路径的 `engine.commit()` 会把这半状态顺带提交 → "新文本可检索、向量还是旧版"的不同步窗口。
- **为何延后**：需给 apply_upsert 做事务式回滚/暂存（嵌入失败则撤销 text 写），较大重构；且与 M11（嵌入改批/并发）耦合，宜一起做。
- **建议做法**：apply_upsert 内先算好向量（嵌入）再一起写 text+vector，任一失败不写 text；或引入 per-chunk 的"待提交"缓冲，失败整体丢弃。

### M15 — LSN 水位契约与 pgoutput 语义不符（文档/API 澄清）
- **落点**：[sync/lib.rs](../../crates/fastsearch-sync/src/lib.rs) 的 `Applier` 水位续传契约。
- **问题**：文档承诺"重启从持久化 LSN 续传、`lsn <= applied_lsn` 跳过=exactly-once"，但事务首个数据行 LSN=Begin，按文档用法会把首事务第一条误判已应用而跳过。生产已绕开（engine `consume_once` 每拍 `Applier::new(Lsn(0))`），水位逻辑在生产路径实为死代码。
- **为何延后**：不是 bug（生产正确），是公开 API 文档会误导库形态使用者。低优先。
- **建议做法**：改 `Applier` 文档说明真实语义，或删掉生产已绕开的水位跳过逻辑（YAGNI）。

## B. 性能

### M11 — CDC 逐条同步嵌入阻塞
- **落点**：engine `apply_upsert` / `bootstrap_snapshot`（独占 `&mut self` 期间 `embed_multi(slice::from_ref(...))` 每 chunk 一次网络往返）。
- **问题**：server 用单 `Mutex<Engine>`，CDC 一批 N chunk = N 次串行 HTTP（每次最长 30s），期间所有搜索排队；bootstrap 灌大库逐条嵌入线性放大。
- **为何延后**：perf，需并发/批处理改造（与 M10 原子性耦合）+ bench 佐证收益。M12 已把 `HttpEmbedder` 批量切分/重试补上，减轻单请求 413/超时；批处理调用侧是下一步。
- **建议做法**：apply 一批时收集所有待嵌 chunk 文本 → 一次 `embed_multi`（M12 已支持批量切分）→ 分发；或嵌入移出引擎锁（先算向量再拿锁写）。bench `bench_capacity` 对拍。

## D. 低危鲁棒性 / 清理

### fuse / normalize 遇 NaN 分数
- **落点**：[core/fusion.rs:101-102,140-141](../../crates/fastsearch-core/src/fusion.rs#L101)（`partial_cmp().unwrap_or(Equal)`）。
- **问题**：NaN 分数（来自异常嵌入向量的点积）使比较器非传递，`sort_by` 结果不稳，破坏确定性 #4。
- **为何延后**：需谨慎不破坏正常的 GlobalId tie-break；NaN 向量仅来自异常嵌入服务，**R5 已在 PG 写侧加 `ensure_finite` 拦一道**（[pg/lib.rs](../../crates/fastsearch-pg/src/lib.rs)），引擎侧暴力后端未拦。
- **建议做法**：融合入口把非有限分数 sanitize 成固定哨兵（如 `f64::MIN`），或 embed `l2_normalize` 对非有限 norm 报错而非留 NaN 向量。

### ensure_slot TOCTOU
- **落点**：[sync/replication.rs:50-51](../../crates/fastsearch-sync/src/replication.rs#L50)（`WHERE NOT EXISTS` 判断与建 slot 非原子）。
- **问题**：多副本并发首启，两连接都判"无 slot"，后者 `pg_create_logical_replication_slot` 报 already-exists → server boot 失败。与 `ensure_schema` 精心的 advisory-lock 防护不对称。
- **为何延后**：仅多副本**并发首启**窗口触发；低频。
- **建议做法**：用 `pg_advisory_xact_lock`（仿 `ensure_schema`）包裹建 slot，或捕获 already-exists 错误码视为成功。

### media_bytes / query_image 走 JSON 数字数组
- **落点**：[core/model.rs](../../crates/fastsearch-core/src/model.rs) `Chunk.media_bytes` / `query.rs` `SearchRequest.query_image`（`Vec<u8>` → JSON `[137,80,...]`）。
- **问题**：一张 100KB 裁图序列化成 ~350KB+ 数字数组，解析也慢；标准做法 base64（serde with）。
- **为何延后**：改动跨线缆兼容（server 测试以数字数组喂 media_bytes），需两端同步 + 兼容旧格式。
- **建议做法**：`#[serde(with="base64_bytes")]`，server/SDK 同步；或保留数字数组兼容、新增 base64 字段。

### 引擎侧 5xx 错误串脱敏（R3.5 未尽）
- **落点**：server 多处 `(INTERNAL_SERVER_ERROR, e.to_string())`（`EngineError`）。
- **状态**：R3.5 已脱敏**明确泄露 SQL/schema 的 PG 出口**（`internal_err`）；引擎/嵌入错误是 typed `EngineError`、敏感度低，未统一。
- **建议做法**：把剩余 `e.to_string()` 也过 `internal_err`（日志记详情、客户端回通用文案）。

### 其他低危（review §四）
- **API Key 走 HashMap 查找非常量时间**（[server/lib.rs](../../crates/fastsearch-server/src/lib.rs) `keys.get`）——被 std SipHash 随机化大幅缓解，仅理论侧信道。
- **`Fusion::Weighted` 与 `Normalized` 数学等价**且 Weighted 零使用（[core/fusion.rs:33](../../crates/fastsearch-core/src/fusion.rs#L33)）——删其一或使 Weighted 真"不归一化加权"以区分（cosmetic）。
- **presigned 字节 URL 默认 TTL 300s、无 principal 绑定**——presigned 设计取舍；可考虑更短 TTL / 一次性 / 文档明确。
- **`/metrics` 未认证监听**——仅聚合计数，建议网络层限制。
