# MM8b 图像检索模型选型调研

> 状态：**调研完成 · 待选型决策**｜日期：2026-07-01｜上游：[多模态功能设计与开发计划](2026-06-25-多模态功能设计与开发计划.md)、[下一步改进路线图 §2.4](2026-06-30-下一步改进路线图.md)、[spec 16-embed](../specs/16-embed.md)。
>
> **一句话**：给 fastsearch 的「真图像检索」（MM8b，把线上"图像检索=搜图注文字"升级成真按图像内容检索）选一个**图文同空间嵌入模型**。结论：**主选 `jina-embeddings-v4`（经 vLLM 走 OpenAI 兼容 `/v1/embeddings`）**；关键约束修正——**Ollama 无法嵌图像，MM8b 必须走 OpenAI 兼容路径，不是 Ollama**。

---

## 1. 背景与硬约束

MM8b 是路线图第 ② 组"最大生产缺口"：当前图像检索实为 caption-as-text（搜图注文字），非真"看图"。要真看图需接一个**图文同空间**（cross-modal）嵌入模型。选型受 fastsearch 架构硬约束：

1. **HTTP 服务化**：Rust 侧 `HttpEmbedder` 经 Ollama 原生 `/api/embed` 或 OpenAI 兼容 `/v1/embeddings` 调用；**不进程内推理**（不引 Candle/ort）。
2. **真跨模态共享空间**：文查询召回图（以文搜图）+ 图查询召回图（以图搜图），文图**同维同空间**。
3. **图像 base64 输入**。
4. **多语言/中文**优先（语料中文为主）。
5. **开源可自托管**优先（vLLM/TEI 等），也评估托管 API（Voyage/Cohere/Jina）。

## 2. 方法与可信度（诚实记账）

- 手段：deep-research harness（5 角度 fan-out → 17 一手来源 → 79 声明抽取 → 对抗验证 → 综合）。
- **局限**：验证阶段遭 API 限流，14 个验证投票失败 → 自动综合偏薄。故对**两个决定性事实**（jina-v4 规格、vLLM 服务路径）另行**直接复核一手来源**（见 §7 来源 [1][7]）。
- 因此下文：模型规格/服务路径为**已复核**；单项基准数字（如 Flickr30K）为**单源高置信**；被限流未充分验证的按"待复核"标注，不冒充定论。

## 3. 候选对比

| 模型 | 维度 | 真跨模态文↔图 | 中文 | 许可 | 自托管 HTTP | 备注 |
|---|---|---|---|---|---|---|
| **jina-embeddings-v4** ⭐ | 2048（Matryoshka 可裁 128/256/512/1024） | ✅ 单向量 + 多向量（late-interaction） | ✅ 30+ 语言 | ⚠️ **Qwen Research License**（非 Apache，商用需核条款） | ✅ vLLM `/v1/embeddings` | 4B（基于 Qwen2.5-VL-3B），需 GPU；Flickr30K R@5≈91.4（单源） |
| **SigLIP 2**（Google） | 768/1152 档 | ✅（CLIP 式，短文本→图强） | ⚠️ 多语言变体有、中文一般 | ✅ 宽松（Apache 系） | ✅ vLLM 确认 | 轻、稳；长查询/复杂语义弱于 jina-v4 |
| **Qwen3-VL-Embedding-2B** | Qwen 系 | ✅ | ✅ 中文原生 | ⚠️ Qwen 系 | ✅ vLLM 确认 | 比 jina-v4 轻约一半，中文性价比高 |
| **JinaCLIP v2** | 1024（可裁 64） | ✅（CLIP 式） | ✅ 多语言 | ⚠️ CC-BY-NC（非商用） | ✅ | 0.9B 轻；CLIP 式 text 编码器，长文本弱 |
| **Nomic Embed Vision** | 768（与 nomic-text 同空间） | ✅ 但英文为主 | ❌ **中文弱** | ✅ Apache | ✅ | **陷阱**：与现用 nomic-text 同空间很诱人，但中文语料会塌 |
| **Cohere Embed v4** | 可变 | ✅ | ✅ | 托管 API | ❌ 仅 API | 无 GPU 时稳妥托管选项 |
| **Voyage multimodal-3** | 1024 | ✅ | ✅ | 托管 API | ❌ 仅 API | "比次优高 19.63%"营销说法**已被证伪**（0-3 投票），按普通强模型看待 |
| **ColPali / ColQwen** | 多向量 | 视觉**文档**（PDF 页截图）检索 | ✅ | 开源 | 需专门 MaxSim 后端 | **非同一赛道**——属路线图 MM11/③，不是自然图检索 |

## 4. 关键发现：服务路径（决定架构）

- ✅ **vLLM 可经 OpenAI 兼容 `/v1/embeddings` 服务图文嵌入模型**（CLIP / SigLIP / Qwen3-VL-Embedding-2B / DSE-Qwen2-VL / VLM2Vec），图像以 `image_url` 对象（支持 `data:image/jpeg;base64,...`）发送，响应 `data[].embedding` 返回向量——**正是 fastsearch `HttpEmbedder` 的 OpenAI 协议**（来源 [1]）。
- ❌ **Ollama 无法嵌图像**：`/api/embed` 只接文本；官方 issue #7677 明确 closed "not planned"（来源 [4]）。
- **推论**：MM8b 的服务目标是 **OpenAI 兼容端点（vLLM/TEI），不是 Ollama**。spec 16 里"经 Ollama 接图像"的设想需**修正**。

## 5. 建议

| 场景 | 选型 | 理由 |
|---|---|---|
| **质量优先 + 中文 + 文/图/文档通吃** | **jina-embeddings-v4**（vLLM） | 唯一同时满足真跨模态 + 强多语言 + 维度可裁 + 单/多向量；先核 Qwen Research License 商用条款 |
| **许可最干净 + 轻量** | **SigLIP 2** | Apache 系，short text→image 足够；无 GPU 压力小 |
| **中文最重 + 省显存** | **Qwen3-VL-Embedding-2B** | 中文原生，2B 比 jina-v4 轻一半 |
| **无 GPU** | **Cohere Embed v4**（托管） | 免自托管；代价=外部依赖 + 成本，偏离自托管哲学 |

**主选 = jina-embeddings-v4**；**备选 = SigLIP 2**（许可/轻量）或 **Qwen3-VL-Embedding-2B**（中文/显存）。

## 6. 陷阱清单

- ❌ **Ollama 嵌图像**：不支持（#7677），别按原 spec 走 Ollama 图像路。
- ❌ **原版 OpenAI CLIP / CLIP ViT**：77-token 英文 text 编码器，真·文→图（尤其中文/长查询）很弱。
- ❌ **Nomic Embed Vision 用于中文**：英文为主，中文会塌（尽管与现用 nomic-text 同空间很诱人）。
- ⚠️ **许可**：jina-v4=Qwen Research License、JinaCLIP v2=CC-BY-NC——**商用前核条款**。这是**模型权重许可**（作用于"如何用模型"），不污染 fastsearch 的 Apache 二进制，但仍是合规项；SigLIP2 系最干净。参见 [许可治理](../governance/2026-06-26-词典与第三方许可审.md)。
- ⚠️ **Voyage 营销数字**：被证伪，别当选型依据。

## 7. 对 fastsearch 的落地路径（MM8b 实现清单）

1. **嵌入服务**：起 vLLM，load `jina-embeddings-v4`（或 SigLIP2），暴露 OpenAI 兼容 `/v1/embeddings`。
2. **fastsearch 侧**（MM8b 本体）：打通 `HttpEmbedder` 当前 gated 的 `embed_multi(Image)` 路径——请求体带图 base64，按 OpenAI `image_url` data-URI 格式；`caps.image=true`；**断言文图同维同空间**（`cross_modal`）。目标协议 **OpenAI 兼容，非 Ollama**。
3. **维度**：Matryoshka 裁到 1024/512 进 pgvector，平衡召回/存储。
4. **验证**：本机起 vLLM + SigLIP2/jina-v4 → index 图 + 以文搜图 e2e，把 2.4 从 `gated` 解锁为可验证。文本通道的真机验证方法可参 [Ollama 语义 e2e devlog](../devlog/2026-06-30-真机验证-pg并发与ollama语义.md)。
