## 一、总体结论

工程底子扎实，**架构不变量在正常输入下守得很好**：ACL 纵深防御全链无泄露、CDC 崩溃安全次序正确、SUPERSET 下推 + Rust 权威后过滤的双层一致性严谨、移植性约束（只用 pgvector + 逻辑复制，实测无原生扩展依赖）成立。收口三件套现状全绿：

| 检查 | 结果 |
|---|---|
| `cargo fmt --all --check` | ✅ 通过 |
| `cargo clippy --workspace --all-targets -- -D warnings` | ✅ 零警告 |
| `cargo test --workspace` | ✅ 252 passed / 0 failed（2 个 env-gated 跳过） |

**但测试全绿掩盖了一批"边界输入下的正确性破洞"**——它们大多不在测试覆盖内，且好几处是"功能表面存在、实际反向劣化"。本轮共 40+ 处发现，其中 **7 高危 / 20 中 / 十余低**。最需要立刻处理的三个模式：

1. **请求 rerank 反而摧毁排序**（两条独立链路），
2. **浮点数过滤值静默漏召回**（预过滤不是超集，从两端交叉确认），
3. **开启 CDC+嵌入后，摄取一个大 chunk 的正常主流程会把同步永久卡死**（真容器实测）。

## 二、高危发现（7）

### H1. 请求 rerank 反而摧毁排序 —— 两条链路，中文/向量场景必现

**根因 A（任何向量召回）**：[engine/lib.rs:1562-1637](crates/fastsearch-engine/src/lib.rs#L1562)。`text_map` 只从 `kw_hits` 填充，向量独有命中在 rerank 时取 `text_map.get(&h.id).cloned().unwrap_or_default()` → 空串。
`mode=Vector` + rerank 时全部候选文本为空 → LexicalOverlap 全 0 → 排序退化为 `GlobalId` 升序，**向量相似度排名被完全丢弃**；`mode=Hybrid` 时向量独有命中（语义相关但词面不重叠，恰是向量路的价值）被系统性压到所有 keyword 命中之下。引擎已有 `text.stored_text(gid)`（more_like_this 在用）却未在此补取。

**根因 B（中文）**：[rerank/lib.rs:20-25](crates/fastsearch-rerank/src/lib.rs#L20)。`tokenize` 按"非字母数字"切分，而 CJK 字符 `is_alphanumeric()` 为 true → 整句中文成单个 token。查询"毛利率是多少"对候选"公司2023年毛利率为38%"Jaccard=0，所有中文候选同为 0 分 → 又退化到 gid 升序。中文是项目一等场景，该 fallback 生效即"反向破坏"。

**影响**：默认 `RerankSpec` 一旦启用，在中文或向量主路径上排序质量不升反降。
**建议**：A → rerank 前对缺失文本的命中补 `stored_text`；B → 退化为字符 bigram 或接入已有的 jieba。补 rerank×Vector/Hybrid、rerank×CJK 的回归测试。

### H2. 浮点过滤值不是超集：`Lt`/`Ne`/`Eq` 带小数时静默漏召回

从 core 与 text 两端交叉确认，已在临时工程复现。
- **翻译端**：[text/query_build.rs:33-39](crates/fastsearch-text/src/query_build.rs#L33)（`field_as_u64` 对 `Float` 做 `as u64` 截断）。`Lt("page", 5.5)` 截成 `Excluded(5)` → 只召回 page≤4，**page=5 的行永久漏掉**（eval 认为 5.0<5.5 应命中）。实测返回 `[]`。
- **求值端**：[core/filter.rs:76-77](crates/fastsearch-core/src/filter.rs#L76)，`Eq`/`Ne` 用派生 `PartialEq` **变体严格**比较（`Int(6) != Float(6.0)`），与 `Gt/Gte/Lt/Lte` 经 `as_f64` 的数值互比语义不一致。REST 客户端发 `{"eq":["page",10.0]}`（serde 把任何带小数点的数解析成 `Float`，Python `json.dumps(10.0)` 即产生）→ 命中恒 0，而同行 `Gte(10.0) && Lte(10.0)` 为 true，即 `Eq ≠ (Gte∧Lte)`。实测 `Ne(page,6.0)` 返回 `[1]`，eval 语义应返回两条。

**影响**：分页/页码/时间等数值过滤经 JSON 传入即可能悄悄丢结果，且被后过滤"正确地"错杀而不报错。
**建议**：`exact_eq`/`exact_range` 对 `Float` 一律返回 `None`（退 AllQuery 交后过滤），或 `Lt` 用 `Excluded(f.ceil())`；core 的 `Eq/Ne` 对数值走 `as_f64` 互比。补 Float 值的翻译一致性测试。

### H3. CDC 毒丸 + 写穿事件断言为假 —— 开启 CDC+嵌入摄取大 chunk 会自我卡死

真 pgvector 容器实测（列清单与本项目完全一致）。
- **毒丸**：[sync/replication.rs:190-213](crates/fastsearch-sync/src/replication.rs#L190)。TOAST 大字段（text 原文 >8KB）在"未改 text 的 UPDATE"复制事件里以 `UnchangedToast('u')` 出现，被 `cols()` 折叠成 None → `row_to_chunk` Err → 整批失败 → slot 永不 advance → 每拍重读同一毒丸，**CDC 永久停摆，无死信/跳过机制**。可空列遇 'u' 更隐蔽地 `.flatten()→None` 静默丢字段，派生索引与真源发散。
- **架构断言为假**：[pg/sql.rs:59-68](crates/fastsearch-pg/src/sql.rs#L59)。实测"列清单 publication → 写穿 UPDATE 不产生逻辑复制事件"为假：仅 UPDATE `embedding`（不在列清单）仍产生完整 Begin/Relation/**Update**/Commit（PG 列清单只过滤"列"、不抑制"事件"）。于是 `set_embedding` 的写穿本身就会为大 text 行产出毒丸载体——**摄取一个大 chunk 的正常主流程即触发 H3**。防环只剩 `IS DISTINCT FROM` 幂等守卫一道（该守卫本身正确，会在第二轮收敛）。

**建议**：`UnchangedToast` 分支应保留旧值或跳过重嵌（而非报错整批）；引入毒丸跳过/死信；回写文档去掉"写穿不产生事件"这一失效的结构性断言。

### H4. `candidates`/`top_k` 无上界 → 单请求 OOM abort 打崩全服务

[core/query.rs:123-131](crates/fastsearch-core/src/query.rs#L123) 的 `validate` 只校验 `top_k>0` 与 `candidates>=top_k`，**无上界**；`candidates` 直达 [text/lib.rs:382](crates/fastsearch-text/src/lib.rs#L382) 的 `Vec::with_capacity(k)`。任一合法低权 key `POST /v1/search {"top_k":1,"candidates":100000000000}` → 巨额分配触发 `handle_alloc_error` → **进程 abort（不可 unwind）**，多租户服务整体宕机。`/v1/similar` 同理；[text/lib.rs:349](crates/fastsearch-text/src/lib.rs#L349) 的 `k*8` 还会整数溢出。
**建议**：`validate` 对 `top_k`/`candidates` 设合理上界（如 ≤10000）并在 server 侧收口，补回归测试。

### H5. HNSW 墓碑聚簇 + 无过滤路径可返回 0 条

[vector/hnsw.rs:391,423](crates/fastsearch-vector/src/hnsw.rs#L391)。`needs_escalation = filter.is_some() || acl.is_some()`；无 filter/acl 时首轮即 break，被墓碑（`entries[id]=None`）占满的 `k×over_fetch` 候选窗口**不触发翻倍重搜**。自动压实阈值是"全局墓碑过半"，挡不住局部聚簇。实测：2010 活条目 + 790 聚簇墓碑（删一整个相似 doc 后），`k=10` 无过滤检索返回 **0 条**。
**建议**：`needs_escalation` 加 `|| self.dead_count() > 0`，或无过滤路径也下推"活条目"谓词。补墓碑聚簇回归测试。

### H6. Python `FastsearchRetriever` 与真实客户端组合必崩

[clients/python/.../integrations.py:126](clients/python/fastsearch_client/integrations.py#L126) 把 `highlight=` 透传给 `FastsearchClient.search()`，而后者签名（[__init__.py:74-83](clients/python/fastsearch_client/__init__.py#L74)）不收该参数。已实测 `TypeError: unexpected keyword argument 'highlight'`——每次 `retriever.invoke()` 用真实客户端都崩。测试之所以绿，是 `_FakeClient.search(**kw)` 吞任意 kwargs 遮住了 bug。即便修签名，真实 `search` 也不把 highlight 放进请求体 → `page_content` 恒空，docstring 承诺不成立。
**建议**：一行修签名 + body 透传 highlight，并**补 SDK 的 CI job**（当前 clients/ 完全无 CI，正是此类崩溃的盲区来源）。

### H7. README.zh-CN 五分钟上手命令已失效

[README.zh-CN.md:26-30](README.zh-CN.md#L26) 仍是 CLI 嵌引擎时代命令（`--data ./idx`、"本机零依赖"）。CLI 已改纯 REST 客户端，现无 `--data` flag，照抄即报 `unexpected argument`。英文 README 已更新，中文版没跟——用户第一触点即失败。

## 三、中危发现（20，按主题）

### 过滤翻译 / 检索语义
- **M1** [text/query_build.rs:44-49](crates/fastsearch-text/src/query_build.rs#L44)：`Filter::And(vec![])` 翻成空 `BooleanQuery`（Tantivy `EmptyScorer`）→ 整个检索归零，违反"不可翻译→match-all"。core 语义空 And 恒真。客户端把"无过滤"序列化成 `{"and":[]}` 即触发。
- **M2** [text/query_build.rs:227-244](crates/fastsearch-text/src/query_build.rs#L227) vs [vector/lib.rs:235-236](crates/fastsearch-vector/src/lib.rs#L235)：两端 `FieldSource` 不同构，`time_start_ms`/`time_end_ms` 只有 vector 侧可过滤。`Gte("time_start_ms",…)` 在 text 侧 `get→None` → keyword 召回整支归零（数据其实存在 `StoredRow.media`）。
- **M3** [text/lib.rs:347-353](crates/fastsearch-text/src/lib.rs#L347)：text 侧 over-fetch 固定窗口（4k/8k）无升级循环。选择性"不可翻译过滤"（HeadingPrefix/Exists/未索引字段翻成 AllQuery）+ 高频词时，窗口内过筛后可能远少于 k 甚至 0，尽管索引有足量匹配。HNSW 已有翻倍重搜安全网，text 侧缺。
- **M4** [vector/lib.rs:191-192,422-452](crates/fastsearch-vector/src/lib.rs#L191)：`VecMeta.modality` 物化 + `serde(default)`，旧快照 load 后为 `""`，`Eq("modality","text")` 全不匹配（text 侧是动态派生，两端策略不一致）。索引可重灌修复，故列中。

### 排序 / 管线
- **M5** [engine/lib.rs:1634](crates/fastsearch-engine/src/lib.rs#L1634) + [core/query.rs:20-28](crates/fastsearch-core/src/query.rs#L20)：`RerankSpec.top_k`（默认 20）全 workspace 无处读取，rerank 窗口实为整个融合候选集（可达 ~300）。接入真 cross-encoder 后延迟/费用失控，API 声明的截窗语义静默不生效。
- **M6** [engine/lib.rs:1710,1717,1753-1763](crates/fastsearch-engine/src/lib.rs#L1753)：`collapse_groups`/`auto_merge` 分组键缺 collection（且 collapse 缺 doc_id）。section_id 是 doc 内编号，`collapse{field:"section_id"}` 会让不同文档的同号 section 互相挤占名额、静默丢命中。引擎层不强制 collection 过滤，跨集合同名 doc_id 也会被错误归并。
- **M7** [engine/lib.rs:1495-1517](crates/fastsearch-engine/src/lib.rs#L1495)：`mode=Hybrid`（serde 默认）+ `query_image` 且 `query` 空时，keyword 路走 `AllQuery` match-all → 以图搜图排名混入 match-all 前 150 条无关文档。客户端只发 `query_image` 不显式 `mode=Vector` 即中招。
- **M8** [engine/lib.rs:1143-1159](crates/fastsearch-engine/src/lib.rs#L1143)：`embed_query_image` 只校验 `caps().image` 不校验 `cross_modal`；非 cross_modal 后端把查询图嵌成图域向量，与索引里仅存的文本向量做余弦 → 返回一批无意义命中而非报错。
- **M9** [text/lib.rs:375-379](crates/fastsearch-text/src/lib.rs#L375)：text 侧同分 tie-break 用 `DocAddress`（插入顺序）而非 `GlobalId`，同一真源不同顺序重建后，进 over-fetch 窗口的同分文档会变——违反不变量 #4（vector 两档均已正确落实 GlobalId tie-break）。

### 写路径 / 一致性
- **M10** [engine/lib.rs:1775-1833](crates/fastsearch-engine/src/lib.rs#L1775)：`IndexSink::apply_upsert` 无原子性/回滚。text.upsert 成功后嵌入失败即返回 Err，text writer 留半状态；`commit` 是引擎全局的，其他写路径的 `engine.commit()` 会顺带提交这半状态 → "新文本可检索、向量还是旧版"的不同步窗口。持续失败的事件（超长文本被 4xx 拒）永久卡死 CDC 主循环，无毒丸跳过。
- **M11** [engine/lib.rs:1784-1824](crates/fastsearch-engine/src/lib.rs#L1784)：独占引擎期间逐条同步嵌入（每 chunk 一次 HTTP，最长 30s）。server 单 `Mutex<Engine>`，一批 N chunk = N 次串行 HTTP，期间所有搜索排队；bootstrap 灌大库线性放大。
- **M12** [embed/http.rs:280-301](crates/fastsearch-embed/src/http.rs#L280)：`HttpEmbedder` 无批量切分、无重试，超时硬编码 30s 不可配。server `/v1/index` 把整 doc 的全部 chunk（含 base64 图片膨胀 4/3）合成单次请求 → 触发 413 / 提供方条数上限（OpenAI 2048）/ 30s 超时 → 整个索引请求 500，无部分成功。

### CDC / Postgres
- **M13** [sync/replication.rs:179](crates/fastsearch-sync/src/replication.rs#L179)：TRUNCATE 事件被静默吞（默认 publication 发布 truncate，`map` 落 `_ => Ok(None)`）。真源清空后引擎仍返回旧结果，永久发散。
- **M14** [sync/replication.rs:157-169](crates/fastsearch-sync/src/replication.rs#L157)：Update 忽略 `key`/`old` 元组，改主键（chunk_id/doc_id/collection）的 UPDATE 在派生索引留幽灵行。
- **M15** [sync/lib.rs:71-85](crates/fastsearch-sync/src/lib.rs#L71)：Applier 的 LSN 水位续传契约与真实 pgoutput 语义不符（事务首个数据行 LSN = Begin），engine 生产路径已被迫每拍 `Applier::new(Lsn(0))` 绕开，水位逻辑成死代码但公开 API 契约会误导库形态使用者。
- **M16** [sync/replication.rs:126-130](crates/fastsearch-sync/src/replication.rs#L126) + [pg/sql.rs:290](crates/fastsearch-pg/src/sql.rs#L290)：`fetch_changes` 无上限拉取 + `media_bytes`（可达 20MB/张）走复制流入内存再丢弃——与该处注释"复制流只搬指针"直接矛盾。停机积压后首次 peek 一次载入全部图片字节 → OOM。
- **M17** [pg/sql.rs:269-274](crates/fastsearch-pg/src/sql.rs#L269)：`ann_index_sql` 硬编码 `vector_cosine_ops`，与默认 `halfvec` 列类型不兼容（需 `halfvec_cosine_ops`），且生产代码从不调用它 → B6"pgvector ANN 直查"实际全程顺序扫描，`SET hnsw.*` 空转。
- **M18** [pg/lib.rs:30-37](crates/fastsearch-pg/src/lib.rs#L30) + [server/main.rs:137](crates/fastsearch-server/src/main.rs#L137)：向量维度硬编码 384，与运行时 embedder 维度零联动。配任何非 384 维模型 → `set_embedding` 逐 chunk 运行时报维度不匹配、embedding 列永空，而非启动快速失败。

### 安全（server/mcp）
> 前提正面结论：**所有检索入口的 ACL 强制注入经逐一核查无遗漏**，媒资脱敏彻底，object token 绑定（encrypt-then-MAC + CSPRNG nonce + `uri==target` 复核）扎实。以下为其余问题。
- **M19** [engine/lib.rs:1453](crates/fastsearch-engine/src/lib.rs#L1453)（经 `/v1/similar`）：`more_like_this` 取种子正文时**不过 ACL**（对比 `resolve_citation` 有）。跨租户 `{"citation_id":"kb:<A租户doc>:1"}` → 用 A 的机密正文构造 MLT，形成跨租户存在性预言机 + 语义相似性泄露。
- **M20** [server/lib.rs:84](crates/fastsearch-server/src/lib.rs#L84)：HMAC 签名串 `format!("{cid}|{exp}|{ct}")` 用未转义 `|` 分隔，`cid`（含客户端可控 doc_id）与 `ct`（可控 media_type）可构造规范化歧义使签名跨对复用。利用受 `GlobalId::parse` 语法约束较窄，但属真实缺陷。建议长度前缀/分字段 hash。
- **M21** [server/lib.rs:294-318](crates/fastsearch-server/src/lib.rs#L294)：限流桶 `HashMap` 只增不逐出，且限流在鉴权**之前**、`rate_key` 取客户端可控 token → 开启限流反而引入未认证内存 DoS（每次伪造不同 token 建永不回收的桶）。
- **M22** [server/lib.rs:2154-2206](crates/fastsearch-server/src/lib.rs#L2154)：集合注册表非租户隔离，任意认证用户可 `POST /v1/collections` 覆盖他人集合维度（针对性 DoS）或 `GET` 枚举全部集合名。

### SDK / 工程
- **M23**：两个 SDK 的 `search` 静默丢弃 `collection` 参数（[python/__init__.py:88](clients/python/fastsearch_client/__init__.py#L88)、[ts/client.ts:163](clients/typescript/src/client.ts#L163) 字面 `void collection`），而 CLI 强制注入 `Filter::Eq("collection",…)`。多集合 server 上 SDK 检索会跨集合返回命中（ACL 仍拦越权，但集合作用域失效），与 spec 声明漂移。
- **M24**：Python SDK 严重落后 server API 与 TS SDK（仅 index/search，缺 similar/assets/facets/rerank/delete 等），spec"两端一致"不成立。
- **M25**：`clients/` 无任何 CI job；Docker 构建无 CI 验证且无 `.dockerignore`（本地 `target/` 实测 32GB 会进 build context）；[Dockerfile:8-9](Dockerfile#L8) 注释承诺的"依赖缓存预热层"实际不存在（无 dummy build），源码改动仍全量重编依赖。
- **M26**：CLI 无读取/整体超时（[cli/lib.rs:48-108](crates/fastsearch-cli/src/lib.rs#L48)，ureq 默认无读超时），server 挂起则 CLI 永久阻塞；TS SDK 反而做了 30s 超时。

## 四、低危 / 卫生（摘选）

- **fuse/normalize 遇 NaN 分数**：[core/fusion.rs:99-143](crates/fastsearch-core/src/fusion.rs#L99) `partial_cmp().unwrap_or(Equal)` 对 NaN 产生非传递比较器，破坏确定性；`embed` 的 `l2_normalize` 对非有限 norm 原样保留 NaN 向量是上游源头。建议入口 sanitize。
- **`Fusion::Weighted` 与 `Normalized` 数学等价**且 Weighted 零测试零调用（[core/fusion.rs:72-91](crates/fastsearch-core/src/fusion.rs#L72)）——两名一义徒增困惑。
- **`Filter::and_acl` 是误导性死代码**（[core/filter.rs:92-94](crates/fastsearch-core/src/filter.rs#L92)）：`Filter` AST 无变体能表达多值 ACL，真实 ACL 走 `AclFilter`；留着会给新增入口错误的集成入口。
- **`media_bytes`/`query_image` 以 `Vec<u8>` 走 JSON**（序列化为数字数组，100KB→350KB+），建议 base64。
- **`text::search(k=0)` 返回 1 条**（[text/lib.rs:422-436](crates/fastsearch-text/src/lib.rs#L422) 先 push 后判长度）；vector 侧正确返回空。
- **同源存储型 XSS**：[server/lib.rs:1273-1307](crates/fastsearch-server/src/lib.rs#L1273) inline 字节用客户端可控 Content-Type 直出、无 `nosniff`/`Content-Disposition`，无 MIME 白名单。缓解：header 型 API Key 无 Cookie，窃取面小。
- **5xx 回显内部错误串**（多处 `e.to_string()`，可能泄露表名/SQL 片段）；**API Key 走 HashMap 查找非常量时间**（被 SipHash 缓解，低）；**`FASTSEARCH_KEYS` 解析容错可产生空 key**。
- **`format_vector` 对 NaN/inf 输出 "NaN"/"inf"**（[pg/lib.rs:347-357](crates/fastsearch-pg/src/lib.rs#L347)）→ pgvector 拒收、`set_embedding` 运行时报错，无入参清洗。
- **`ensure_slot` 的 WHERE NOT EXISTS 是 TOCTOU**（[sync/replication.rs:46-63](crates/fastsearch-sync/src/replication.rs#L46)），多副本并发首启会 already-exists 报错上抛，与 `ensure_schema` 的 advisory-lock 防护不对称。

## 五、亮点（值得保留的工程决策）

1. **ACL 纵深防御且不可绕过**：文本/向量后端既把 ACL 作为查询 MUST 子句、又取回后用 `AclFilter::visible` 精确复核；server 由 principal 注入、`apply_ingest_identity` 强制覆盖客户端传入的 acl/tenant/doc_id；`resolve_citation`/分面计数均在 ACL 之后——**全链核查未发现任何泄露路径**（含 HNSW 对第三方库可疑漏筛分支的防御性复检）。
2. **CDC 崩溃安全次序正确且记账诚实**：peek(不推进)→幂等 apply→persist(text commit→vector 原子落盘→checkpoint 原子写)→才 advance_slot，并写下了"为何不能用逐行 LSN 做水位"的反直觉理由；`peek_with_lsn` 返回含 Commit 的最高 LSN 避免整事务重投（实测 Commit LSN 确实更高）。
3. **SUPERSET 下推 + Rust 权威后过滤的双层一致性**（pg `WhereBuilder` ↔ core `Filter::eval`/`AclFilter::visible`）：tenant NULL 严格排除两侧逐条对齐、可空列 `OR IS NULL` 保超集并配了针对性回归；SQL 注入面基本封死（值全参数化 + 表名白名单 `validate_identifier`）。
4. **测试工程的两个范本**：RaBitQ 把统计性组件测成确定性 CI 门禁（固定种子 + recall 头对头 + 5pt 抖动余量）；CLI 用手写 TCP mock 精确验证"Status 不重试 / Transport 重试"、错误分型用类型而非字符串。
5. **hot-path-isolation CI 门禁**把"搜索热路径零 docparse"从口号做成机器约束（实测默认构建确实零 docparse），配 cargo-deny/about 许可门禁扎实。
6. **benchmarks/RESULTS.md 诚实记账**：对 Qdrant 落后 5-7× 如实公布并正确归因（Mutex 串行 + 管线开销而非 ANN），附完整复现命令，不做冠军榜叙事。
