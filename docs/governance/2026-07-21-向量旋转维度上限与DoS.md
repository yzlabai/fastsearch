# 向量旋转维度上限（MAX_DIM）与 DoS —— 决策记录

> 状态：**已实施**（2026-07-21，vector 61 测绿）｜日期：2026-07-21｜上游：[TurboQuant plan](../plans/2026-07-21-向量量化压缩主索引-TurboQuant借鉴.md)、
> [TurboQuant devlog](../devlog/2026-07-21-TurboQuant压缩主索引.md) 复审"遗留"、[15-vector spec](../specs/15-vector.md) §6。
> 触发：双轴审查复审发现 `turbo::MAX_DIM=65536` 偏松，且旋转粗筛档同一 DoS 面无闸。
> **代码是真源**：落地以代码为准并回写本文。

## 1. 背景：MAX_DIM 守的是什么

TurboQuant / RaBitQ 的数据无关旋转当前用**物化 d×d 高斯矩阵 + 改进 Gram-Schmidt**
（[`binary::Rotation`](../../crates/fastsearch-vector/src/binary.rs)）：

| 量 | 复杂度 | 说明 |
|---|---|---|
| 矩阵存储 | **O(d²)** = `4·d²` 字节 | `vec![0f32; d*d]` |
| 建矩阵 | **O(d³)** | Gram-Schmidt 三重循环，首次 upsert / load 时一次性 |
| apply（每向量） | **O(d²)** | d 行各点积 d；**标量**（`unsafe` 禁 → 无 SIMD） |

`MAX_DIM` 的唯一作用是给这个 d×d 分配封顶。

## 2. 成本曲线（按当前标量代码实算）

| d | 矩阵 `4d²` | 建矩阵 `~d³` | apply/向量 `d²` |
|---|---|---|---|
| 1536（OpenAI 3-small） | 9.4 MB | 3.6e9 | 2.4M |
| 3072（OpenAI 3-large） | 38 MB | 2.9e10 | 9.4M |
| 4096（e5-mistral 等最大实测） | 67 MB | 6.9e10 | 16.8M |
| **8192（拟定上限）** | **268 MB** | 5.5e11 | 67M |
| 16384 | 1.07 GB | 4.4e12 | 268M |
| **65536（当前）** | **17.2 GB** | **2.8e14** | **4.3e9** |

关键：d=65536 时**单向量 apply 就要 43 亿次乘加**——即便不谈内存，标量旋转在此维度本就不可用。

## 3. 两个不同的问题（勿混）

- **(a) load 侧 DoS（真威胁）**：turbo 快照只存**码**（d=65536 @4bit 一条 ≈32KB）却在头声明 `dim`。
  几十 KB 的不可信快照 → `load` 触发 **16GB 分配 + O(d³) 卡死**（放大 ~50 万×）。turbo 压缩存储 →
  比其它档更易被此攻击。
- **(b) ingest 侧实用天花板**：即便**合法**维度，标量 O(d²) apply 也是真实摄取瓶颈（d=4096、100K 向量
  ≈1.7e12 flops 纯耗在旋转）；d>~8192 连一次性建矩阵都难受。

## 4. 真实嵌入维度（定安全上限）

截至 2026 初，**无生产级稠密单向量嵌入模型输出 >4096 维**：主流 384/768/1024/1536/2048/3072，
LLM 系最大 3584（gte-Qwen2-7B）/ **4096**（e5-mistral、NV-Embed-v2、SFR、Linq）；且 **Matryoshka
趋势在把维度往下压**（可截断）。ColPali/ColBERT 多向量是每 token 128 维、gated、不进 turbo。

→ **8192 = 已知最大 4096 的 2× 余量**，覆盖一切真实模型且给未知边缘模型留头。

## 5. 发现的两个真问题

1. **防护面不齐**：`MemVectorIndex` 的 `BruteBinaryRotated` 档
   （[`ensure_rotation`](../../crates/fastsearch-vector/src/lib.rs)）建**同一个 d×d `Rotation`，却无任何
   MAX_DIM 闸**（其 f32 快照 d=65536 单条 ≈256KB 文件 → 16GB，放大 ~6.5 万×，仍严重）。**turbo 有闸、
   旋转粗筛档没闸。**
2. **闸的位置错**：上限应守在**唯一分配点** `Rotation::new`（两个旋转档都经此），而非只在 turbo 里。

## 6. 决策

| 项 | 决策 |
|---|---|
| **上限值** | `65536` → **`8192`** |
| **落点** | 常量 `binary::MAX_ROTATION_DIM = 8192`；**`Rotation::new` 改返回 `Result`**，`dim==0 \|\| dim>MAX` → `Err`。两个旋转档（turbo + BruteBinaryRotated）都经此**唯一分配点**，一处防护、不可绕过 |
| **turbo `MAX_DIM`** | 保留 turbo 侧对 `dim` 的显式早检（清晰错误信息 + 与 packed 校验同处），值改用共享 `MAX_ROTATION_DIM`；`Rotation::new` 的 `Result` 作不可绕过兜底 |
| **根治（下一迭代）** | **结构化旋转（FHT / 快速 JL：随机符号 + Hadamard）**：O(d·log d) apply、O(d) 存储、**无 d×d 矩阵**——同时消掉 DoS 分配与标量瓶颈。注：turbovec 自身也是物化 d×d（靠 BLAS 快），fastsearch 上 FHT 反而在"纯 Rust 无 BLAS"约束下**超过** turbovec |

**不变量守护**：这是纯防御性收紧，不改任何真实模型可用性（≤4096 全覆盖）；确定性/预过滤/PG 真源均不受影响。

## 7. 影响面 / 签名改动

`Rotation::new -> anyhow::Result<Self>` 波及（编译器导航，全在 vector crate）：
- `MemVectorIndex::ensure_rotation` / `set_rabitq_rotation` → 返回 `Result`；`upsert` 已 `Result`、
  `VectorStore::load`（BruteBinaryRotated 臂）已 `Result` → 传播。
- `TurboVectorIndex::ensure_rotation` → 返回 `Result`；`upsert`/`load` 已 `Result` → 传播。

## 8. 测试计划

1. `Rotation::new(8193, _)` → `Err`；`Rotation::new(8192, _)` → `Ok`；`Rotation::new(0, _)` → `Err`。
2. **turbo**：`upsert` dim>8192 → `Err`；`load` 快照声明 dim>8192 → `Err`（小文件不触发巨分配）。
3. **BruteBinaryRotated（补漏）**：`upsert` dim>8192 → `Err`；旋转档 load 声明超维 → `Err`。
4. 现有全部旋转/量化测试仍绿（真实维度 ≤3072 不受影响，零回归）。

## 9. 回退

上限是单常量，若未来出现 >8192 的合法模型，改常量即可（附带评估标量旋转在该维的可用性——大概率
该切 FHT）。默认档（Brute/BruteBinary，无旋转）不受此闸影响。

## 10. 实施记录（2026-07-21）

按 §6/§7/§8 落地：

- `binary.rs`：`pub(crate) const MAX_ROTATION_DIM = 8192`；`Rotation::new -> anyhow::Result<Self>`，
  `dim==0 || dim>MAX` → `Err`（在建矩阵前立即返回，不触发巨分配）。
- 传播 `Result`：`MemVectorIndex::{ensure_rotation, set_rabitq_rotation}`、
  `TurboVectorIndex::ensure_rotation` 改返 `Result`；`upsert` / `VectorStore::load`（旋转档臂）/
  `turbo::load` 均 `?` 传播。turbo 的 `MAX_DIM` 改为 `= binary::MAX_ROTATION_DIM`（显式早检给清晰
  错误，`Rotation::new` 的 Result 作不可绕过兜底）。
- **补漏**：`BruteBinaryRotated` 此前无闸，现经 `Rotation::new` 兜底同样受 8192 约束。
- 测试 +4：`binary::rotation_dim_guard`（0/>MAX 拒、小维正常，**不实建 8192²**）、
  `turbo::{upsert,load}_rejects_oversized_dim`（含小文件声明巨维 load 在分配前拒）、
  `rotated_upsert_rejects_oversized_dim`（旋转粗筛档补漏）。
- 收口三绿：fmt + clippy `-D warnings` + `cargo test --workspace` **307 passed / 0 failed**（vector 61）。

**未做（下一迭代）**：结构化旋转（FHT）—— 见 §6，属真正根治，本次仅收紧上限 + 补齐防护面。
